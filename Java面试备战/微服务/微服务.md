# 微服务

## 核心概念

微服务是将单体应用拆分为一组小型、独立部署的服务，每个服务围绕特定业务能力构建。

### 优缺点

| 优点 | 缺点 |
|------|------|
| 独立部署和扩展 | 分布式复杂性 |
| 技术栈灵活 | 网络延迟和故障 |
| 团队自治 | 数据一致性难保证 |
| 故障隔离 | 运维成本高 |

## 服务注册与发现（Nacos）

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
```

- 服务启动时注册到 Nacos
- 消费者从 Nacos 拉取服务列表
- 心跳检测，自动剔除不健康实例

---

## 网关（Spring Cloud Gateway）

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/user/**
          filters:
            - StripPrefix=1
```

核心功能：路由转发、限流、鉴权、日志、跨域

## 熔断降级（Sentinel）

三种策略：
1. **慢调用比例**：RT 超过阈值的比例触发熔断
2. **异常比例**：异常比例超阈值触发
3. **异常数**：异常数超阈值触发

状态机：`CLOSED → OPEN → HALF_OPEN → CLOSED`

## 分布式事务

### 问题

微服务各自有独立数据库，跨服务调用如何保证数据一致性？

### 解决方案

| 方案 | 一致性 | 性能 | 适用场景 |
|------|--------|------|----------|
| 2PC（XA） | 强一致 | 低 | 银行、支付 |
| TCC | 最终一致 | 中 | 订单、库存 |
| SAGA | 最终一致 | 高 | 长流程业务 |
| 本地消息表 | 最终一致 | 高 | 通用 |
| RocketMQ 事务消息 | 最终一致 | 高 | 异步场景 |

### Seata AT 模式

1. 一阶段：各分支执行本地事务 + undo log
2. 二阶段提交：删除 undo log
3. 二阶段回滚：根据 undo log 反向补偿

---

## CAP 定理

分布式系统最多同时满足两个：

- **C（一致性）**：所有节点数据一致
- **A（可用性）**：每次请求都能得到响应
- **P（分区容忍）**：网络分区时系统仍能运行

> 实际选择：CP（ZooKeeper）或 AP（Nacos 默认、Eureka）
