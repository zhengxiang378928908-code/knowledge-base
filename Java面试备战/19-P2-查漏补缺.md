## 二、补偿重试策略（当前 7 分 ✓ 基本达标）

### 核心答法

> 每 5 分钟跑一次，按目标服务分组 → 逐个服务健康检查 → 不可用直接跳过 → 可用才用原始参数重放 → 有最大重试次数兜底。
>
> 补偿机制不会在服务故障时变成压力放大器。

## 三、RAG 项目 vs 直接调 LLM（当前 7.5 分 ✓ 已达标）

### 核心答法

> 电网领域几十个设备、几十种场景、几百种指标之间有特定联系，不能随意组合。LLM 很难直接输出可用 DSL。所以做了编排链路：语句清洗 → 意图识别 → Slot 抽取 → 触发澄清 → 检索 → DSL 生成 → 校验 → 回译确认。

### 加分点

> 早期试过直接让 LLM 生成，准确率不够，才做了这套编排。

## 四、Slot 识别链路（当前 8 分 ✓ 已达标）

### 核心答法

> 四层：正则抽取结构化信息 → AC 自动机提取设备/场景/指标 → 多候选时按语义距离、共现模式、实体特异性消歧 → 打包发给 LLM 做最终判定。缺少 slot 先反推，无法推断触发澄清。

### 必须主动提的 bad case

> "运行年限超过 15 年"容易被当成时间窗口，但它是设备属性过滤条件。DSL 校验阶段会检查并纠正。

## 五、Feign 补偿 vs MQ 重试为什么不统一（当前 6 分 → 目标 7）

### 核心答法

> Feign 补偿解决同步调用失败后的恢复，需要保留完整调用上下文后面才能重放。MQ 自带重试解决消息消费失败，消息本身在 Broker 里，天然支持重投和退避。
>
> 简单说：Feign 失败时没有现成重试基础设施，所以要自己建；MQ 天然有，直接用。

## 六、场景设计——订单下游通知（当前 7 分 ✓ 基本达标）

### 核心答法

> **同步链路：库存 + 支付。** 直接影响下单结果，必须同步。库存成功但支付失败要回滚库存，用本地事务加 Feign，失败走补偿。
>
> **异步链路：风控。** 不阻塞主流程，发 MQ 异步消费。发现问题再冻结订单。

## 七、Redis Lua 令牌桶限流

### 核心答法

> Redis Hash 存当前令牌数和上次补充时间。请求进来，Lua 脚本算时间差补充令牌（不超容量），够就扣减放行，不够就拒绝。一次网络往返，无并发问题。
>
> **两个设计细节：**
> 1. 时间戳由 Java 侧传入，不依赖 Redis 时钟
> 2. 限流异常时 fail-open，不连带打挂核心业务

### 追问：令牌桶 vs 漏桶？

> 令牌桶允许突发（桶里攒了令牌可以一次用多个），漏桶严格匀速。选令牌桶因为设备批量上报需要允许突发。

## 八、MQ 复杂场景——设备级熔断 + 消息配对

### 核心答法（四层处理）

> **第一层，设备级熔断检查。** 查 Redis 有没有被熔断，有就直接跳过。
>
> **第二层，UUID 去重。** Redis 3 分钟窗口。
>
> **第三层，设备日计数 + 自动熔断。** 原子自增，单设备日超 700 条触发熔断标记，有效期 3 天。
>
> **第四层，消息过滤 + 异步转发。** 特定类型才转发到另一个 Topic。

### 消息配对场景

> 遥信1（位置）和遥信2（变位）两条独立消息，只有配对成功才生成工单。先入库，异步查同设备同时间窗口（10 秒）有没有另一条。设备级内存锁防竞态。
>
> **优化方向：** 把 Thread.sleep 轮询改成 Redis 配对缓存——先到的写 Redis，后到的检查 Redis 触发配对，不需要 sleep 和轮询数据库。

### 追问：可以优化什么？

> 1. 异步转发失败加补偿落库
> 2. 减少重复 JSON 解析
> 3. 区分业务异常和系统异常（前者跳过，后者让 MQ 重投）
> 4. 加全局限流兜底

## 九、任务启动恢复

### 核心答法

> 用 CommandLineRunner，应用启动时做四件事：
> 1. 遗留"运行中"的任务日志改成"系统关闭中断"
> 2. 从 Redis 恢复调用计数同步到数据库
> 3. 更新任务配置表的最近运行状态和累计次数
> 4. 清理 Redis 临时状态
>
> **核心思路：** 不依赖 finally（进程被 kill 时 finally 不一定执行），而是下次启动做状态修复，保证最终一致。
