# S级高频题

## Q1. 你先做个自我介绍。

- 面试官为什么会问这个：
想先判断你的主线是不是和岗位匹配，尤其看你是不是核心后端，而不是履历看着杂但没有主线。
- 我的回答（结合真实项目）：
我一般会这样开场：我做 Java 后端 8 年了，前几年主要在银行和业务系统里做核心模块、跑批优化、缓存和 SQL 性能治理。这两年在电网业务里主要做两条线，一条是变电 3.0 微服务体系里的任务治理、RocketMQ 异步链路、OpenFeign 失败补偿和稳定性优化；另一条是 AI 规则解析平台，我主导了自然语言到 Business DSL 的后端主链路，把规则抽取、混合检索、DSL 生成、多轮校验和回归验证做成了独立服务。所以我的主线一直是后端系统设计和稳定性，只是最近又叠加了 AI 工程化能力。
- 如果我没真正做过，怎么诚实但不弱地回答：
这题不需要夸大，核心是把自己讲成“Java 后端主线 + 有系统能力 + 有新技术延展”。别把自己讲成什么都做过，反而会显得虚。
- 可继续深挖的追问：
你最近最能代表你能力的项目是哪一个。你在项目里具体负责什么。为什么会转向做 AI 应用。

## Q2. 你做过的最复杂的系统是什么？

- 面试官为什么会问这个：
他想看你有没有做过真正复杂的系统，以及你口中的“复杂”到底是业务复杂、技术复杂，还是只是模块多。
- 我的回答（结合真实项目）：
如果是偏 Java 后端岗位，我会答变电 3.0。因为它不是单一服务，而是 7 模块 Maven 聚合工程，里面同时有任务治理、异步消息、失败补偿、规则执行、数据同步这些链路。复杂点不在于接口多，而在于它是一个“任务 + 消息 + 补偿 + 规则”的综合系统。比如我做的任务治理平台，不只是让任务定时跑起来，还包括任务注册、动态启停、手动控制、运行日志、调用次数控制、Redis 锁互斥、应用重启后的状态收敛。再比如 OpenFeign 失败补偿，不是简单重试，而是失败后完整落库原始调用上下文，再由后台任务按服务健康状态分组重放。  
如果面试官明显对 AI 更感兴趣，我会切 RAG / DSL 项目。那个项目最复杂的点在于，它不是调个模型就结束，而是把多源抽取、混合检索、DSL 生成、多轮澄清、结构校验和固定回归基线串成一条可持续迭代的后端链路。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要把“访问量大”当成复杂的唯一标准。你可以说我做的不是超大互联网平台，但复杂在链路协同、状态管理、幂等补偿和稳定性治理，这也是中高级后端最核心的能力。
- 可继续深挖的追问：
为什么不用 XXL-Job。补偿为什么不直接用 Feign 重试。RAG 项目的 hardest part 是什么。

## Q6. 你做过的高并发场景是什么？峰值大概多少？

- 面试官为什么会问这个：
他想验证你对“高并发”有没有真实认知，也想看你会不会为了迎合岗位胡报数字。
- 我的回答（结合真实项目）：
我不会硬报一个互联网广告那种特别夸张的 QPS。我的真实场景更偏持续高吞吐数据入库和异步并发处理。比如变电 3.0 里设备台账、在线监测、试验报告这些链路，需要多源数据持续同步，既有定时拉取，也有 RocketMQ 异步消费，还要落库和业务联动。这里的核心难点不是单接口瞬时 QPS，而是长时间稳定吞吐、批量写入、幂等控制和下游保护。  
我在任务链路里做过设备 ID 分批、写库分批、子任务并发汇总、调用次数控制、Redis 锁互斥、Lua 限流这些治理；在消息链路里做过短时去重、业务幂等和失败重试边界控制。所以如果把高并发理解成“持续高吞吐 + 并发消费 + 系统稳定”，我是有真实经验的。
- 如果我没真正做过，怎么诚实但不弱地回答：
如果没统计过峰值，就别造数据。可以说我所在业务不是纯互联网流量型，但我处理过高吞吐入库、异步消费和高峰保护，这些本质也是高并发系统问题。
- 可继续深挖的追问：
分批粒度怎么定。线程池怎么配。流量打满时你先保哪个环节。

## Q7. 如果让你设计一个点击上报 / 事件采集链路，你会怎么设计？

- 面试官为什么会问这个：
这是广告后台的经典题，主要看你能不能把事件链路设计成高吞吐、可扩展、可追踪、可恢复。
- 我的回答（结合真实项目）：
虽然我没做过广告点击上报，但我会把它抽象成一条高吞吐事件采集链路，这和我现在做的数据同步、消息异步、补偿恢复是同类问题。  
我会拆成六层。第一层是接入层，API 网关先做鉴权、基础参数校验、限流和 traceId 生成，避免脏流量直接打进核心链路。第二层是写入层，接入服务尽量轻，只做事件标准化、补齐 eventId / bizId / 时间戳，然后快速写 Kafka 或 MQ 做削峰，避免同步重处理。第三层是预处理层，做去重、格式校验、黑名单过滤和基础 enrich。第四层是实时处理层，把有效事件分发给实时计算或聚合服务。第五层是落库层，明细、聚合、报表分开存，OLTP 和 OLAP 分层。第六层是补偿和审计层，落失败队列、死信、重放任务和全链路日志。  
我自己的项目里，最接近这类能力的，是 RocketMQ 异步链路里的短时去重 + service 层幂等，以及 Feign 失败落库后按健康状态补偿重放。广告里只不过把业务对象从“告警 / 工单 / 设备数据”换成了“点击 / 曝光 / 转化”。
- 如果我没真正做过，怎么诚实但不弱地回答：
开头就承认“我没做过广告点击链路，但我会按通用事件采集架构来设计”，这比硬装广告经验更稳。
- 可继续深挖的追问：
eventId 怎么生成。乱序怎么办。如何灰度新埋点版本。如何做多机房容灾。

## Q8. 高并发写入场景下，你怎么保证系统扛得住？

- 面试官为什么会问这个：
本质是在看你有没有把写入压力拆解成队列、批处理、缓存、分库分表、限流这些手段。
- 我的回答（结合真实项目）：
我一般会按“入口、队列、计算、存储、保护”五层看。入口层先做限流和快速校验，别让无效请求进入主链路。中间用 MQ 削峰，把同步写改成异步写，接入服务只负责快速接收。处理层做批量消费、批量落库，像我在变电项目里就做过设备 ID 分批、写库分批，避免单批过大拖垮内存和数据库。存储层重点是索引、写入模式和冷热数据分层，能顺序写就别随机写，能批量 insert 就别一条条插。保护层再加调用上限、线程池有界队列、降级和补偿。  
我的真实经验里，银行跑批 OOM 也是同样的道理，一次性全量加载肯定扛不住，后来改成分页拉取、分批处理、分批提交，内存和 RT 都稳定很多。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要一上来就说分库分表。高级岗位看的是你有没有系统化思路，不是堆名词。
- 可继续深挖的追问：
批量大小怎么调。数据库成为瓶颈时怎么办。如何防止热点 key 或热点分区。

## Q10. 如果流量突然上涨 10 倍，你会先看哪里？

- 面试官为什么会问这个：
想看你排查问题时有没有优先级，而不是想到哪说到哪。
- 我的回答（结合真实项目）：
我会先看四层。第一层看入口有没有异常流量，是正常业务增长、埋点 bug，还是重试风暴。第二层看中间件，重点看 MQ 有没有积压、Redis 连接数和命中率、线程池是否打满。第三层看数据库和下游服务，看慢 SQL、连接池、远程调用错误率。第四层再看业务层，是不是某个事件类型、某个 partition、某个 task group 被打热了。  
我自己的经验里，OpenFeign 失败补偿这块就特别怕“下游本来就挂了，我们还持续重放”，所以后来我加了服务健康检查，目标服务整体不可用就跳过本轮补偿，避免把局部故障放大成全链路雪崩。
- 如果我没真正做过，怎么诚实但不弱地回答：
不用装成经历过 10 倍流量洪峰。说“如果遇到这种场景，我会优先看流量来源、中间件、数据库、下游依赖这四层”就很稳。
- 可继续深挖的追问：
什么情况下先限流。什么情况下先扩容。什么情况下先降级。

## Q11. 如何设计一个支持扩展的事件处理链路？

- 面试官为什么会问这个：
看你是不是只会写 if else，还是能把链路做成平台化、可插拔。
- 我的回答（结合真实项目）：
我会把事件链路设计成“统一事件模型 + Handler 编排 + 配置化路由”。先统一事件信封，比如 eventId、bizId、eventType、source、occurTime、traceId、payload、version，这样后面新增事件类型不会动基础设施。然后按责任链或者策略模式拆处理器，比如校验、去重、富化、路由、持久化、通知，每一步职责单一。事件类型和处理器的绑定尽量配置化，而不是写死在大 if else 里。  
这和我 RAG / DSL 项目里的编排思路很像。那个项目不是一个大 Prompt 包打天下，而是文本标准化、Slot 抽取、检索、DSL 生成、校验、回译确认分阶段处理，每一步都可以单独降级、替换和做回归。这个思路迁移到广告事件链路也成立。
- 如果我没真正做过，怎么诚实但不弱地回答：
如果没做过“平台化事件中心”，就说你在别的系统里已经用过相同的编排思想，核心是抽象和边界，而不是具体行业名词。
- 可继续深挖的追问：
怎么做 schema 演进。老事件兼容新版本怎么处理。怎么做 handler 的灰度。

## Q12. 如何让整条链路具备可追踪性？

- 面试官为什么会问这个：
广告和高并发后台很怕“数据进来了但不知道卡哪了”，所以可追踪性是核心能力。
- 我的回答（结合真实项目）：
我会至少保三类标识。第一类是 traceId，用于跨服务串联请求。第二类是 eventId 或 messageId，用于唯一标识事件。第三类是 bizId，比如订单号、转化单号、设备事件唯一键，用于贴近业务排查。  
落地上，入口生成 traceId，写日志和 MQ header 时一路透传；消费者处理时把 traceId、eventId、bizId、reconsumeTimes、handlerName 一起打出来；失败补偿要能查到原始参数、错误信息和重试记录。  
我现在项目里最接近这个的是任务平台和 Feign 补偿。任务平台会记录任务组、任务名、运行状态、调用次数和 URL 维度明细；Feign 失败补偿会把服务标识、方法名、原始参数、错误信息完整落库。虽然我没有在当前项目里系统化落地 SkyWalking，但我对“链路可追踪”不是概念层理解，而是已经在关键链路里做了留痕和可重放。
- 如果我没真正做过，怎么诚实但不弱地回答：
可以明确说“我现在项目里更偏业务留痕和补偿留痕，没有完整地上过全链路追踪平台，但我知道该怎么设计 traceId、bizId 和日志口径”。
- 可继续深挖的追问：
MQ header 怎么透传。异步线程里的 traceId 怎么传。怎么把日志和监控指标关联起来。

## Q13. 重复请求或者重复消息，你怎么处理？

- 面试官为什么会问这个：
因为高并发系统里“重复”几乎一定发生，关键不在能不能绝对避免，而在你怎么兜住副作用。
- 我的回答（结合真实项目）：
我一般分两层做。第一层是入口或消费者侧的短时去重，挡住网络抖动、重试、重复投递带来的短周期重复。第二层是真正的业务幂等，按业务唯一键兜住副作用。  
我在变电项目里就是这么做的。`FaultTripRecordConsumer` 先基于 Redis 做短时去重，处理窗口 3 分钟，成功后再打 10 分钟标记，避免短时间内同一消息反复进来。然后真正的业务层还会按业务键判断是不是已经处理过，防止 Redis 过期或者消息换了 messageId 但本质还是同一业务事件。  
所以我的原则一直是：不要迷信某一种去重手段，最终还是业务幂等最稳。
- 如果我没真正做过，怎么诚实但不弱地回答：
如果没做过消息去重，也可以从重复下单、重复回调、重复任务执行来答，核心思想是一样的。
- 可继续深挖的追问：
业务唯一键怎么选。Redis 去重 TTL 怎么定。数据库唯一索引和状态机怎么组合。

## Q15. 幂等和去重有什么区别？

- 面试官为什么会问这个：
很多人会把这两个词混着说，这题就是看你概念是不是清楚。
- 我的回答（结合真实项目）：
我的理解是，去重更偏“拦截重复输入”，幂等更偏“即使重复执行，结果也一致”。去重一般发生在请求入口、消费者入口，目的是减少无效处理；幂等发生在业务处理层，目的是保证副作用不被重复执行。  
我项目里的 Redis 短时去重，就是典型的去重；service 层按业务键判断记录是否已存在，或者是否已经回写过工单，这就是幂等。两者最好一起上，因为去重会漏，幂等更稳，但幂等不能替代所有入口侧成本控制。
- 如果我没真正做过，怎么诚实但不弱地回答：
只要把这两个层次讲清楚，就算没有做过很复杂的例子也不会弱。
- 可继续深挖的追问：
什么场景只做幂等就够了。什么场景去重收益更高。

## Q16. S2S 回传如果重复、乱序、延迟到达，你会怎么处理？

- 面试官为什么会问这个：
这是广告链路很典型的问题，主要看你有没有事件时序和状态机思维。
- 我的回答（结合真实项目）：
我没做过广告 S2S 回传，但这类问题我会按“唯一键、事件时间、状态机、补偿窗口”来处理。  
第一，先定义业务唯一键，比如 clickId、conversionId 或者 orderId，不要只看 messageId。重复到达时先做幂等判断。第二，处理逻辑尽量基于事件时间而不是到达时间，乱序就按版本号或事件时间决定是否覆盖。第三，状态流转要有限定，比如已确认转化的记录，后面再来一条旧状态回调不能回滚主状态。第四，延迟到达要有可接受窗口，超窗口单独标记，用补偿或离线修正，而不是直接污染实时口径。  
这和我现在做的告警到工单、失败补偿其实是同样思路。不是消息来了就盲目更新，而是要看它对应的业务状态处在哪个阶段，能不能转，是否需要留补偿痕迹。
- 如果我没真正做过，怎么诚实但不弱地回答：
这题建议直接承认“没做过广告回传”，但把事件状态机思路讲出来，反而会显得靠谱。
- 可继续深挖的追问：
如果同一转化先到、点击后到怎么办。如何设计回传状态表。如何处理跨天延迟。

## Q17. 如何保证业务“最终一致性”？

- 面试官为什么会问这个：
分布式场景下很难强一致，所以他想看你有没有成熟的最终一致性手法。
- 我的回答（结合真实项目）：
我的思路一直是“主链路先完成核心事实，副作用异步化，失败可补偿”。像变电项目里，Feign 失败补偿不是要求当前调用必须成功，而是先把失败调用上下文完整落库，后面再补偿重放。RocketMQ 链路里也是类似，主处理失败就进入重试或者死信，副作用像短信通知要和主事务隔离。  
所以我理解最终一致性，核心不是一句“MQ 异步解耦”，而是要有状态记录、失败留痕、可重试、可终止、可人工介入。没有这些，所谓最终一致性只是口号。
- 如果我没真正做过，怎么诚实但不弱地回答：
如果没上过复杂分布式事务框架，也没关系。把本地事务 + 消息 / 补偿 / 状态表这套思路讲明白就够了。
- 可继续深挖的追问：
哪些场景你会接受最终一致性。哪些绝不能接受。补偿重试上限怎么定。

## Q18. 如果某一步成功、下一步失败了，你怎么补偿？

- 面试官为什么会问这个：
想看你是不是只会说“加重试”，还是能考虑补偿边界和副作用。
- 我的回答（结合真实项目）：
我会先判断失败的是主流程还是副作用。如果主流程已经提交成功，后面只是通知、回写、统计失败，那我更倾向异步补偿，不回滚主事实。比如故障跳闸链路里，主记录落库和短信通知就应该拆开，不能因为短信失败把主数据也回滚掉。  
如果失败的是强相关步骤，我会保留状态表或者补偿表，记录原始参数、错误信息、已执行到哪一步，再由后台任务按规则重试。这个设计我在 Feign 失败补偿里真实做过，而且还加了服务健康检查，避免下游都挂了还一直放大重试。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要说“我会用分布式事务框架解决”。先把业务拆成主事实和副作用，面试官更爱听这个。
- 可继续深挖的追问：
补偿失败后怎么处理。怎么避免补偿任务和实时请求打架。人工介入入口怎么留。

## Q19. 你项目里为什么用 MQ？解决了什么问题？

- 面试官为什么会问这个：
他想听的不是“为了异步”，而是你到底用 MQ 消除了什么系统问题。
- 我的回答（结合真实项目）：
我项目里用 RocketMQ，核心解决三个问题。第一是解耦，告警、工单、统计这些动作没必要强同步串在一个请求里，不然一个环节慢就把全链路拖慢。第二是削峰，特别是集中同步和告警高峰时，用 MQ 把写入和处理拆开，入口更稳。第三是失败恢复，消息失败后有重试、死信和人工排查入口，比同步链路里直接失败更容易治理。  
我不会泛泛说“我们用了 MQ”，我会直接举 `FaultTripRecordConsumer` 这个例子，里面就有 Redis 短时去重、业务幂等、失败抛异常重试、超过次数停止重试这些生产细节。
- 如果我没真正做过，怎么诚实但不弱地回答：
如果主要用别的 MQ，也可以直接说“我生产里用的是 RocketMQ，但用 MQ 的原因在业务系统里是共通的”。
- 可继续深挖的追问：
为什么选 RocketMQ 不是 Kafka。哪些场景不应该上 MQ。

## Q23. 如何保证消息不丢？

- 面试官为什么会问这个：
这是 MQ 基础题，但高级岗位会看你是不是能按全链路讲。
- 我的回答（结合真实项目）：
我会按三段说。Producer 到 Broker、Broker 存储、Broker 到 Consumer。  
Producer 侧，关键消息我会同步发送，失败重试，必要时结合发送结果和落库兜底。Broker 侧，业务可靠性要求高的场景要走更稳的刷盘和副本策略。Consumer 侧，只有业务真正处理成功了才确认消费，失败就重试，不要为了“看起来不报错”提前 ack。  
我自己项目里虽然不是每条消息都做到极致强保证，但在关键链路上，思路就是这三段拆开看，而不是只盯消费者。
- 如果我没真正做过，怎么诚实但不弱地回答：
只要别把“不丢”说成一句话就行。高级岗位最怕答得太笼统。
- 可继续深挖的追问：
发送成功但落库失败怎么办。消费成功但回执丢了怎么办。死信如何处理。

## Q24. 如何避免重复消费？

- 面试官为什么会问这个：
因为大多数 MQ 都只能保证至少一次，重复消费是常态。
- 我的回答（结合真实项目）：
我一般会直接说，不追求“绝不重复”，而是把消费端做成幂等。项目里我用过三种方式。第一种是 Redis 短时去重，挡住快速重复投递。第二种是数据库唯一键或业务状态机，兜住真正的业务副作用。第三种是把通知类副作用和主事务拆开，只对真正新插入的数据触发通知，避免重试时重复发短信或重复创建工单。  
像故障跳闸链路，我就会特别强调：真正要避免的不是消息重复本身，而是重复副作用。
- 如果我没真正做过，怎么诚实但不弱地回答：
哪怕没写过消息消费代码，只要你做过重复请求防护，也能把思路迁移过来。
- 可继续深挖的追问：
Redis 去重失效了怎么办。数据库唯一约束冲突如何处理。顺序消息如何做幂等。

## Q26. 消息积压怎么排查？

- 面试官为什么会问这个：
他想看你能不能快速判断积压发生在哪，是生产太快、消费太慢，还是局部热点。
- 我的回答（结合真实项目）：
我会先看积压是全局性的还是局部性的。如果全局积压，先看生产侧是不是突然放量，或者消费者整体处理能力下降。再看消费者日志，是业务异常重试导致吞吐下降，还是下游数据库、Redis、远程服务变慢。  
如果是局部积压，我会重点看某个 Topic、某个分区、某类消息是不是特别热。因为很多时候不是所有消息都慢，而是某个业务键、某个消费者分支拖住了。  
我项目里最接近的经验，是 RocketMQ 消费失败重试和下游服务不稳定导致的吞吐下降，所以我会很关注 reconsume 次数、异常分布、下游 RT 和数据库慢 SQL，而不是只看 MQ 自己。
- 如果我没真正做过，怎么诚实但不弱地回答：
可以明确说“我没有处理过特别夸张的大积压，但我会按生产速度、消费速度、热点分区、下游依赖四个方向排查”。
- 可继续深挖的追问：
积压时先扩容消费者还是先止血。怎么处理死信。热点 partition 怎么迁移。

## Q30. 你线上有没有遇到过 MQ 堆积、重复消费、消费失败？

- 面试官为什么会问这个：
这题是在找真实线上经验，最好能说出一个具体链路。
- 我的回答（结合真实项目）：
我遇到过消费失败和重复投递。像故障跳闸消息这条链路里，我们就专门做了两层兜底。消费者入口先做 Redis 短时去重，防止同一消息短时间内多次进入；业务处理失败则抛异常给 RocketMQ 重试，但重试不是无限的，超过阈值后就停止继续重试，避免一直空转。  
我对这类问题的经验是，线上最怕两件事，一个是重复消息把副作用做重了，一个是下游已经异常还继续疯狂重试，所以边界一定要清楚。
- 如果我没真正做过，怎么诚实但不弱地回答：
可以承认没遇到过超大规模堆积，但要把重复消费和失败重试的处理逻辑讲清楚。
- 可继续深挖的追问：
超过重试上限之后怎么处理。是否有死信和人工兜底。如何区分临时失败和永久失败。

## Q31. 你怎么拆服务？

- 面试官为什么会问这个：
他想看你是按技术层拆，还是按业务边界拆。
- 我的回答（结合真实项目）：
我更偏向按领域职责和变化边界拆，而不是按 controller / service / dao 这种技术层拆。变电 3.0 里其实就能看到这种思路，业务服务、数据处理、调度、事件调度、公共能力是分开的。因为它们的变化频率、资源模型和稳定性要求都不一样。  
我自己判断拆不拆，主要看三点。第一，业务语义是否独立。第二，数据和状态是否有清晰归属。第三，扩容和故障隔离需求是否明显。如果这三点都不成立，我不会为了微服务而微服务。
- 如果我没真正做过，怎么诚实但不弱地回答：
别装成主导过大型拆分工程。可以说我在现有项目里参与过按职责拆分和模块边界治理，也理解拆分原则。
- 可继续深挖的追问：
任务平台为什么独立出来。什么情况下继续保留单体更好。

## Q37. 跨服务事务怎么处理？

- 面试官为什么会问这个：
看你是不是一上来就喊 Seata、2PC，而不理解业务取舍。
- 我的回答（结合真实项目）：
我一般优先接受最终一致性，不会轻易上强一致分布式事务。因为高并发和复杂链路下，强一致成本很高。我的做法更偏本地事务先保证本服务内状态正确，跨服务靠消息、补偿、状态表和幂等来收敛。  
变电项目里 OpenFeign 失败补偿就是这个思路。调用失败后不强依赖当前请求立刻成功，而是把原始上下文落库，后面再按规则补偿。这样做的好处是主流程不被拖死，失败也有留痕和恢复入口。
- 如果我没真正做过，怎么诚实但不弱地回答：
可以直接说“我没深度落过 Seata 这类框架的生产链路，但我真实做过补偿和最终一致性方案”。
- 可继续深挖的追问：
什么业务你绝不会接受最终一致性。补偿乱序怎么处理。如何设计状态机。

## Q41. MySQL 慢 SQL 你怎么排查？

- 面试官为什么会问这个：
数据库问题是后端岗位必问，而且广告后台也离不开高频写和报表查。
- 我的回答（结合真实项目）：
我一般是四步。第一步先从慢查询日志或监控里定位具体 SQL，不猜。第二步用 `EXPLAIN` 看执行计划，重点看 type、key、rows、Extra，有没有全表扫、回表、filesort。第三步看索引是不是和 where、join、order by 真正匹配。第四步再决定是改索引、改 SQL、拆批还是把这类查询从 OLTP 链路里拆出去。  
我在银行项目里处理过报表和跑批慢 SQL。一个典型问题就是统计 SQL 没走到合适索引，执行时间太长，最终把连接池拖满，导致请求堆积。后面通过索引调整、SQL 改写和分批处理把问题压下来。
- 如果我没真正做过，怎么诚实但不弱地回答：
如果没亲手开过慢日志，也可以从 `EXPLAIN` 和执行计划讲起，但最好带一个真实项目例子。
- 可继续深挖的追问：
联合索引怎么设计。为什么会回表。复杂报表什么时候该迁移到分析引擎。

## Q45. Redis 在你的项目里是怎么用的？

- 面试官为什么会问这个：
高级岗位会看你是不是只把 Redis 当缓存，还是会拿它做分布式协调。
- 我的回答（结合真实项目）：
我项目里 Redis 绝对不只是缓存。至少用了四类能力。  
第一类是分布式锁，任务执行前做跨节点互斥，避免多节点重复跑。第二类是限流，用 Lua 令牌桶控制外部调用和数据库调用，限流器异常时默认 fail-open，保护主业务。第三类是消息去重，RocketMQ 消费前先做短时间窗口去重。第四类是计数和运行状态，任务平台会统计调用次数、记录 URL 维度明细，应用重启后还要和日志表、配置表一起做状态收敛。  
所以我平时讲 Redis，会更强调它在工程治理里的作用，而不是“提高查询速度”那么简单。
- 如果我没真正做过，怎么诚实但不弱地回答：
如果某一类没做过，就重点讲自己真实做过的那几类，不要把缓存、分布式锁、布隆过滤器全往自己身上揽。
- 可继续深挖的追问：
为什么限流用 Lua。锁为什么还要业务幂等兜底。Redis 挂了怎么办。

## Q46. 缓存一致性怎么做？

- 面试官为什么会问这个：
缓存一致性是高频老题，但高级岗位主要看你是不是知道它没有银弹。
- 我的回答（结合真实项目）：
我常用的是 cache aside，也就是更新数据库后删除缓存，而不是先更缓存。因为删缓存更简单，也更容易保证幂等。如果是一致性要求特别高的配置类数据，我会把过期时间设短一点，必要时加消息通知或主动刷新。  
我在银行项目里做黑白名单和高频查询时，会把 Redis 作为热点查询加速层，但我不会把它讲成“强一致缓存系统”。我的原则一直是，缓存是性能组件，不应该反过来定义业务真相。
- 如果我没真正做过，怎么诚实但不弱地回答：
别把双写一致性说得太轻松。承认它本质是最终一致，反而更像做过生产系统的人。
- 可继续深挖的追问：
为什么不是先删缓存再更新库。延迟双删有没有必要。热点 key 怎么处理。

## Q51. 你做过哪些稳定性建设？

- 面试官为什么会问这个：
这个岗位明确强调稳定性建设，所以这题相当于你的高光题。
- 我的回答（结合真实项目）：
我做过的稳定性建设比较成体系，主要有五类。  
第一类是任务治理，统一任务注册、动态启停、手动控制、运行日志、启动恢复，避免任务失控。第二类是失败补偿，Feign 调用失败后完整落库，按服务健康状态补偿重放。第三类是消息可靠性，RocketMQ 消费做短时去重、业务幂等和重试边界控制。第四类是边界保护，Redis Lua 令牌桶限流、调用上限控制、线程池和批处理控制。第五类是线上排障和恢复，比如 OOM、慢 SQL、连接池耗尽、应用重启后的状态收敛。  
如果让我总结一句，我做的稳定性不是口号，而是围绕“可控、可追踪、可恢复”落到具体链路上的。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要硬往 SLA、演练、全链路压测上靠。把自己真实做过的稳定性动作讲透就够了。
- 可继续深挖的追问：
哪个建设最有成效。做过哪些稳定性取舍。出过什么线上事故。

## Q55. 线上 RT 飙高、错误率上升，你怎么排查？

- 面试官为什么会问这个：
这题考的是你的实战排障顺序和系统观。
- 我的回答（结合真实项目）：
我会先止血，再定位。止血手段包括回滚、限流、降级、关闭某些高风险任务或补偿任务。定位上我会按入口、应用、中间件、数据库、下游服务这五层排。  
先看是不是某次发布后开始的，再看应用线程池、GC、CPU、错误日志。然后看 Redis、MQ、数据库慢 SQL、连接池、Feign 下游 RT。最后再回到业务层，看是不是某个 task group、某个 Topic、某类请求特别异常。  
我自己的经验里，报表慢 SQL 把连接池拖满、Feign 补偿在下游不可用时放大故障、跑批一次性加载全量导致 OOM，都是这种链路化排查思路定位出来的。
- 如果我没真正做过，怎么诚实但不弱地回答：
不用装成每次都亲自总控故障。你可以说自己参与过排查，知道如何分层定位和先止血。
- 可继续深挖的追问：
如何判断是 CPU 问题还是 IO 问题。如何快速定位慢 SQL。怎么设计故障后的复盘动作。

## Q56. 如何设计监控和告警？

- 面试官为什么会问这个：
稳定性建设如果没有监控告警，就是空的。
- 我的回答（结合真实项目）：
我一般会分四层设计。系统层看 CPU、内存、磁盘、GC、线程池。中间件层看 Redis、MQ、数据库连接池、慢查询、积压。应用层看 RT、错误率、超时率、重试次数、补偿队列长度。业务层看关键任务成功率、消息消费成功率、工单创建成功率、DSL 解析成功率这种业务指标。  
告警我会尽量做成分级，不是任何异常都立刻轰炸。像瞬时抖动可以用时间窗口和连续触发阈值，持续失败或者关键业务失败率异常才升级。  
我现在项目里虽然不是专门做监控平台，但做任务治理和补偿链路时，已经很重视运行日志、调用次数、失败留痕和状态收敛，这些其实就是监控和告警设计的基础数据。
- 如果我没真正做过，怎么诚实但不弱地回答：
可以承认自己不是监控平台 owner，但要证明你理解“技术指标 + 业务指标 + 分级告警”三层。
- 可继续深挖的追问：
怎么避免告警风暴。业务指标怎么定阈值。如何把告警和自动化恢复结合起来。

## Q71. 你平时怎么用 AI 大模型提升研发效率？

- 面试官为什么会问这个：
岗位 JD 明确提到这一点，所以不是加分项，已经接近必答题了。
- 我的回答（结合真实项目）：
我平时用 AI 主要做四类事。第一类是代码理解和重构建议，尤其是接手老模块、梳理调用链和补偿链路时，用它先帮我做结构化总结，但关键代码我一定自己过。第二类是测试和回归辅助，比如生成 bad case、补 smoke case、帮我整理 DSL 解析失败样例。第三类是排障辅助，像慢 SQL、线程栈、错误日志，我会让 AI 帮我先做归因假设和排查 checklist。第四类是文档沉淀，我会把项目机制、复盘、面试材料整理成知识库。  
我自己现在就在做 RAG / DSL 项目，所以我不是把 AI 当聊天工具，而是已经在做 AI 工程化，也更清楚它在哪些环节真能提效，哪些地方不能直接信。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要说“我平时让它帮我写代码”。更好的答法是讲清楚你把 AI 用在理解、验证、文档化和重复工作自动化上。
- 可继续深挖的追问：
你如何验证 AI 生成代码。你做过哪些 AI 辅助排障。你们团队怎么落地 AI 工具。

## Q75. 你现在做的 AI / RAG / DSL 项目，最难的点是什么？

- 面试官为什么会问这个：
他想看你是不是只会说模型名和框架名，还是理解系统难点。
- 我的回答（结合真实项目）：
我觉得最难的点不是“把模型调通”，而是让自然语言规则稳定落成可执行 DSL。因为电网规则有几个天然难点，第一是别名多，第二是结构化约束强，第三是用户输入经常不完整，第四是不能生成一个看起来像样但实际不可执行的结果。  
所以我们做的不是单轮生成，而是多源 Slot 抽取、混合检索、DSL 生成、多轮澄清、结构校验、回译确认这一整条链。像“110kV 及以上断路器 SF6 泄漏超 50mL 且运行年限超 15 年”这种输入，LLM 很容易把“运行年限”理解错，我们就通过规则抽取、知识库约束和 DSL Validator 去纠正。  
这也是我一直强调这个项目是 AI 后端系统，不是 prompt demo 的原因。
- 如果我没真正做过，怎么诚实但不弱地回答：
如果面试官问得很深，就守住你真实做过的部分，比如编排、校验、回归，不要临场编模型训练细节。
- 可继续深挖的追问：
为什么不用纯向量检索。为什么不用纯 LLM。怎么评估效果。bad case 怎么闭环。
