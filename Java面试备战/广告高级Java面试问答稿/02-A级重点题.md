# A级重点题

## Q9. 低延迟和高可靠冲突时，你怎么取舍？

- 面试官为什么会问这个：
想看你是不是只会两头都要，还是知道不同链路该怎么分层取舍。
- 我的回答（结合真实项目）：
我不会抽象地说“平衡一下”，我会先看链路是不是核心事实链路。像变电项目里主记录落库这类核心事实，我更偏可靠性优先；像短信通知、统计回写这类副作用，我会优先低耦合和异步化，允许稍微延迟一点甚至失败后补偿。  
我的原则是，核心事实先保证正确和可恢复，非核心副作用尽量异步化。这样系统既不会因为追求绝对实时把自己拖死，也不会因为一味追求低延迟把数据做脏。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要说“看业务”。你可以说“我会先区分主事实和副作用，再决定优先级”，这就比空话强很多。
- 可继续深挖的追问：
广告点击和转化回传哪个更偏可靠性。哪些指标可以接受异步修正。

## Q14. 事件去重一般有哪些做法？分别适合什么场景？

- 面试官为什么会问这个：
看你是不是只会 Redis `SETNX` 一招鲜。
- 我的回答（结合真实项目）：
我通常分四类。第一类是入口侧短时去重，比如 Redis `SETNX + EX`，适合挡住瞬时重复请求、重复消息。第二类是数据库唯一键，适合同一业务事实不能重复插入的场景。第三类是状态机幂等，适合回调、状态流转类业务，不只判断有无，还判断能不能转。第四类是内容指纹去重，适合同内容短时间重复到达，但没有稳定业务 ID 的场景。  
我项目里就不是一刀切。有的消息按 UUID 去重，有的按消息内容或业务键兜底。这个思路迁移到广告链路也一样，点击事件更适合 eventId 或 clickId 去重，S2S 回传更适合业务键 + 状态机。
- 如果我没真正做过，怎么诚实但不弱地回答：
你不用每种都做过，只要说明自己知道不同业务语义对应不同去重粒度，就已经比大多数答法强。
- 可继续深挖的追问：
TTL 怎么定。状态机幂等怎么设计。数据库唯一键冲突怎么处理。

## Q20. Kafka 为什么吞吐高？

- 面试官为什么会问这个：
虽然你主用 RocketMQ，但广告岗位高概率会问 Kafka 基础原理。
- 我的回答（结合真实项目）：
我生产里主要用 RocketMQ，但 Kafka 吞吐高的原因我理解主要有四点。第一，它是顺序追加写，磁盘访问模式很友好。第二，充分利用了操作系统 page cache，不是每次都真的同步落盘到物理介质。第三，分区模型天然支持并行读写。第四，零拷贝和批量发送、批量拉取减少了用户态和内核态切换成本。  
我一般会补一句，Kafka 的优势更适合大吞吐日志流和实时计算生态，这也是为什么广告、埋点、Flink 联动场景里大家更常选它。
- 如果我没真正做过，怎么诚实但不弱地回答：
直接说“我生产主要用 RocketMQ，但我理解 Kafka 的高吞吐原理和适用场景”，这样最稳。
- 可继续深挖的追问：
和 RocketMQ 的差别。什么时候宁愿选可靠性更强的方案。

## Q21. Kafka 分区怎么设计？

- 面试官为什么会问这个：
这题本质是在看你如何平衡吞吐、顺序和热点。
- 我的回答（结合真实项目）：
分区数我不会拍脑袋定，主要看三件事。第一是未来吞吐量和消费者并行度，分区太少扛不住，太多会增加管理成本。第二是顺序性要求，如果同一个业务键必须有序，就要按这个键做分区，比如 clickId、userId、campaignId。第三是热点风险，不能把特别偏的 key 全路由到一个分区。  
虽然我没在线上配过 Kafka 分区，但我做 RocketMQ 和任务分组时也是同样思路，都是先想业务分片键，再想扩容和热点。
- 如果我没真正做过，怎么诚实但不弱地回答：
别编自己配过多少 partition。把分区设计原则讲清楚就行。
- 可继续深挖的追问：
怎么评估热点 key。分区数改大后会有什么代价。如何做 key 打散。

## Q22. 消费者组是怎么工作的？

- 面试官为什么会问这个：
这是 MQ 基础概念题，但能看出你有没有系统理解。
- 我的回答（结合真实项目）：
我理解消费者组的核心是“组内分摊、组间广播”。同一个消费者组内，一条消息只会被组内某个消费者处理，用来做水平扩展；不同消费者组之间互不影响，可以让同一份数据被多个下游系统各自消费。  
对我来说，这个概念和 RocketMQ 的消费组思路是共通的，所以虽然我主用 RocketMQ，但我回答 Kafka 消费组不会虚。
- 如果我没真正做过，怎么诚实但不弱地回答：
只要把“组内竞争消费、组间独立消费”讲清楚就够了。
- 可继续深挖的追问：
rebalance 会带来什么影响。如何减少消费抖动。

## Q25. at least once、at most once、exactly once 你怎么理解？

- 面试官为什么会问这个：
这是消息语义的基本功，广告链路尤其关心。
- 我的回答（结合真实项目）：
我的理解很直接。`at most once` 是宁可丢也不重，通常先提交再处理。`at least once` 是宁可重也不丢，通常处理成功后再确认，所以消费端必须做幂等。`exactly once` 更多是端到端的系统语义，不是说消息中间件自己一招解决，往往要配合生产幂等、事务、消费幂等和存储约束一起实现。  
我自己的项目里，实际落得最多的是 `at least once + 消费幂等`，这在业务系统里是最现实的方案。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要把 `exactly once` 说成“绝对不会重复”。面试官一听就知道你在背概念。
- 可继续深挖的追问：
为什么业务系统里 often 选择至少一次。什么时候接受最多一次。

## Q27. 某个 partition 热点很高怎么办？

- 面试官为什么会问这个：
因为真实线上很容易出现热点分区、热点队列把吞吐拖垮。
- 我的回答（结合真实项目）：
我会先区分是设计性热点还是流量突发热点。设计性热点说明分区键选错了，比如全按一个很偏的字段路由，需要重选分区键或做 key 打散。突发热点则要看能不能做局部降级、缓存、异步化或拆分消费者。  
如果业务要求同 key 有序，那热点就更难完全消掉，这时候通常要在顺序和吞吐之间取舍。我虽然没在线上处理过 Kafka 热 partition，但在任务和消息治理里一直有同样的判断逻辑，就是先找“热点来自哪里”，再决定是打散还是隔离。
- 如果我没真正做过，怎么诚实但不弱地回答：
不用硬讲技术细节，关键是说明你知道热点问题首先是路由设计问题。
- 可继续深挖的追问：
如何做 key 打散后再聚合。顺序消费遇到热点怎么办。

## Q28. 顺序消费怎么保证？

- 面试官为什么会问这个：
顺序性和并发度天然冲突，这题就是看你是否清楚边界。
- 我的回答（结合真实项目）：
顺序消费通常只能保证分区内顺序，不会保证全局顺序。所以第一步是选好顺序键，把同一业务实体的消息路由到同一个 partition 或 queue。第二步是消费端按这个分区串行消费，至少不能并发打乱同 key 顺序。第三步还要在业务层做版本号或状态判断，因为即使分区内顺序没乱，重试、补偿、回放也可能影响最终状态。  
我项目里虽然更偏 RocketMQ，但我处理幂等和状态流转时也是这个思路。
- 如果我没真正做过，怎么诚实但不弱地回答：
可以直说“我理解的是分区级顺序，不是全局顺序”，这一句就很加分。
- 可继续深挖的追问：
顺序消息失败如何处理。一个慢消息会不会拖死整个分区。

## Q29. 生产者幂等、事务消息，你了解多少？

- 面试官为什么会问这个：
看你对消息可靠性是不是只停留在消费者侧。
- 我的回答（结合真实项目）：
我生产里主要用 RocketMQ，对事务消息的理解会更贴近它。事务消息本质是先把消息半投递出去，本地事务成功再提交，失败就回滚，Broker 如果不确定还会回查。它适合“本地事务成功后必须发消息”的场景。  
生产者幂等我理解也不是玄学，本质上还是避免因为重试把同一业务消息发成多份，常见做法是给消息绑定业务唯一键，或者让下游能识别重复。  
我项目里更常落的是消费者幂等和失败补偿，但对事务消息的适用场景我是清楚的。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要把“了解一点”装成“深度落地过”。说清楚自己实际主要做的是哪一侧，反而更稳。
- 可继续深挖的追问：
为什么不是所有场景都上事务消息。事务回查会有什么成本。

## Q32. 单体拆微服务时，最重要的原则是什么？

- 面试官为什么会问这个：
高级岗位会看你是不是理解拆分的代价，不是盲目推微服务。
- 我的回答（结合真实项目）：
我觉得最重要的是按业务边界拆，不要按技术层或组织架构拍脑袋拆。因为服务一旦拆出来，最麻烦的不是多了几个仓库，而是数据归属、接口依赖、发布协调和故障传播。  
所以我看一个边界值不值得拆，主要看业务独立性、数据闭环、变更频率和扩容隔离需求。变电 3.0 里把任务治理、数据处理、业务服务拆开，就是因为它们不是同一类职责。
- 如果我没真正做过，怎么诚实但不弱地回答：
即使没主导拆单体，也可以说自己在现有微服务体系里观察到拆分最关键的是边界，而不是“越细越先进”。
- 可继续深挖的追问：
拆完后如何治理配置、调用、监控。哪些模块不适合先拆。

## Q33. 服务拆太细会有什么问题？

- 面试官为什么会问这个：
这是反向题，考你有没有真实踩坑意识。
- 我的回答（结合真实项目）：
拆太细最直接的问题有四个。第一，网络调用变多，链路更长，延迟和故障点都增加。第二，数据边界容易碎掉，跨服务事务和一致性成本上升。第三，排障和发布复杂度会明显上升。第四，团队如果没有足够治理能力，最后只是把单体里的复杂度搬到了分布式环境里。  
所以我不认同为了 DDD 或微服务而拆分，尤其在团队规模不大、业务边界还没稳定时，适度模块化比盲拆更值。
- 如果我没真正做过，怎么诚实但不弱地回答：
这题不需要有亲手拆过的经历，只要你理解拆细的真实成本就行。
- 可继续深挖的追问：
如何判断“太细”了。有没有合并过服务边界。

## Q34. 你怎么理解 DDD？你真实用过吗？

- 面试官为什么会问这个：
很多人会背术语，所以面试官想听真实落地程度。
- 我的回答（结合真实项目）：
我对 DDD 的理解不是一套固定模板，而是“先把业务概念和边界抽清楚，再决定代码怎么落”。我不会说自己做过特别重的 DDD 落地，比如到处建聚合根、领域事件那种，但我在两个项目里其实都用到了 DDD 思想。  
变电项目里，任务治理、告警工单、数据同步就是不同领域边界，不能混在一起。RAG / DSL 项目里更明显，Business DSL 和底层执行层是分开的，业务层表达“规则想做什么”，执行层再决定具体怎么查、怎么触发。这个其实就是典型的领域抽象。
- 如果我没真正做过，怎么诚实但不弱地回答：
最好的说法就是“我用过 DDD 的边界划分和领域抽象思想，但没有做过特别重框架化的 DDD 落地”。
- 可继续深挖的追问：
你们项目里哪个对象最像聚合。Business DSL 为什么算一种领域建模。

## Q35. 聚合、实体、值对象、领域服务、应用服务分别是什么？

- 面试官为什么会问这个：
这是 DDD 概念题，通常是看你能不能结合项目说，而不是背定义。
- 我的回答（结合真实项目）：
实体我理解是有唯一标识、生命周期会变化的对象，比如任务实例、告警记录、工单、规则解析会话。值对象更像一组不可变语义组合，比如时间窗口、阈值条件、DSL 里的某个过滤条件。聚合是把一组强一致边界内的对象收在一起管理，比如工单及其状态变更、任务及其运行状态。领域服务是多个实体和值对象配合时的业务规则承载处。应用服务更像编排层，负责调领域对象、仓储和外部接口。  
在我项目里，RAG 的 `PhaseAOrchestrator` 就更像应用服务；真正的业务 DSL 结构和校验规则，更接近领域模型和领域服务。
- 如果我没真正做过，怎么诚实但不弱地回答：
不用硬把每个类都往 DDD 术语上套。说出你理解的边界就够了。
- 可继续深挖的追问：
任务平台里哪个对象算聚合根。为什么会话上下文不只是 DTO。

## Q36. 你如何做领域边界划分？

- 面试官为什么会问这个：
这是比“什么是 DDD”更实战的一题。
- 我的回答（结合真实项目）：
我通常看三件事。第一，看业务语言是不是天然不同。第二，看数据是不是应该由同一组规则维护。第三，看失败和扩容边界是否一致。  
像变电项目里，任务治理的关注点是注册、调度、运行状态和恢复；告警工单的关注点是业务触发、幂等、副作用和闭环；这两个边界天然不一样。RAG 项目里，Business DSL 和底层执行层也必须分开，不然业务口径和物理实现会耦死。
- 如果我没真正做过，怎么诚实但不弱地回答：
即使没有正式做过领域建模工作坊，也可以从自己项目中的职责分层来回答。
- 可继续深挖的追问：
两个边界之间如何通信。边界划错了怎么调整。

## Q38. 如何避免服务之间强耦合？

- 面试官为什么会问这个：
微服务不是拆出来就完事，强耦合会让系统越来越难维护。
- 我的回答（结合真实项目）：
我觉得关键是三件事。第一，服务之间尽量传业务语义明确的接口，不要直接共享内部表结构。第二，能异步就别强同步，特别是副作用链路。第三，不要让一个服务同时掌握另一个服务的太多内部规则。  
我做 Feign 失败补偿时，其实也是在降低强耦合。因为上游不需要因为下游瞬时失败就完全阻塞，而是把失败调用变成可追踪、可补偿的异步任务。
- 如果我没真正做过，怎么诚实但不弱地回答：
这题重点不在是否主导过全局架构，而在你是否知道怎么减少同步依赖和共享数据。
- 可继续深挖的追问：
什么场景必须同步。共享库和共享 DTO 的边界怎么把握。

## Q39. 什么时候应该平台化，什么时候不应该？

- 面试官为什么会问这个：
想看你有没有抽象能力，也看你会不会过度设计。
- 我的回答（结合真实项目）：
我判断要不要平台化，一般看是否出现了重复问题、是否跨多个业务都存在、是否需要统一治理。像定时任务，如果只是两三个简单任务，没必要上平台；但变电项目里已经有 300+ 任务，而且有启停、日志、人工控制、限流、恢复这些共性诉求，那平台化就很值。  
反过来，如果一个能力只服务单一场景，边界还在快速变化，我更愿意先在业务里做薄抽象，不急着平台化。
- 如果我没真正做过，怎么诚实但不弱地回答：
用你自研任务治理平台做正例，用“不要为平台而平台”做反例，回答会很稳。
- 可继续深挖的追问：
平台化最容易失败的点是什么。如何避免平台和业务脱节。

## Q40. 你做过哪些架构设计取舍？

- 面试官为什么会问这个：
高级岗位很看重“为什么这么做”，不只是“做了什么”。
- 我的回答（结合真实项目）：
我印象比较深的有三个取舍。第一，自研任务治理平台而不是直接上 XXL-Job，因为我们的核心诉求不是复杂编排，而是轻量部署和深度治理。第二，Feign 失败不只做同步重试，而是落库补偿，因为短重试解决不了留痕和恢复问题。第三，RAG 项目选择 Python 独立服务而不是硬塞进 Java 主体系，因为 AI 链路迭代快、生态也更适合 Python，但它和 Java 主业务通过 API 解耦。  
这些取舍背后都是一个原则，先看问题本身是什么，再选方案，不为了追热点技术。
- 如果我没真正做过，怎么诚实但不弱地回答：
挑两三个你最熟的取舍讲透，比堆十个名词强得多。
- 可继续深挖的追问：
如果重来一遍会改什么。哪次取舍后来证明不够好。

## Q42. 索引怎么设计？

- 面试官为什么会问这个：
广告后台写多查多，索引设计非常关键。
- 我的回答（结合真实项目）：
我设计索引会先看查询模式，不会先看表结构。where 条件、join 条件、排序、分页字段谁最稳定、选择性怎么样，这些决定索引顺序。联合索引也要尽量遵守最左匹配，把过滤度高、最常用的条件放前面。  
我在银行报表和风控查询里，更多是根据真实 SQL 来补索引，而不是泛泛说“给高频字段建索引”。另外也会考虑写放大成本，索引不是越多越好。
- 如果我没真正做过，怎么诚实但不弱地回答：
即使没主导过库表设计，也可以从你调优慢 SQL 的经验反推索引设计原则。
- 可继续深挖的追问：
覆盖索引怎么用。为什么 order by 会用不上索引。

## Q44. 大表统计或复杂报表慢，你怎么处理？

- 面试官为什么会问这个：
这题和岗位里的 Doris / StarRocks 实时报表很贴近。
- 我的回答（结合真实项目）：
我先分清楚这是 OLTP 查询被写成报表了，还是本来就应该走分析链路。如果是前者，先做索引、SQL 改写、分批统计、预聚合；如果数据量和维度已经超出 OLTP 库适合承载的范围，那就应该把明细采集和分析查询拆开。  
我在银行项目里做过财务报表和跑批优化，真实经验就是不能指望 MySQL 一直硬扛复杂统计。数据量一上来，就得考虑异步汇总、预计算，甚至把分析查询放到更适合 OLAP 的存储里。
- 如果我没真正做过，怎么诚实但不弱地回答：
可以承认自己没落地过 Doris，但你对“为什么 MySQL 报表会慢、什么时候要分层”是有实践感知的。
- 可继续深挖的追问：
什么情况下做宽表。什么情况下做物化汇总。为什么 OLTP 不适合复杂聚合。

## Q47. 缓存穿透、击穿、雪崩怎么处理？

- 面试官为什么会问这个：
这是 Redis 基本功，但高级岗位看的是你能不能结合真实治理手段讲。
- 我的回答（结合真实项目）：
穿透是查根本不存在的数据，我会用布隆过滤器或空值缓存挡一下。击穿是热点 key 失效瞬间把数据库打爆，我会用互斥锁、逻辑过期或后台刷新。雪崩是大量 key 同时失效，我会打散过期时间、多级缓存、限流降级。  
我自己项目里虽然 Redis 更常拿来做锁、限流和去重，但这些缓存治理思路是清楚的。如果以后做广告画像或配置类热点查询，也都是这些套路。
- 如果我没真正做过，怎么诚实但不弱地回答：
这题别只背定义，至少补一句你会优先怎么落地。
- 可继续深挖的追问：
逻辑过期和物理过期怎么选。缓存空值 TTL 设多久。

## Q49. 报表系统为什么容易慢？

- 面试官为什么会问这个：
因为这类岗位会涉及实时分析和报表服务。
- 我的回答（结合真实项目）：
报表系统容易慢，核心原因有四个。第一，天然多维聚合、排序、分页，很难像 OLTP 那样走简单索引。第二，很多报表还会跨时间范围、跨业务维度 join。第三，用户会不断改口径，导致查询越来越重。第四，大家一开始往往直接复用业务库，结果把在线交易和离线统计混在一起。  
我在银行项目里就踩过这个坑，所以我现在回答这类问题，都会强调 OLTP 和 OLAP 该分层。
- 如果我没真正做过，怎么诚实但不弱地回答：
结合你处理过的慢查询和报表跑批经验来答就够了。
- 可继续深挖的追问：
报表服务和明细服务如何拆。预计算和实时计算如何取舍。

## Q50. OLTP 和 OLAP 的差别是什么？

- 面试官为什么会问这个：
这是 Doris / StarRocks 方向的基础判断题。
- 我的回答（结合真实项目）：
OLTP 更偏高并发、短事务、点查和小范围更新，关注的是低延迟和事务一致性。OLAP 更偏大范围扫描、聚合分析、多维报表，关注的是分析效率和压缩、列存、向量化这类能力。  
我真实做得更多的是 OLTP 和报表优化，所以我会补一句，当报表和统计越来越重时，就不应该继续压在 OLTP 库上，这也是 Doris / StarRocks 这类分析引擎的价值。
- 如果我没真正做过，怎么诚实但不弱地回答：
不用装成做过 OLAP 集群，只要概念清楚，并且知道为什么要分层，就够用了。
- 可继续深挖的追问：
什么场景适合实时明细。什么场景适合预聚合。

## Q52. 限流、熔断、降级分别怎么用？

- 面试官为什么会问这个：
这是稳定性建设里的高频题。
- 我的回答（结合真实项目）：
我理解这三个词是不同层次。限流是流量保护，防止系统被打穿；熔断是依赖保护，下游已经不稳了，就别继续把流量压过去；降级是业务保护，核心功能优先，非核心先让步。  
我项目里最贴近限流的是 Redis Lua 令牌桶和任务调用上限控制。最贴近熔断思路的是 Feign 失败补偿加健康检查，目标服务整体不健康就先不重放。最贴近降级的是 RAG 项目里的向量检索降级到 BM25、LLM 超时 fallback。
- 如果我没真正做过，怎么诚实但不弱地回答：
哪怕没用过 Sentinel，也可以把这三层语义讲清楚，不要混着说。
- 可继续深挖的追问：
限流阈值怎么定。熔断后如何恢复。降级后如何感知业务影响。

## Q53. 灰度发布怎么做？

- 面试官为什么会问这个：
这个岗位提到了灰度和容灾，所以要看你是否理解上线风险控制。
- 我的回答（结合真实项目）：
我理解灰度发布的关键不是“先上几台”，而是“新旧版本可区分、可观测、可快速回滚”。如果是接口或事件链路，我会先保证版本兼容，再按机器、流量比例、用户标签或业务开关逐步放量。灰度期间重点盯错误率、RT、消息积压、关键业务成功率。  
我现在项目里没有特别成熟的平台化灰度体系，但任务治理和补偿链路让我对“开关、渐进放量、回滚窗口、关键指标观察”这套思路是有实感的。
- 如果我没真正做过，怎么诚实但不弱地回答：
直接承认“没主导过成熟灰度平台”，但能讲出兼容、观察、回滚这三件事，就不会弱。
- 可继续深挖的追问：
事件 schema 变更怎么灰度。消费者新旧版本如何兼容。

## Q54. 压测怎么做？重点看哪些指标？

- 面试官为什么会问这个：
稳定性建设不能只靠猜，压测是验证取舍的关键手段。
- 我的回答（结合真实项目）：
我做压测会先定目标，不是为了跑个 TPS 数字。先明确是测接口吞吐、消息消费能力、数据库写入能力，还是下游承压点。然后准备尽量接近真实的数据模型和流量分布，分阶段压，从单机、单链路到全链路。  
指标上我会重点看 RT、吞吐、错误率、线程池、GC、CPU、内存、数据库慢 SQL、MQ 积压和 Redis 响应。  
我虽然没有做过特别大规模全链路广告压测，但做限流算法取舍、跑批优化和链路治理时，都是按这个思路验证的。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要说“用 JMeter 压一下就行”。高级岗位更看方法论。
- 可继续深挖的追问：
压测数据怎么构造。为什么压测和线上表现不一致。瓶颈定位顺序是什么。

## Q57. 怎么避免告警风暴？

- 面试官为什么会问这个：
会做告警不难，难的是告警别把人淹没。
- 我的回答（结合真实项目）：
我会从三方面做。第一，分级，关键业务失败和瞬时抖动不能一个等级。第二，聚合，相同根因的告警尽量合并，不要一台机器一个告警、一条消息一个告警。第三，抑制，设置时间窗口和持续阈值，避免瞬时尖刺就疯狂轰炸。  
我真实项目里虽然不是做统一告警平台，但做任务治理和补偿机制时，已经很强调失败留痕和状态明确，否则后面根本没法做有效告警。
- 如果我没真正做过，怎么诚实但不弱地回答：
只要你知道“不是异常就告警，而是要可操作”，这题就不会答弱。
- 可继续深挖的追问：
哪些指标需要秒级告警。哪些适合日报或巡检。

## Q58. 你有没有参与过故障复盘？

- 面试官为什么会问这个：
看你有没有从问题里沉淀机制，而不是只会临时救火。
- 我的回答（结合真实项目）：
有参与过。像银行项目里的跑批 OOM、慢 SQL 拖慢链路，或者电网项目里对失败补偿和消息幂等边界的治理，其实都不是问题过去就算了，后面都会反推怎么避免再次发生。  
我理解复盘至少要有四部分，现象、影响、根因、改进动作。最重要的是改进动作要能落地，比如分批处理、加索引、补监控、限制重试边界，而不是写一堆空话。
- 如果我没真正做过，怎么诚实但不弱地回答：
如果不是主导复盘，就说自己参与过并落了哪些改进项，不要硬说自己主持过事故复盘会。
- 可继续深挖的追问：
你印象最深的一次复盘是什么。复盘后真的改了什么。

## Q60. 如果新版本上线后问题变多，你会怎么处理？

- 面试官为什么会问这个：
这是典型上线治理题，想看你会不会先止血。
- 我的回答（结合真实项目）：
我会先判断是不是发布导致的，如果是，优先止血，能回滚就回滚，不能回滚就先关功能开关、限流或停补偿任务。止血之后再对比新旧版本差异，看是 schema 兼容问题、配置问题、线程池 / SQL / 下游调用变化，还是新逻辑引入了副作用。  
我自己的习惯是，上线后重点盯关键指标，不要等用户反馈才知道有问题。
- 如果我没真正做过，怎么诚实但不弱地回答：
即使没当过发布负责人，也可以把“先止血、再定位、后复盘”这三步讲清楚。
- 可继续深挖的追问：
什么情况下不适合立刻回滚。数据库变更怎么处理。

## Q72. 你用 AI 做过哪些代码生成、重构、排查问题、文档沉淀？

- 面试官为什么会问这个：
他想听具体实践，不是泛泛说“提高效率”。
- 我的回答（结合真实项目）：
我做过几类比较实用的。代码层面，我会让 AI 帮我先梳理老模块调用链、补充单测骨架、生成重复样板代码的初稿。排障层面，我会把错误日志、线程栈、慢 SQL 扔给 AI，让它先给出排查假设和关注点，但最后我会自己验证。文档层面，我把任务治理、MQ 幂等、RAG / DSL 链路、面试问答这些内容都沉淀成结构化文档，这也是 AI 很适合做的事情。  
在 RAG 项目里，我还会用 AI 帮我整理 bad case、改写用户规则样例、补回归 case 候选。
- 如果我没真正做过，怎么诚实但不弱地回答：
重点讲具体场景，不要只说“写代码、查 bug、写文档”。
- 可继续深挖的追问：
哪些场景 AI 帮助最大。哪些场景你明确不会交给 AI。

## Q73. 你如何避免 AI 生成代码直接进生产？

- 面试官为什么会问这个：
这题看的是工程纪律，而不是会不会用工具。
- 我的回答（结合真实项目）：
我的原则很简单，AI 可以参与起草，但不能跳过工程流程。第一，核心逻辑我一定自己 review，尤其是并发、事务、幂等、SQL 和异常边界。第二，能加单测的要补单测，至少把关键路径和 bad case 跑过。第三，要过静态检查、集成验证和必要的回归。第四，涉及安全、性能和稳定性的代码，我会更谨慎，不会因为 AI 生成得像就直接合。  
因为我自己就在做 AI 系统，所以我反而更清楚模型特别容易在边界条件上“写得很像对的”，这点必须靠工程机制兜住。
- 如果我没真正做过，怎么诚实但不弱地回答：
哪怕你现在团队还没有成熟 AI coding 流程，也可以明确讲自己的审查标准。
- 可继续深挖的追问：
有没有一次 AI 生成代码看起来对、实际上有坑。你怎么识别这种坑。
