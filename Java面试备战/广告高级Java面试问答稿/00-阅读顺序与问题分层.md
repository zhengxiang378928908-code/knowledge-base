# 广告高级 Java 面试问答稿 - 阅读顺序与问题分层

## 1. 先说结论

这类“高级 Java 开发工程师”岗位，虽然业务关键词是广告，但面试官真正想验证的核心还是五件事：

1. 你是不是能扛核心后端链路，而不只是写 CRUD。
2. 你能不能把高并发、低延迟、幂等、补偿、稳定性这些问题讲成真实项目，不是背八股。
3. 你有没有事件驱动、消息链路、异步解耦、失败恢复这类基础能力。
4. 你遇到自己没深做过的方向，比如 Kafka / Flink / Doris / 广告归因，能不能诚实回答，同时证明迁移能力。
5. 你是不是已经在用 AI 提升研发效率，而且不是“会用工具”那么浅。

你当前背景和岗位的最佳对齐方式，不是硬说自己做过广告，而是走这条主线：

> 我是 Java 后端出身，这几年主要做两条线。一条是电网业务里的任务治理、异步消息、失败补偿和稳定性建设；另一条是 AI 规则解析应用，把自然语言规则做成可回归、可校验、可迭代的后端系统。虽然行业不是广告，但底层能力和广告后台很多核心问题是相通的，比如事件链路设计、幂等去重、异步解耦、最终一致性、链路可追踪和系统稳定性。

## 4. 能力迁移口径

### 广告事件采集链路

你没做过广告点击归因，但你做过这些可以迁移的能力：

- 多源设备数据同步和高吞吐入库
- RocketMQ 异步链路、短时去重、业务幂等
- OpenFeign 失败补偿和最终一致性恢复
- 任务治理、限流、启动恢复、异常留痕

所以你可以把广告点击 / 转化链路，抽象成“高吞吐事件链路 + 幂等去重 + 异步计算 + 失败补偿 + 追踪审计”来回答。

### Kafka / Flink

你真实做的是 RocketMQ，不要装作深度做过 Kafka / Flink。

正确姿势是：

- 先承认生产里主要用 RocketMQ
- 再把消息模型、分区、消费组、幂等、积压、顺序性这些共性讲扎实
- 最后补一句自己理解 Kafka / Flink 在日志流、实时计算场景更合适，迁移成本主要在生态而不是理念

### Doris / StarRocks

你可以从报表慢 SQL、跑批统计、OLTP / OLAP 分工来切入。

不要说“我落地过大规模 Doris 集群”，要说：

> 我在生产里更多做的是 MySQL 报表优化和统计链路治理，但我理解这类场景后面如果数据量继续放大，OLTP 库不应该继续硬扛，应该把明细采集、实时聚合和分析查询分层，Doris / StarRocks 就是在这个位置上发挥价值。
