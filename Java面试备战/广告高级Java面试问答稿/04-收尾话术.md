# 收尾话术

## 一版 3 分钟自我介绍

我做 Java 后端 8 年了，前几年的主线是在银行和业务系统里做核心模块开发，主要覆盖客户准入、黑白名单校验、跑批和报表优化，也处理过 OOM、慢 SQL、CPU 飙高这类线上问题。  

这两年我在电网业务里主要做两条线。第一条是偏核心后端和稳定性建设这块，像变电 3.0 微服务体系里的任务治理、RocketMQ 异步链路、OpenFeign 失败补偿、Redis 限流和分布式锁这些，我都比较深入。这里面我主导过一个轻量级分布式任务治理平台，不只是定时执行，还包括任务注册、动态启停、手动控制、运行日志、调用上限和应用重启后的状态恢复。另一条是 AI 工程化，我做了一个电网预警规则解析平台，把业务人员写的自然语言规则转成结构化 Business DSL，里面我负责的是后端主链路，包括 Slot 抽取、混合检索、DSL 生成、多轮校验和回归验证。  

所以我给自己的定位一直很明确，主线还是 Java 后端和复杂链路治理，只是最近把 AI 能力也工程化落地了。  

我看这个岗位，虽然业务是广告后台，但它关注的点像高并发、低延迟、事件采集、幂等去重、链路追踪、稳定性建设，其实和我现在做的数据同步、异步消息、失败补偿、任务治理是同一类问题。我不是广告行业出身，这点我会坦诚，但我对这类核心后端问题是有真实经验的，而且我最近也一直在用 AI 帮助研发提效，所以我觉得自己的背景和这个岗位是比较匹配的。

## 一版“最复杂项目介绍”

如果是投这个岗位，我建议主讲变电 3.0，而不是先讲 AI 项目。  

我做过最复杂的系统，是现在的变电站生产管控平台。它不是一个单服务，而是一个 7 模块的微服务体系，里面有设备台账、在线监测、预警规则、工单流转、统计分析这些业务，同时还要处理任务调度、消息异步、远程调用补偿和规则执行。  

我在里面负责的不是普通业务接口，更多是核心治理链路。第一个是轻量级分布式任务治理平台，从 0 到 1 做了任务注册、动态启停、运行日志、手动触发、调用上限控制和启动恢复，解决 300+ 定时任务分散失控的问题。第二个是 OpenFeign 失败补偿机制，不是简单重试，而是通过 AOP 拦截失败调用，把服务标识、方法名、原始参数、错误信息完整落库，后面再由后台任务按服务健康状态补偿重放。第三个是 RocketMQ 异步告警链路治理，重点做了短时去重、业务幂等、失败重试边界和副作用隔离，避免重复发短信、重复创建工单这类问题。  

我觉得这个系统复杂，主要不是因为模块多，而是因为它同时有高吞吐数据同步、异步消息、分布式任务、幂等补偿、规则执行和线上稳定性这些问题交织在一起。这个项目也最能体现我的后端能力，因为它要求的不只是开发功能，而是设计边界、处理异常、保证链路可恢复。

## 一版“线上故障排查案例”

我比较推荐讲银行项目里那次报表链路问题，因为它很像高级后端会遇到的真实故障。  

当时的现象是，某次跑批和报表统计期间，服务 RT 明显升高，错误率也上来了，部分请求开始超时。我们先看监控，发现应用线程堆积、数据库连接池很紧张，然后继续看慢查询日志，定位到一条复杂统计 SQL 执行时间特别长，没有命中合适索引。因为这条 SQL 一慢，连接被长期占住，请求越堆越多，最后把整个链路拖慢。  

排查思路上，我当时不是一上来就盯代码，而是先分层看。先确认是不是发布问题，不是。再看应用层和数据库层，最后才落到具体 SQL。拿到 SQL 以后，我用 `EXPLAIN` 看执行计划，发现确实有全表扫描和排序开销，后面做了几件事：一是补和调整索引，二是改写 SQL，减少不必要的扫描，三是把原来一次性处理的大批量统计拆成分批处理，避免长事务和大结果集长期占内存。  

这次事情对我最大的收获是，线上故障很多时候不是单点 bug，而是“慢 SQL -> 连接池紧张 -> 请求堆积 -> RT 飙高”这种链式放大。所以后面我做项目会更重视慢查询监控、批处理边界和资源隔离。

## 一版“为什么想来做这个岗位”

我想来做这个岗位，核心有三点。  

第一，这个岗位关注的问题类型和我的主线经历很匹配。我过去这几年做的，主要就是任务治理、消息异步、幂等补偿、稳定性建设和线上问题处理，本质上都是高并发后台系统的问题。  

第二，虽然我不是广告行业背景，但广告后台里的点击采集、转化回传、去重、追踪、实时链路这些事情，在我看来本质上还是事件系统设计。我现在做的数据同步、RocketMQ 链路、补偿重放、任务平台，其实是可以迁移过来的。  

第三，我最近一直在做 AI 工程化，一方面自己会用 AI 提升代码理解、测试生成、排障和文档沉淀效率，另一方面也做过 RAG / DSL 这种真实 AI 后端服务。我看到这个岗位也强调熟练使用 AI 提升研发效率，这一点我能比较快带来实际价值。  

所以我对这个岗位的判断，不是“行业看起来不错”，而是它关注的技术问题和我想继续深耕的方向是对得上的。

## 一版“如果问到没做过 Flink / Doris / 广告归因，怎么回答”

### 如果问 Flink

我会直接说，我生产里没有深度落过 Flink，这点我不会硬说有。但我理解它在实时链路里的位置，像流式处理、窗口、状态、checkpoint、exactly-once 这些能力，都是广告和实时数据场景很核心的。我自己的经验更多是在 RocketMQ 异步链路、幂等补偿和事件系统设计上，所以虽然我不是 Flink 老手，但消息链路和业务建模这块我有基础，上手会比较快。

### 如果问 Doris / StarRocks

我会说，我没有主导过 Doris 或 StarRocks 的生产落地，但我做过报表和跑批优化，所以我很清楚 OLTP 库在复杂分析场景下为什么会吃力，也理解为什么明细采集、实时聚合和分析查询要分层。也就是说，这块我不是有直接落地经验，但我理解它解决的问题、适用场景和演进方向。

### 如果问广告归因

我会说，我不是广告行业背景，没有完整做过归因链路，这点我会坦诚。但我会把这个问题抽象回后端能力来看。广告归因本质上是多事件关联、乱序处理、幂等去重、延迟修正和口径一致性，而这些能力我虽然不是在广告里做的，但在消息链路、失败补偿、数据同步和状态管理里都做过相似问题。所以我不会装作懂具体归因模型，但我有能力把它当成事件系统快速吃透。

## 面试时的最后提醒

1. 不要把自己讲成“广告专家”，要讲成“核心后端问题我做过，而且能迁移”。
2. 不要为了迎合岗位硬编 Kafka / Flink / Doris 生产经验，诚实加迁移能力反而更强。
3. 你最稳的人设就是：`Java 后端主线 + 任务治理 / 消息可靠性 / 失败补偿 / 稳定性建设 + AI 工程化落地`。
