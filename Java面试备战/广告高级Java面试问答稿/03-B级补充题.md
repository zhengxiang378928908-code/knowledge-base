# B级补充题

## Q43. 深分页怎么优化？

- 面试官为什么会问这个：
这是数据库和报表类常见追问题。
- 我的回答（结合真实项目）：
深分页慢的核心是 `offset` 越大，数据库跳过的行越多。我一般优先用基于游标或主键的翻页，也就是记住上一页最后一个 ID，再往后查。实在要页码分页，也会尽量让排序字段和过滤字段命中索引。  
我在报表和批量处理场景里，更偏向分批扫描而不是传统深分页，因为本质上都是“不要让数据库白白跳过大量数据”。
- 如果我没真正做过，怎么诚实但不弱地回答：
这题就讲 keyset pagination 思路就够了，不需要编具体线上案例。
- 可继续深挖的追问：
排序字段不唯一怎么办。深分页和导出场景怎么区分。

## Q48. ES 适合做什么？和 MySQL 怎么分工？

- 面试官为什么会问这个：
看你是否理解不同存储的边界。
- 我的回答（结合真实项目）：
ES 更适合全文检索、多条件组合搜索、模糊匹配和聚合分析，不适合承载强事务和复杂关联。MySQL 更适合事务型业务、主数据和强一致更新。  
我早期做制图系统时，就用过 Python 抓取素材并写入 Elasticsearch 做热点词统计和检索。那个经验让我很清楚，ES 是检索引擎，不是业务真相库。
- 如果我没真正做过，怎么诚实但不弱地回答：
即使 ES 用得不深，也可以从“检索 vs 事务”这个分工来答。
- 可继续深挖的追问：
ES 为什么不适合做订单主库。ES 和向量检索怎么结合。

## Q59. SLA / SLO / SLI 你怎么理解？

- 面试官为什么会问这个：
这题是稳定性建设偏方法论的追问。
- 我的回答（结合真实项目）：
我理解 SLI 是指标，比如成功率、延迟、可用性；SLO 是团队内部目标，比如核心接口 99.9% 成功率；SLA 是对外承诺，通常更偏合同或服务承诺。  
我现在项目里没有特别正式地用这三个词管理，但我做的任务成功率、消息消费成功率、补偿队列长度、解析成功率这些，本质上都可以往 SLI / SLO 上收口。
- 如果我没真正做过，怎么诚实但不弱地回答：
直接说“没有在组织层面正式制定过 SLA，但理解这套方法，并且做过相关指标建设”就很好。
- 可继续深挖的追问：
广告采集链路你会选哪些 SLI。SLO 定太高会有什么问题。

## Q61. Java 内存模型你怎么理解？

- 面试官为什么会问这个：
高级 Java 岗经常会抽一两题并发基础。
- 我的回答（结合真实项目）：
我会把 JMM 理解成多线程下共享变量可见性、有序性和原子性的规则。它不是在讲 JVM 内存分区，而是在讲线程之间什么时候能看到彼此的写入。  
对业务开发来说，最常用的落点就是 `volatile`、`synchronized`、锁和 happens-before 规则。虽然我平时不直接跟 JMM 打交道，但线程池、并发任务和锁设计背后都离不开这个基础。
- 如果我没真正做过，怎么诚实但不弱地回答：
把可见性、有序性、原子性说清楚就够了，不要绕太多术语。
- 可继续深挖的追问：
什么是 happens-before。为什么双重检查要配合 `volatile`。

## Q62. synchronized 和 ReentrantLock 区别？

- 面试官为什么会问这个：
这是 Java 并发经典题。
- 我的回答（结合真实项目）：
`synchronized` 用起来简单，JVM 层面支持更直接，适合同步块或同步方法。`ReentrantLock` 更灵活，支持可中断、可超时、可公平锁、多个条件队列。  
我实际业务里如果只是简单互斥，不会为了炫技上 `ReentrantLock`；但涉及超时控制、可中断等待时，`Lock` 的表达力更强。这个思路和我在任务停止、中断处理上很一致，能优雅中断的地方尽量别写死。
- 如果我没真正做过，怎么诚实但不弱地回答：
不用假装线上大量用过 `ReentrantLock`，理解差异就够了。
- 可继续深挖的追问：
为什么 `synchronized` 现在也不一定慢。什么是锁优化。

## Q63. volatile 的作用是什么？

- 面试官为什么会问这个：
考你并发基本功。
- 我的回答（结合真实项目）：
`volatile` 主要保证可见性和一定程度的有序性，不能保证复合操作的原子性。它适合做状态标记、开关位、双重检查里的实例引用。  
像任务治理这类场景，如果只是标记“是否停止”“是否关闭”，`volatile` 就很合适；但如果涉及计数和复合更新，就不能只靠它。
- 如果我没真正做过，怎么诚实但不弱地回答：
把“看得见，但不等于原子”这句记住就够用了。
- 可继续深挖的追问：
为什么 `i++` 不安全。`volatile` 和锁如何选。

## Q64. ThreadLocal 有什么坑？

- 面试官为什么会问这个：
这题经常和链路追踪、线程池串用。
- 我的回答（结合真实项目）：
最大的坑是线程池复用导致数据串线程，如果用完不清理，下一个任务可能拿到上一次残留的上下文。另一个坑是弱引用 key 被回收后，value 还可能挂在线程上，长期不清理会内存泄漏。  
所以我如果用 `ThreadLocal` 放 traceId、用户上下文，用完一定会 `remove()`。
- 如果我没真正做过，怎么诚实但不弱地回答：
你不一定要大量用过 ThreadLocal，只要知道线程池环境下必须清理就够了。
- 可继续深挖的追问：
为什么在线程池里更危险。异步场景 traceId 怎么传。

## Q65. 线程池参数怎么设置？设置不好会怎样？

- 面试官为什么会问这个：
后端高并发和批处理都离不开线程池。
- 我的回答（结合真实项目）：
我会先区分 CPU 密集还是 IO 密集，再定核心线程数。队列我倾向有界，避免任务无限堆积导致 OOM。拒绝策略我更偏 `CallerRunsPolicy` 这类能自然形成反压的方式。  
设置不好最常见两个坑，一个是队列无界，最后内存打爆；一个是线程数开太大，反而导致上下文切换严重、下游被打穿。  
我在跑批和任务处理上，一直很在意分批和并发度控制，本质就是同一个问题。
- 如果我没真正做过，怎么诚实但不弱地回答：
即使没手配过很多线程池，也别用 `Executors` 默认工厂当标准答案。
- 可继续深挖的追问：
为什么不推荐 `newFixedThreadPool`。IO 密集怎么估算线程数。

## Q66. 线上 CPU 飙高你怎么排查？

- 面试官为什么会问这个：
高级 Java 岗很常见的故障排查题。
- 我的回答（结合真实项目）：
我一般会按这个顺序排查：先 `top` 看整机 CPU，定位高占用进程；如果机器上有多个 Java 进程，再用 `jps -l` 确认具体应用；然后用 `top -H -p <pid>` 找出高 CPU 线程，把线程 ID 转成十六进制，再用 `jstack` 看对应线程栈。  
拿到线程栈之后，我会继续判断到底是死循环、频繁 GC、锁竞争，还是某段业务代码本身成了热点。如果是 GC 线程高，就继续看 GC 日志和堆；如果是业务线程高，就沿着堆栈定位到具体代码；如果是锁问题，就结合线程状态看是不是有大量阻塞。  
我简历里写过处理 CPU 飙高，所以回答这题时我会强调，不能只停留在“找到线程”，最后一定要落到具体代码和修复动作。
- 如果我没真正做过，怎么诚实但不弱地回答：
就按标准排查路径答，不用假装自己天天线上打火。
- 可继续深挖的追问：
如果是 GC 频繁导致 CPU 高怎么办。如何区分用户态和内核态问题。

## Q67. Full GC 频繁怎么排查？

- 面试官为什么会问这个：
这题常和 OOM、内存泄漏一起问。
- 我的回答（结合真实项目）：
我会先看 GC 日志和堆使用趋势，判断是内存设置过小、对象创建过猛，还是确实存在泄漏。然后配合 dump 用 MAT 看大对象和引用链。  
我在银行项目里遇到过跑批一次性全量加载导致内存打高，本质不是 JVM 参数没调好，而是业务代码的处理方式有问题，后来改成分页拉取、分批处理，效果比硬调参数更直接。
- 如果我没真正做过，怎么诚实但不弱地回答：
把“先看现象，再看 dump，最后区分泄漏和容量不足”这个思路记住就行。
- 可继续深挖的追问：
什么对象最容易撑爆老年代。如何通过 GC 日志判断问题。

## Q68. 内存泄漏你怎么定位？

- 面试官为什么会问这个：
想看你会不会用 dump 和分析工具。
- 我的回答（结合真实项目）：
我会先拿堆 dump，然后用 MAT 看 dominator tree、retained size 和 GC Roots，找到谁一直持有对象没释放。常见问题像大集合长期持有、缓存没淘汰、ThreadLocal 没清理、监听器没反注册。  
我真实做过的更偏“类泄漏症状排查”，尤其是跑批全量加载和对象生命周期过长这类问题。
- 如果我没真正做过，怎么诚实但不弱地回答：
不用编具体工具链，`HeapDump + MAT + GC Roots` 这条线讲清楚就够了。
- 可继续深挖的追问：
如何区分泄漏和短时间流量高峰。线上 dump 有什么风险。

## Q69. 你做过哪些 JVM 调优？

- 面试官为什么会问这个：
一般是看你有没有真正根据业务调整，而不是只会背参数。
- 我的回答（结合真实项目）：
我做过的 JVM 调优更多是“先改代码，再调参数”。像银行跑批 OOM，核心优化是改成分页拉取和分批处理，而不是先改堆大小。参数层面我会关注堆大小、GC 日志、OOM 自动 dump、元空间这些基础配置，确保出了问题有证据。  
所以如果问“有没有特别重的 JVM 调优”，我会诚实说没有做过很极限的专职调优，但做过结合业务问题的 JVM 排查和基础优化。
- 如果我没真正做过，怎么诚实但不弱地回答：
直接守住“先业务、后参数”的思路，这比背一堆 GC 参数靠谱。
- 可继续深挖的追问：
为什么很多 JVM 问题本质是代码问题。GC 选型怎么看。

## Q70. I/O 模型、网络通信、BIO / NIO 你怎么理解？

- 面试官为什么会问这个：
通常是看基础深度，不一定会深挖到底层。
- 我的回答（结合真实项目）：
BIO 是一个连接一个线程，模型简单，但连接多了成本高。NIO 基于事件和多路复用，一个线程可以处理多个连接，更适合高并发网络场景。  
我实际业务开发层面更多是站在 Spring Boot 和中间件上工作，不会手写底层网络框架，但理解这些基础有助于理解为什么 Kafka、Redis、Netty 类组件能扛高并发。
- 如果我没真正做过，怎么诚实但不弱地回答：
不要硬装自己深做过 Netty。把 BIO / NIO 的适用场景说清楚就够了。
- 可继续深挖的追问：
什么是 select、poll、epoll。Redis 为什么命令执行还是单线程。
