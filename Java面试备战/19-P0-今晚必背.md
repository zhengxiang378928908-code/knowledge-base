## 二、任务扫描注册流程（当前 4 分 → 目标 7）

### 面试官问题

> 你的 Starter 启动的时候，任务是怎么被发现和注册的？

### 达标答法

> 我定义了一个自定义注解，上面会标任务分组、cron 表达式、负责人、最大调用次数这些治理信息。
>
> 启动时有一个扫描器，实现了 Spring 的 CommandLineRunner，容器初始化完成后自动触发。它会扫描所有 Bean 上标了这个注解的方法，把注解里的元数据提取出来。
>
> 提取完之后，跟数据库里已有的任务配置做比对——新任务就插入，已有的就更新 cron 和分组。这样每次启动，数据库里的任务清单就和代码里的注解保持同步了。后续动态启停、执行监控都是基于这张配置表来驱动的。

## 三、MQ 消费端怎么保证不重复消费？消费失败怎么处理？（当前 5 分 → 目标 7）

### 面试官常见问法

> - "你们消费端幂等怎么做的？"
> - "消息重复消费怎么防？"
> - "消费失败了怎么办？"

### 达标答法

> 我做了两层幂等，再加失败兜底：
>
> **第一层，Redis 快速去重。** 消息进来后，拿消息里的唯一 ID 往 Redis 写一个处理中的标记，设 3 分钟过期。如果标记已存在，说明正在处理或刚处理完，直接跳过。处理成功后把过期时间延长到 10 分钟，防止刚处理完标记就过期又被当成新消息。
>
> **第二层，数据库业务幂等。** Redis 去重只是快速拦截，真正的业务幂等在 service 层按业务唯一键做，比如设备 ID + 事件时间建唯一索引。insert 冲突就说明已处理过。
>
> **副作用隔离。** 落库是幂等的，但发通知不是。所以只有 insert 返回影响行数为 1（真正新插入）才触发异步通知，重试进来发现数据已存在就不会重复发。
>
> **消费失败处理：** 清掉 Redis 的处理中标记，抛异常让 MQ 重新投递。最多重试 3 次加上初始那次总共 4 次机会，超了就不再重试，避免一条坏消息把消费者卡死。
>
> 举个具体场景，比如故障跳闸消息进来，要落主记录、子事件和后果记录，还要发短信通知。落库这一步靠唯一索引保证幂等，短信只对新插入的数据发，重试不会重复通知。

## 四、HashMap put 流程（当前 4 分 → 目标 7）

### 达标答法

> **第一步，算 hash。** 拿 key 的 hashCode，高 16 位和低 16 位异或，让高位参与运算减少冲突。
>
> **第二步，定位桶。** 用 (n-1) & hash 算桶下标，位运算代替取模，所以容量必须是 2 的幂。
>
> **第三步，放元素：**
> - 桶是空的，直接放
> - 桶里第一个节点 key 相同，覆盖 value
> - 桶里是链表，遍历到尾部追加，中间发现相同 key 就覆盖
> - 桶里是红黑树，走树的插入逻辑
>
> **第四步，判断树化。** 链表追加后长度 ≥ 8，再看数组容量是否 ≥ 64。都满足才转红黑树，容量不够就先扩容。
>
> **第五步，判断扩容。** 元素总数超过容量 × 0.75，触发扩容，数组翻倍，所有元素重新定位。

### 关键数字（必背）

| 参数 | 值 |
|---|---|
| 默认容量 | 16 |
| 负载因子 | 0.75 |
| 树化阈值 | 链表 ≥ 8 且容量 ≥ 64 |
| 退化阈值 | 红黑树节点 ≤ 6 |

## 五、MySQL 索引和 B+ 树（必考八股）

### 达标答法

> **B+ 树是 InnoDB 的默认索引结构，** 它和 B 树最大的区别有两个：
>
> **第一，非叶子节点只存 key 不存数据。** 这样一个节点能放更多索引项，树更矮，查询时磁盘 IO 次数更少。一般 3-4 层的 B+ 树就能存几千万条数据。
>
> **第二，叶子节点之间有双向链表。** 范围查询的时候可以直接沿着链表遍历，不需要回到上层节点，这对 SQL 里的 BETWEEN、ORDER BY 这些操作非常友好。
>
> **聚簇索引和非聚簇索引的区别：** 主键索引是聚簇索引，叶子节点存的是完整行数据。普通索引是非聚簇索引，叶子节点只存主键值，查完还要拿着主键去主键索引再查一次，这就是回表。要避免回表就用覆盖索引，让查询字段都在索引里。

### 追问：联合索引的最左前缀原则？

> 联合索引 `(a, b, c)` 是按 a → b → c 的顺序排列的。查询时必须从最左列开始匹配。
>
> - `WHERE a = 1 AND b = 2` → 能用上 a 和 b
> - `WHERE b = 2` → 用不上索引，跳过了 a
> - `WHERE a = 1 AND c = 3` → 只能用上 a，c 用不上（中间断了 b）
> - `WHERE a > 1 AND b = 2` → a 能用上，但 a 用了范围查询后 b 就用不上了

### 追问：索引什么时候会失效？

> 1. 左模糊 `LIKE '%xxx'`
> 2. 对索引列用了函数或计算
> 3. 隐式类型转换（varchar 列用数字查）
> 4. OR 条件中有一边没索引
> 5. 违反最左前缀原则
> 6. 优化器判断全表扫描比走索引更快

## 六、Redis 缓存穿透、击穿、雪崩（必考八股）

### 达标答法

> **缓存穿透：** 查的数据在数据库里根本不存在，缓存里自然也没有，每次请求都打到数据库。
>
> 解决方案有两个：一是布隆过滤器，在缓存前加一层拦截；二是缓存空值，设个短过期时间比如 2-5 分钟。
>
> **缓存击穿：** 某个热点 key 刚好过期，大量并发请求同时打到数据库。
>
> 解决方案：一是用互斥锁重建缓存，第一个请求用 `SET NX PX` 拿到锁去查数据库、更新缓存，其他请求等一下或者返回旧值；二是逻辑过期，key 永不过期，但 value 里带一个过期时间字段，发现过期了异步更新。
>
> **缓存雪崩：** 大量 key 同时过期，或者 Redis 整体宕机。
>
> 解决方案：第一，过期时间加随机偏移；第二，热点数据不设过期时间；第三，Redis 宕机做降级限流，本地缓存兜底。

### 追问：你项目里哪里用到了？

> 我的项目里用 Redis 做短时去重和设备级计数（每日消息上限 700），这些 key 都设了 TTL。为了防止 Redis 异常导致所有消息被拦截或全部放行，我做了 fail-open 设计——Redis 不可用时默认放行消息。这其实就是缓存雪崩场景下的一种降级策略。

## 七、Spring 事务传播机制和失效场景（必考八股）

### 达标答法

> **@Transactional 的默认传播行为是 REQUIRED，** 如果当前已经有事务就加入，没有就新建。
>
> **最常考的三种传播行为：**
>
> - **REQUIRED（默认）：** 加入当前事务，没有就新建
> - **REQUIRES_NEW：** 不管有没有事务都新建独立事务。比如记操作日志，不想因为主业务回滚把日志也回滚了
> - **NESTED：** 创建保存点，内层回滚只回到保存点，不影响外层
>
> **@Transactional 什么时候失效？五种：**
>
> 1. **方法不是 public 的。** 非 public 方法代理拦截不到
> 2. **同类自调用。** 走的是 this 调用不是代理对象
> 3. **异常被 catch 吞掉了。** 默认只对 RuntimeException 和 Error 回滚
> 4. **数据库引擎不支持事务。** 比如 MyISAM
> 5. **传播行为设置不当。** 比如 NOT_SUPPORTED 挂起当前事务

### 追问：你项目里怎么用事务的？

> 故障跳闸消息消费场景，落主记录和落子事件在同一个事务里，要么都成功要么都回滚。但发短信通知是异步的，放在事务外面，不会因为通知失败导致整个事务回滚。先幂等写入，再根据写入结果决定是否触发副作用。

## 八、线程池（必考八股）

### 达标答法：七个核心参数

> 1. **corePoolSize**：核心线程数，即使空闲也不会回收
> 2. **maximumPoolSize**：最大线程数
> 3. **keepAliveTime**：非核心线程空闲多久后回收
> 4. **TimeUnit**：存活时间的单位
> 5. **workQueue**：任务等待队列
> 6. **threadFactory**：线程工厂，可以自定义线程名
> 7. **handler**：拒绝策略

### 达标答法：执行流程

> 第一步，看核心线程有没有空闲的，有就直接执行。
>
> 第二步，核心线程都在忙，任务进等待队列排队。
>
> 第三步，队列也满了，创建非核心线程来处理，但不超过最大线程数。
>
> 第四步，线程数已经到最大、队列也满了，触发拒绝策略。
>
> **注意：不是先创建到最大线程数再入队列，而是先入队列再创建非核心线程。**

### 达标答法：为什么不用 Executors

> `newFixedThreadPool` 和 `newSingleThreadExecutor` 用的是无界队列，任务堆积起来能 OOM。`newCachedThreadPool` 最大线程数是 Integer.MAX_VALUE，可能创建几万个线程。实际项目中都是手动创建 ThreadPoolExecutor，队列一定用有界的。

### 达标答法：项目里怎么用的

> **第一个，数据同步任务里的并发拉取。** 设备 ID 分批，每批 100 台，线程池并发执行，Future 收集结果。IO 密集型配置，核心线程数 CPU 核数 × 2，拒绝策略 CallerRunsPolicy。
>
> **第二个，消息配对里的异步检查。** 隔离开关遥信配对用 @Async 异步执行，底层是 Spring 线程池。

### 追问：四种拒绝策略？

> - **AbortPolicy**：直接抛异常（默认）
> - **CallerRunsPolicy**：让提交线程自己执行（限流反压）
> - **DiscardPolicy**：静默丢弃
> - **DiscardOldestPolicy**：丢最老的任务

### 追问：CPU 密集 vs IO 密集？

> CPU 密集型：核心线程数 = CPU 核数 + 1
> IO 密集型：核心线程数 = CPU 核数 × 2
