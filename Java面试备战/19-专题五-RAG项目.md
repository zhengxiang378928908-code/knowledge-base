# 专题五：RAG 项目

> 这是你的差异化标签，面试官问"AI 项目怎么做的"就从这里接。
> 这条线的逻辑：为什么不直接调 LLM → Slot 怎么抽取 → 检索怎么做 → 回归怎么判断
> 不需要讲太多算法细节，重点讲工程思路和设计判断。

---

## 入口问题："你的 AI 项目具体做了什么？和直接调大模型有什么区别？"

### 完整话术（直接背）

> 这个项目的核心是把业务人员用自然语言写的电网预警规则转成结构化 DSL，用于支撑后续的设备预警映射和执行。
>
> 为什么不直接调大模型？因为电网领域有几十个设备类型、几十种业务场景、几百种监测指标，它们之间有特定的组合约束，不能随意搭配。我们早期试过直接让 LLM 生成 DSL，准确率不够，经常出现实体识别错误、指标搭配不合法的情况，后续执行链路根本跑不通。
>
> 所以我做了一套编排链路，核心思路是不把所有事情都交给 LLM，而是让规则、词典和 LLM 分工。整个流程大概分几步：
>
> 第一步，文本标准化，全角转半角、去噪声。第二步，多源 Slot 抽取，先用正则抽数字、电压、时间这些结构化信息，再用 AC 自动机基于别名词典提取设备、场景、指标，然后如果存在多个候选实体就根据语义距离、共现模式和实体特异性做消歧，最后把这些结果打包发给 LLM 做最终的槽位判定。第三步，Slot 校验，检查实体、场景、指标是不是完整的、合法的，不完整就触发多轮澄清让用户补充。第四步，混合检索，用 BM25 加 pgvector 向量检索召回相似的规则范文。第五步，用检索到的范文作为 few-shot 示例让 LLM 生成 DSL。第六步，DSL 校验，检查结构是不是合法、指标引用对不对。第七步，回译确认，把 DSL 翻译回自然语言让用户确认。
>
> 这不是一个调一次 API 就完的 demo，而是一个有状态管理、有多轮澄清、有校验兜底、有回归基线的 AI 后端系统。

---

## 追问："Slot 抽取具体怎么分工的？为什么不全交给 LLM？"

### 完整话术

> 因为不同类型的信息适合不同的抽取方式。
>
> 数字、电压等级、时间窗口这些结构化信息，正则表达式又快又准。比如"110kV 及以上"用正则直接提取成电压大于等于 110kV，"近 3 年"提取成时间窗口 3 年。这些交给 LLM 反而容易出错，而且多一次 API 调用就多几秒延迟。
>
> 设备名称、场景名称这些有大量同义词和行业缩写的，用 AC 自动机基于别名词典做多模式匹配。比如"断路器"、"开关"、"SF6 开关"都映射到同一个实体。这一步速度很快，毫秒级。
>
> 如果前面识别出多个候选实体，比如"油"既可能是变压器的油温也可能是油色谱，就需要根据上下文消歧。这一步我会看语义距离、实体在文本中的共现模式和特异性来选。
>
> LLM 负责的是最后的语义综合判断——前面几层识别出的结果打包成 JSON 发给 LLM，让它做最终的槽位确认和补全。temperature 设成 0 保证确定性输出，超时 8 秒，超时就 fallback 到纯规则结果。
>
> 这样分工的好处是：结构化信息走规则又快又准，语义理解走 LLM 兜底，既控制了延迟也控制了错误率。

## 追问："混合检索怎么做的？为什么不只用向量检索？"

### 完整话术

> 电网领域有两种情况必须同时处理。
>
> 第一种是关键词完全一样但业务含义不同。比如"SF6 泄漏"和"油气泄漏"，向量检索觉得它们语义相近可能搞混，但 BM25 靠精确关键词匹配能区分开。
>
> 第二种是用词不同但含义一致。比如"超过额定值"和"大于标准值"，关键词不同 BM25 搜不到，但向量检索靠语义相似性能找到。
>
> 所以两路互补。我用 BM25 和 pgvector 分别检索，然后用 RRF 算法融合。RRF 的核心思想是两路检索分别给每个文档一个排名，两路都排前面的文档综合得分最高。k 参数取 60，这是一个排名平滑参数，不是权重系数。k 越大排名差异的影响越平滑。我在 20 到 100 之间调过，60 在我们大概 300 条规则语料上效果最好。

## 追问："改了 prompt 之后怎么判断能不能上线？"

### 完整话术

> 我们有一套回归基线，当前是 68 条核心 case。每次改 prompt 或者调链路参数，都要跑一遍回归集。
>
> 但不是说总准确率提升了就能上。比如 68 条里 3 条从对变错了、5 条从错变对了，表面上净增 2 条看着不错。但我会先看变错的 3 条是什么——如果是断路器、主变这种主力设备的核心场景变错了，哪怕整体准确率提升了也不能直接上，得先修。
>
> 然后分析变错的原因，是 prompt 改动影响了意图识别，还是检索召回的范文变了。搞清原因才知道能不能定向修复。
>
> 如果变错的都是边缘场景，变对的覆盖了更重要的业务场景，那可以上。同时把那 3 条加入回归基线，确保下次迭代不退化。
>
> 准确率是门槛，但最终决策要看变错的 case 影响面有多大。

## 追问："这句话有什么典型难点？"

> （面试官可能给你一句规则让你现场分析）

### 完整话术

> 比如"110kV 及以上断路器 SF6 气体泄漏年超过 50mL 且运行年限超过 15 年的设备生成预警"这句话，有一个典型的 bad case——"运行年限超过 15 年"很容易被当成时间窗口来处理，但它其实是设备属性过滤条件，应该放在设备筛选条件里而不是指标条件里。我们在 DSL 校验阶段会检查这种归类错误，发现位置不对会自动纠正。
