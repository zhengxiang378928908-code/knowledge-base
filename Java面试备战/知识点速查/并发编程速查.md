# Java 并发编程

## 线程基础

### 创建线程的方式

1. 继承 `Thread`
2. 实现 `Runnable`
3. 实现 `Callable`（有返回值）
4. 线程池 `ExecutorService`

### 线程状态

```
NEW → RUNNABLE ↔ BLOCKED / WAITING / TIMED_WAITING → TERMINATED
```

### sleep vs wait

| 对比 | sleep | wait |
|------|-------|------|
| 所属类 | Thread | Object |
| 释放锁 | 不释放 | 释放 |
| 唤醒 | 超时自动恢复 | notify/notifyAll |
| 使用条件 | 任意位置 | synchronized 块内 |

## synchronized

- **修饰方法**：锁对象是 this（实例方法）或 Class（静态方法）
- **修饰代码块**：锁指定对象

### 锁升级（Java 6+）

```
无锁 → 偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁（Monitor）
```

## volatile

- **可见性**：修改后立即刷新主内存，其他线程立即可见
- **有序性**：禁止指令重排序（内存屏障）
- **不保证原子性**：`i++` 不是原子操作

### 经典应用：双重检查锁单例

```java
public class Singleton {
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

## JUC 核心类

### ReentrantLock

```java
Lock lock = new ReentrantLock();
lock.lock();
try {
    // 临界区
} finally {
    lock.unlock();
}
```

vs synchronized：支持公平锁、可中断、超时、条件变量（Condition）

### AQS（AbstractQueuedSynchronizer）

JUC 的核心基础框架：

- 内部维护一个 `volatile int state` 和 FIFO 等待队列
- ReentrantLock、Semaphore、CountDownLatch 等都基于 AQS
- 独占模式（ReentrantLock）vs 共享模式（Semaphore）

### 常用同步工具

| 工具 | 用途 |
|------|------|
| CountDownLatch | 等待 N 个任务完成 |
| CyclicBarrier | N 个线程互相等待到达屏障点 |
| Semaphore | 限制并发数量 |
| Exchanger | 两个线程交换数据 |

## 线程池

### 核心参数

```java
new ThreadPoolExecutor(
    corePoolSize,    // 核心线程数
    maximumPoolSize, // 最大线程数
    keepAliveTime,   // 空闲线程存活时间
    unit,            // 时间单位
    workQueue,       // 任务队列
    threadFactory,   // 线程工厂
    handler          // 拒绝策略
);
```

### 执行流程

1. 核心线程未满 → 创建新线程
2. 核心线程满 → 放入队列
3. 队列满 → 创建非核心线程（不超过 max）
4. 全满 → 执行拒绝策略

### 拒绝策略

- `AbortPolicy`：抛异常（默认）
- `CallerRunsPolicy`：调用者线程执行
- `DiscardPolicy`：静默丢弃
- `DiscardOldestPolicy`：丢弃队首任务

### 为什么不推荐 Executors？

- `newFixedThreadPool`：无界队列，可能 OOM
- `newCachedThreadPool`：无限创建线程，可能 OOM
- `newSingleThreadExecutor`：无界队列，同上

> 推荐手动 `new ThreadPoolExecutor()` 明确参数。

## CAS 与原子类

- CAS = Compare And Swap（比较并交换）
- 底层依赖 `Unsafe.compareAndSwapInt`
- ABA 问题 → 用 `AtomicStampedReference`

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet(); // 原子自增
```

## ThreadLocal

- 线程本地变量，每个线程独立副本
- 常用于存 userId、traceId、数据库连接
- **内存泄漏**：ThreadLocalMap 的 key 是弱引用，value 是强引用 → 用完必须 `remove()`

## CompletableFuture

### 核心用法

```java
// 异步执行
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    return queryDB();
}, executor);

// 链式处理
future.thenApply(result -> process(result))     // 同步转换
      .thenAccept(result -> save(result))        // 消费结果
      .exceptionally(ex -> handleError(ex));     // 异常处理

// 组合多个异步任务
CompletableFuture.allOf(future1, future2, future3).join();  // 等待全部完成
CompletableFuture.anyOf(future1, future2).join();           // 任一完成即可
```

### 实际场景

```java
// 并行调用多个服务，汇总结果
CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(() -> userService.getUser(id));
CompletableFuture<List<Order>> orderFuture = CompletableFuture.supplyAsync(() -> orderService.getOrders(id));

CompletableFuture.allOf(userFuture, orderFuture).join();
UserVO vo = new UserVO(userFuture.get(), orderFuture.get());
```

**面试话术**：CompletableFuture 是 Java 8 提供的异步编程工具，支持链式调用和组合。比如需要并行调用用户服务和订单服务，用 allOf 等待两个都完成再汇总，比串行调用快一倍。

## 线程池参数设置经验

### CPU 密集型

- 特点：计算多、IO 少
- 核心线程数 = **CPU 核数 + 1**
- 队列：用有界队列

### IO 密集型

- 特点：IO 等待多（数据库、网络调用）
- 核心线程数 = **CPU 核数 × 2**（或根据 IO 等待比例调整）
- 更精确公式：`线程数 = CPU 核数 × (1 + IO 耗时 / CPU 耗时)`

### 实际项目中的建议

```java
// 不要用 Executors，手动创建
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    8,                            // corePoolSize（8 核机器，IO 密集）
    16,                           // maximumPoolSize
    60, TimeUnit.SECONDS,         // 空闲线程存活 60 秒
    new LinkedBlockingQueue<>(1000), // 有界队列，防止 OOM
    new ThreadFactoryBuilder().setNameFormat("biz-pool-%d").build(),
    new CallerRunsPolicy()        // 拒绝策略：调用者执行，起到限流作用
);
```

**面试话术**：CPU 密集型设置核心线程数为 CPU 核数 +1，IO 密集型设置为 CPU 核数 ×2。实际项目中不用 Executors，手动创建线程池，队列一定要有界防止 OOM，拒绝策略推荐 CallerRunsPolicy 起到限流效果。

## 死锁

### 四个必要条件

1. **互斥**：资源同时只能一个线程持有
2. **占有且等待**：持有资源的同时等待其他资源
3. **不可抢占**：不能强行夺取别人的资源
4. **循环等待**：线程之间形成环状等待链

### 排查方式

```bash
# 1. 用 jstack 查看线程 dump
jstack <pid> | grep -A 20 "deadlock"

# 2. jconsole / VisualVM 的线程面板可以自动检测死锁

# 3. Arthas
thread -b  # 找出阻塞其他线程的线程
```

### 避免死锁

- **固定加锁顺序**：所有线程按同一顺序获取锁
- **超时机制**：`tryLock(timeout)` 获取不到就放弃
- **减小锁粒度**：缩小 synchronized 范围

**面试话术**：死锁需要四个条件同时成立。排查用 jstack 看线程 dump，搜 deadlock 关键字就能定位。预防主要靠固定加锁顺序和使用 tryLock 超时机制。

## 高频面试题

1. **synchronized 和 ReentrantLock 区别？**（synchronized 是关键字自动释放，ReentrantLock 是 API 手动释放；Lock 支持公平锁、可中断、超时、多条件变量）
2. **volatile 能保证原子性吗？**（不能，只保证可见性和有序性。i++ 是读-改-写三步，需要用 AtomicInteger）
3. **线程池参数怎么设置？CPU 密集 vs IO 密集？**（CPU 密集：核数+1；IO 密集：核数×2；队列必须有界）
4. **AQS 原理？**（volatile state + CAS 修改 + FIFO 等待队列；独占模式用于 ReentrantLock，共享模式用于 Semaphore/CountDownLatch）
5. **ThreadLocal 内存泄漏原因和解决方案？**（key 是弱引用会被回收，value 是强引用不会，导致 value 泄漏。用完必须 remove()）
6. **CompletableFuture 怎么用？和 Future 的区别？**（支持链式调用、组合、异常处理，不用阻塞等待结果）
7. **死锁怎么排查和预防？**（jstack 看 dump、固定加锁顺序、tryLock 超时）
8. **线程池的执行流程？为什么不推荐 Executors？**（核心线程→队列→非核心线程→拒绝策略；Executors 用无界队列可能 OOM）
