# 微服务

## 核心概念

微服务是将单体应用拆分为一组小型、独立部署的服务，每个服务围绕特定业务能力构建。

### 优缺点

| 优点 | 缺点 |
|------|------|
| 独立部署和扩展 | 分布式复杂性 |
| 技术栈灵活 | 网络延迟和故障 |
| 团队自治 | 数据一致性难保证 |
| 故障隔离 | 运维成本高 |

## 服务注册与发现（Nacos）

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
```

- 服务启动时注册到 Nacos
- 消费者从 Nacos 拉取服务列表
- 心跳检测，自动剔除不健康实例

## 网关（Spring Cloud Gateway）

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/user/**
          filters:
            - StripPrefix=1
```

核心功能：路由转发、限流、鉴权、日志、跨域

## 熔断降级（Sentinel）

三种策略：
1. **慢调用比例**：RT 超过阈值的比例触发熔断
2. **异常比例**：异常比例超阈值触发
3. **异常数**：异常数超阈值触发

状态机：`CLOSED → OPEN → HALF_OPEN → CLOSED`

## 分布式事务

### 问题

微服务各自有独立数据库，跨服务调用如何保证数据一致性？

### 解决方案

| 方案 | 一致性 | 性能 | 适用场景 |
|------|--------|------|----------|
| 2PC（XA） | 强一致 | 低 | 银行、支付 |
| TCC | 最终一致 | 中 | 订单、库存 |
| SAGA | 最终一致 | 高 | 长流程业务 |
| 本地消息表 | 最终一致 | 高 | 通用 |
| RocketMQ 事务消息 | 最终一致 | 高 | 异步场景 |

### Seata AT 模式

1. 一阶段：各分支执行本地事务 + undo log
2. 二阶段提交：删除 undo log
3. 二阶段回滚：根据 undo log 反向补偿

---

## CAP 定理

分布式系统最多同时满足两个：

- **C（一致性）**：所有节点数据一致
- **A（可用性）**：每次请求都能得到响应
- **P（分区容忍）**：网络分区时系统仍能运行

> 实际选择：CP（ZooKeeper）或 AP（Nacos 默认、Eureka）

## 限流算法

| 算法 | 原理 | 特点 |
|------|------|------|
| **计数器（固定窗口）** | 固定时间窗口内计数 | 简单，但有临界突刺问题 |
| **滑动窗口** | 窗口细分为多个小格子滑动 | 解决临界问题，Sentinel 默认 |
| **漏桶** | 请求进桶，固定速率流出 | 流量平滑，但不能应对突发 |
| **令牌桶** | 固定速率生成令牌，请求取令牌 | 允许一定突发流量，Guava RateLimiter |

**面试话术**：固定窗口有临界突刺，滑动窗口解决了这个问题。漏桶严格平滑但不能突发，令牌桶允许一定突发所以更常用。Sentinel 用滑动窗口统计 + 多种策略限流。

## 高频面试题

1. **微服务怎么拆分？拆分原则是什么？**
   - 按业务领域拆（DDD 限界上下文）、高内聚低耦合、独立数据库、独立部署
   - 不要拆太细，2-3 个人能维护一个服务为宜

2. **服务间怎么通信？RPC vs REST？**
   - REST（HTTP + JSON）：简单通用，跨语言方便，性能一般（OpenFeign）
   - RPC（二进制协议）：性能高，强类型，但耦合度高（Dubbo、gRPC）
   - 内部服务间追求性能用 RPC，对外暴露用 REST

3. **分布式事务方案比较？**（2PC 强一致性能低；TCC 最终一致需要写补偿逻辑；SAGA 长流程；本地消息表通用；RocketMQ 事务消息异步场景）

4. **服务雪崩怎么防止？**
   - 超时机制：设置合理的调用超时
   - 熔断：Sentinel / Resilience4j，错误率达到阈值后快速失败
   - 降级：核心功能保障，非核心功能返回兜底数据
   - 限流：防止流量暴增压垮服务
   - 隔离：线程池隔离 / 信号量隔离

5. **CAP 和 BASE 理论？**
   - CAP：分布式系统最多满足两个（P 必选，所以是 CP 或 AP）
   - BASE：基本可用（Basically Available）、软状态（Soft State）、最终一致（Eventually Consistent）
   - BASE 是 CAP 中 AP 的补充，放弃强一致性，追求最终一致性

6. **网关的作用？怎么实现限流？**
   - 作用：路由转发、负载均衡、限流、鉴权、日志、跨域、灰度发布
   - 限流：Gateway + Sentinel / Redis + Lua（令牌桶）/ RequestRateLimiter 过滤器
