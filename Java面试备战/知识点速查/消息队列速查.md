# 消息队列（RocketMQ / Kafka）

## RocketMQ 架构

```
Producer → Broker (Master/Slave) → Consumer
               ↑
          NameServer（路由注册，无状态）
```

- **NameServer**：存储 Broker 路由信息，Broker 每 30 秒心跳
- **Broker**：消息存储转发，CommitLog 顺序写入保证性能
- **存储结构**：CommitLog（消息数据）→ ConsumeQueue（消费索引）→ IndexFile（按 key/时间查询）

## Kafka 架构

```
Producer → Broker Cluster → Consumer Group
               ↑
          ZooKeeper / KRaft（元数据管理）
```

- **Broker**：消息存储节点，集群部署
- **Topic**：逻辑分类，分为多个 Partition
- **Partition**：物理分片，顺序写入，是并行度的基本单位
- **Consumer Group**：组内每个 Partition 只被一个消费者消费

## RocketMQ vs Kafka

| 特性 | RocketMQ | Kafka |
|------|----------|-------|
| 语言 | Java | Scala + Java |
| 延迟消息 | 原生支持 | 不支持 |
| 事务消息 | 原生支持（Half Message） | 支持但更复杂 |
| 消息查询 | 支持按 key / 时间查询 | 不支持 |
| 消息回溯 | 按时间点回溯 | 按 offset 回溯 |
| 吞吐量 | 十万级 | 百万级 |
| 消息顺序 | Queue 级别有序 | Partition 级别有序 |
| 元数据管理 | NameServer（无状态，轻量） | ZooKeeper / KRaft |
| 适用场景 | 业务消息、事务、延迟任务 | 大数据、日志流、流计算 |

**面试话术**：业务系统用 RocketMQ，因为需要延迟消息、事务消息、消息查询这些特性，可靠性要求高于吞吐量。大数据场景用 Kafka，吞吐量更高，和 Flink/Spark 生态集成更好。

## 消息类型（RocketMQ）

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| 普通消息 | 同步/异步/单向发送 | 一般业务通知 |
| 顺序消息 | 同一业务 ID 路由到同一 Queue | 订单状态流转 |
| 延迟消息 | 固定延迟级别（1s~2h） | 超时取消、延迟通知 |
| 事务消息 | Half Message + 本地事务 + 回查 | 分布式事务最终一致性 |

### 事务消息流程

```
Producer 发送 Half Message（Consumer 不可见）
  → Broker 确认
  → Producer 执行本地事务
  → 成功：Commit（消息可见）/ 失败：Rollback（消息丢弃）
  → 超时未回复：Broker 定期回查 Producer
```

## Kafka 核心机制

### Partition 与消费者的关系

- 一个 Partition 只能被同一 Consumer Group 内的一个消费者消费
- 消费者数 > Partition 数 → 多余消费者空闲
- 想提高并行度 → 增加 Partition 数量

### ISR 机制（In-Sync Replicas）

- 每个 Partition 有一个 Leader + 多个 Follower
- ISR：与 Leader 保持同步的副本集合
- Producer 写入 Leader，Follower 从 Leader 拉取同步
- `acks=all`：等 ISR 中所有副本确认才返回成功（最可靠）
- Follower 落后太多 → 踢出 ISR → 追上后重新加入

### acks 参数

| acks | 含义 | 可靠性 | 性能 |
|------|------|--------|------|
| 0 | 不等确认 | 最低，可能丢消息 | 最高 |
| 1 | Leader 确认 | 中等，Leader 挂了可能丢 | 中等 |
| all(-1) | ISR 全部确认 | 最高 | 最低 |

### Consumer Rebalance（重平衡）

**触发条件**：
- 消费者加入/退出 Consumer Group
- Partition 数量变化
- 消费者心跳超时

**影响**：重平衡期间该 Group 暂停消费

**优化**：
- 合理设置心跳和超时（`session.timeout.ms`、`heartbeat.interval.ms`）
- 避免消费处理时间过长导致被误判为掉线

## 消息可靠性（防丢失）

### 三个环节通用思路

| 环节 | RocketMQ | Kafka |
|------|----------|-------|
| Producer → Broker | 同步发送 + 重试 | `acks=all` + 重试 |
| Broker 存储 | 同步刷盘 + 主从同步复制 | 副本机制（`replication.factor ≥ 3`） |
| Broker → Consumer | 消费成功才提交 offset / 失败重试 | 手动提交 offset（`enable.auto.commit=false`） |

**面试话术**：消息丢失从三个环节分析。Producer 端用同步发送加重试；Broker 端 RocketMQ 用同步刷盘加主从同步，Kafka 用多副本加 acks=all；Consumer 端处理成功后再提交 offset，失败走重试机制。

## 重复消费与幂等

### 为什么会重复

- Consumer 处理成功但 offset 提交失败
- Rebalance 导致 offset 回退
- Producer 重试导致消息重复发送

### 幂等方案

| 方案 | 做法 | 适用场景 |
|------|------|----------|
| 数据库唯一约束 | 业务唯一 ID 做 unique key | 插入类操作 |
| Redis 去重 | `SET msgId NX EX` 消费前检查 | 通用方案 |
| 状态机 | 只有正确状态才流转，重复消息无法改变已完成状态 | 有状态流转的业务 |

## 消息积压

**短期应急**：
1. 增加消费者实例（不超过 Queue/Partition 数）
2. 跳过非关键消息
3. 消费者批量消费

**长期优化**：
1. 消费者内部多线程处理
2. 增加 Queue/Partition 数量
3. 优化消费逻辑（异步 IO、批量写库）

**面试话术**：先看是消费者挂了还是处理太慢。紧急情况先加消费者实例横向扩容，注意消费者数不能超过 Partition 数。长期要优化消费逻辑，比如批量写库、异步处理，或者增加 Partition 数提高并行度。

## 消息顺序性

### RocketMQ

- 全局有序：Topic 只设 1 个 Queue（牺牲吞吐）
- 分区有序：同一业务 ID hash 到同一 Queue + Consumer 用 `MessageListenerOrderly`

### Kafka

- Partition 内有序，跨 Partition 无序
- 同一业务 key 的消息路由到同一 Partition
- 单 Partition 单消费者线程保证顺序

**面试话术**：RocketMQ 和 Kafka 都只保证分区级别有序。实际业务中用分区有序就够了，把同一业务 ID 的消息路由到同一个 Queue 或 Partition，消费端单线程顺序消费。

## 高频面试题

1. **RocketMQ 和 Kafka 怎么选？**（业务消息选 RocketMQ，大数据日志选 Kafka）
2. **如何保证消息不丢？**（Producer 同步发送+重试、Broker 同步刷盘+多副本、Consumer 成功后提交 offset）
3. **如何保证消息不重复消费？**（幂等：数据库唯一约束 / Redis 去重 / 状态机）
4. **消息积压怎么处理？**（加消费者实例 + 优化消费逻辑 + 增加 Partition）
5. **如何保证消息顺序？**（同一业务 key 路由到同一分区，消费端单线程）
6. **RocketMQ 事务消息原理？**（Half Message + 本地事务 + Broker 回查）
7. **Kafka ISR 机制？**（与 Leader 同步的副本集合，acks=all 时等 ISR 全部确认）
8. **Kafka Rebalance 触发条件和影响？**（消费者增减/Partition 变化/心跳超时，期间暂停消费）
