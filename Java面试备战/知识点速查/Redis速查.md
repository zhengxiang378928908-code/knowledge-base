# Redis

## 数据结构

| 类型 | 底层实现 | 应用场景 |
|------|----------|----------|
| String | SDS | 缓存、计数器、分布式锁 |
| List | quicklist（节点通常为 listpack） | 消息队列、时间线 |
| Hash | listpack / hashtable | 对象存储 |
| Set | hashtable / intset | 标签、交集并集 |
| ZSet | listpack / skiplist + hashtable | 排行榜、延迟队列 |
| Bitmap | String | 签到、布隆过滤器 |
| HyperLogLog | 概率算法 | UV 统计 |
| Stream | Radix Tree | 消息队列（类 Kafka） |

> 注：旧版本资料常写 `ziplist`，新版本 Redis 更常见的是 `listpack`。

## 持久化

### RDB（快照）

- `save`：阻塞主线程
- `bgsave`：fork 子进程，COW 机制
- 优点：恢复快、文件紧凑
- 缺点：可能丢失最后一次快照后的数据

### AOF（追加日志）

- 记录每条写命令
- 刷盘策略：`always` / `everysec`（推荐） / `no`
- AOF 重写：压缩日志文件

### 混合持久化（Redis 4.0+）

AOF 重写时前半段用 RDB 格式，后半段用 AOF 增量。

## 缓存三大问题

### 缓存穿透（查不存在的数据）

- 布隆过滤器拦截
- 缓存空值（设短过期时间）

### 缓存击穿（热点 key 过期）

- 互斥锁（`SET NX PX`）重建缓存
- 逻辑过期（不设 TTL，异步更新）

### 缓存雪崩（大量 key 同时过期）

- 过期时间加随机偏移
- 多级缓存（本地缓存 + Redis）
- 熔断降级

## 分布式锁

```text
// 加锁
SET lock_key unique_value NX PX 30000

// 释放（Lua 脚本保证原子性，不要先 GET 再 DEL）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
end
```

### Redisson

- 看门狗自动续期
- 可重入锁
- 公平锁
- RedLock（多节点）

## 过期淘汰策略

### 过期策略

- **惰性删除**：访问时检查是否过期
- **定期删除**：定时随机检查

### 淘汰策略（内存满时）

- `noeviction`：不淘汰，写入报错
- `allkeys-lru`：全局 LRU（推荐）
- `volatile-lru`：设了过期时间的 key LRU
- `allkeys-lfu`：全局 LFU
- `allkeys-random`：随机淘汰

## 线程模型

**Redis 为什么快？**
- 纯内存操作
- 单线程执行命令，无锁竞争、无上下文切换
- IO 多路复用（epoll）同时监听大量连接
- 高效数据结构（SDS、跳表、hashtable）

**Redis 6.0 多线程**
- 网络 IO（读请求 / 写响应）用多线程处理
- 命令执行仍然是单线程（保证原子性）
- 面试话术：瓶颈不在 CPU 而在网络 IO，所以只对网络层做多线程，命令执行不变，既提升吞吐又不引入并发问题

## 缓存与数据库一致性

### Cache Aside 模式（最常用）

```
读：先查缓存 → 命中返回 → 未命中查 DB → 写入缓存
写：先更新 DB → 再删除缓存
```

**为什么是删缓存而不是更新缓存？**
- 避免并发写导致缓存和 DB 不一致
- 删除是幂等的，更简单安全

### 延迟双删

```
删缓存 → 更新 DB → 延迟一段时间（如 500ms）→ 再删一次缓存
```

- 解决：先删缓存后、DB 还没更新完时，另一个请求读到旧数据写回缓存
- 延迟时间 > 一次读请求的耗时即可

### 强一致场景

- 用 Canal 监听 binlog，异步删除/更新缓存
- 或加分布式锁串行化读写（牺牲性能）

## 大 key / 热 key

### 大 key

**判断标准**：String > 10KB，集合类元素 > 5000 个

**发现方式**：
- `redis-cli --bigkeys`
- `MEMORY USAGE key`

**危害**：阻塞主线程、网络带宽打满、主从同步延迟

**处理方案**：
- 拆分：大 Hash 按字段分组拆成多个小 Hash
- 压缩：String 类型用 gzip 压缩后存储
- 删除：用 `UNLINK`（异步删除）代替 `DEL`

### 热 key

**发现方式**：
- `redis-cli --hotkeys`（需开启 LFU 淘汰策略）
- 业务侧统计访问频次

**处理方案**：
- 本地缓存（Caffeine / Guava Cache）兜底，减少 Redis 压力
- 读写分离 + 多从节点分摊读流量
- key 加随机后缀分散到多个节点：`hot_key_1`, `hot_key_2`...

## 事务 vs Lua 脚本

| 对比项 | 事务（MULTI/EXEC） | Lua 脚本 |
|--------|-------------------|----------|
| 原子性 | 命令排队批量执行，不保证回滚 | 整个脚本原子执行 |
| 错误处理 | 单条失败其他继续执行 | 脚本报错则全部不生效 |
| 灵活性 | 不能用前一条命令的结果 | 可以用中间结果做逻辑判断 |
| 适用场景 | 简单批量操作 | 需要条件判断的原子操作（如分布式锁释放） |

面试话术：事务只是把命令打包发送，不能在中间做判断，也不能回滚。Lua 脚本是真正的原子执行，可以写 if/else 逻辑，所以分布式锁的释放用 Lua 而不是事务。

## 主从复制流程

### 全量同步（首次连接）

```
从节点 → 发送 PSYNC ? -1
主节点 → 执行 bgsave 生成 RDB → 发送 RDB 给从节点
         同时缓冲期间的写命令到 repl_backlog
从节点 → 加载 RDB → 主节点发送缓冲区积压的写命令 → 从节点重放
```

### 增量同步（断线重连）

```
从节点 → 发送 PSYNC runid offset
主节点 → 检查 offset 是否在 repl_backlog 范围内
  → 在：只发送缺失部分的命令（增量）
  → 不在：退化为全量同步
```

**面试话术**：首次连接走全量同步，主节点 bgsave 生成 RDB 发给从节点，期间的写命令缓冲起来之后补发。断线重连走增量同步，根据 offset 只补发缺失的命令，如果 offset 已经不在 backlog 里了才退化为全量。

## 高频面试题

3. **缓存和数据库一致性怎么保证？**（Cache Aside + 延迟双删，强一致用 Canal 监听 binlog）
4. **Redis 事务和 Lua 脚本区别？**（事务不能回滚不能判断，Lua 原子执行可写逻辑）
5. **大 key 和热 key 怎么处理？**（大 key 拆分/压缩/UNLINK，热 key 本地缓存/分散多节点）
6. **主从复制流程？**（首次全量 RDB + 断线增量 repl_backlog）
