## 一、动态调度感知（当前 5 分 → 目标 7）

### 面试官问题

> 数据库里改了 cron，运行中的应用能感知到吗？

### 达标答法

> 我没有用 Spring 的 @Scheduled，而是自己接管了调度。启动时扫描器把任务注册完之后，会用 Spring 的 TaskScheduler 按数据库里的 cron 表达式注册调度，拿到的 ScheduledFuture 会缓存在内存 Map 里。
>
> 然后组件会开一个刷新线程，每分钟从数据库重新拉一次任务配置。如果发现某个任务的 cron 变了，就先 cancel 掉内存里旧的 ScheduledFuture，再按新 cron 重新注册一个。如果发现任务被禁用了，就直接 cancel，不再触发。
>
> 所以运维人员改数据库配置后，最多一分钟就能生效，不需要重启应用。

## 二、Feign 补偿幂等（当前 5 分 → 目标 7）

### 面试官问题

> 补偿重试的时候，怎么保证被调用方不会重复处理？

### 达标答法

> 每次 Feign 调用都会带一个业务唯一 ID，比如工单号、设备 ID 加时间戳的组合。被调方处理的时候，按这个业务 ID 做幂等校验——最直接的就是数据库唯一索引，第二次进来 insert 冲突，直接返回成功。
>
> 补偿任务重试时，用的是跟第一次完全一样的参数和业务 ID，被调方发现已经处理过就直接返回。
>
> 这也是为什么落库时要保留完整的原始参数，而不是重新生成——就是为了保证重试和首次调用的幂等基准一致。

## 三、混合检索 RRF（当前 6.5 分 → 目标 7）

### 面试官问题

> 为什么不直接用向量检索？两路结果怎么融合的？

### 达标答法

> 电网领域两种情况都有：
>
> **纯向量的问题**：SF6 泄漏和油气泄漏语义相近，但业务含义完全不同，向量检索会搞混。BM25 靠精确关键词匹配，能区分开。
>
> **纯 BM25 的问题**："超过额定值"和"大于标准值"关键词不同但含义一致，BM25 搜不到。向量检索靠语义相似性能找到。
>
> 所以两路互补。融合用的是 RRF 算法，核心思想是两路检索分别给每个文档一个排名，两路都排前面的文档总分最高。k 取 60，是排名平滑参数，在 20 到 100 之间调过，60 在我们 300 条规则语料上效果最好。

### 必记

- BM25 赢在精确匹配行业术语
- 向量赢在语义相似但用词不同
- RRF 的 k=60，**不是权重系数**

## 四、OOM 排查（当前 5 分 → 目标 7）

### 达标答法（讲电网项目数据同步 OOM）

> **现象**：有一个定时任务负责拉取设备在线监测数据并入库，某天任务跑到一半服务直接挂了，日志抛 OutOfMemoryError: Java heap space。
>
> **定位**：启动参数配了 -XX:+HeapDumpOnOutOfMemoryError，自动 dump 了堆快照。用 MAT 打开 Dominator Tree，发现一个 ArrayList 占了大量内存，顺着引用链往上查，定位到数据同步 Service 里——它一次性把几万台设备的监测数据全查出来放在内存里。
>
> **原因**：设备规模增长后，单次全量拉取的数据量远超预期。
>
> **解决**：改成设备 ID 分批拉取，每批 100 台，写库也分批，每批 1000 条。处理完一批就释放引用。
>
> **效果**：内存峰值降到正常水平，后来把分批策略推广到其他同步任务。

## 五、CPU 飙高排查

### 达标答法

> **排查步骤：**
>
> 第一步，`top` 看整机 CPU，确认是哪个 Java 进程。用 `jps -l` 确认具体服务。
>
> 第二步，`top -H -p <pid>` 找出 CPU 最高的线程。
>
> 第三步，线程 ID 转十六进制，`jstack` 导出线程栈，搜这个十六进制 ID。
>
> 第四步，看线程栈定位代码。我那次发现是数据清洗里大量字符串拼接和正则匹配。
>
> **解决**：字符串拼接改 StringBuilder，正则预编译复用，加分批处理。CPU 从 90% 降到 40%。

### 追问：不是代码热点还可能是什么？

> - **频繁 Full GC**：GC 线程一直跑，通过 GC 日志确认
> - **死循环**：线程栈里同一方法反复出现
> - **锁竞争严重**：大量线程 BLOCKED 状态

### 口诀

> top → jps → top -H → 十六进制 → jstack

## 六、高吞吐数据同步链路

### 面试官问题

> 简历上写了高吞吐数据同步，具体怎么做的？

### 达标答法

> 我们有一批定时任务负责从外部系统拉取设备遥测数据，单次任务要处理几千台设备、几十万条记录。
>
> **三个核心设计：**
>
> **第一，时间窗口滚动拉取。** 按最近 30 分钟的窗口增量拉取，减少重复数据和接口压力。
>
> **第二，并发控制。** 设备 ID 分批后线程池并发执行，子任务用 Future 收集结果，IO 密集型参数。
>
> **第三，可中断。** 子任务检查中断状态，停止指令能及时中断。Future.get 抛中断异常时恢复中断标志。
>
> 分批细节（每批 100 台设备、1000 条入库）和 OOM 排查里讲的是同一个场景，不重复说了。

### 追问：数据重复怎么处理？

> 设备 ID + 时间戳联合唯一索引，批量插入用 `INSERT IGNORE`，数据库兜底。

## 七、慢查询排查

### 达标答法（STAR 结构）

> **背景**：设备台账查询接口，联表查设备信息、监测数据和试验记录，页面加载 3-4 秒。
>
> **定位**：EXPLAIN 分析发现 type 是 ALL 全表扫描，Extra 有 Using filesort。查表结构发现联合索引顺序不对。
>
> **解决**：
> 1. 调整联合索引顺序，区分度高的字段放前面
> 2. `SELECT *` 改成只查需要的字段，走覆盖索引
> 3. 大偏移分页改成 `WHERE id > last_id LIMIT N`
>
> 优化后从 3-4 秒降到 200ms 以内。

### 追问：EXPLAIN 重点看什么？

> - **type**：ALL 必须优化，至少到 ref 或 range
> - **key**：NULL 就是没走索引
> - **Extra**：Using filesort（额外排序）和 Using temporary（临时表）要优化，Using index 是好的

## 八、数据库死锁排查

### 面试官常见问法

> - "遇到过数据库死锁吗？怎么排查的？"
> - "死锁怎么避免？"

### 达标答法

> **排查流程：**
>
> 第一步，看日志。InnoDB 检测到死锁会自动回滚一个事务，错误日志有 `Deadlock found` 记录。
>
> 第二步，`SHOW ENGINE INNODB STATUS\G` 看最近一次死锁详情——两个事务分别持有什么锁、在等什么锁、执行的 SQL。
>
> 第三步，分析加锁顺序。根本原因几乎都是两个事务以相反顺序加锁。
>
> **预防四招：**
> 1. **统一加锁顺序**——都先锁 A 表再锁 B 表
> 2. **缩短事务**——持锁时间越短概率越低，不要在事务里做远程调用
> 3. **合理加索引**——没索引的 UPDATE 会锁全表，大幅增加死锁概率
> 4. **InnoDB 自动检测**——`innodb_deadlock_detect = ON`（默认开启），回滚代价小的事务

### 追问：你项目里遇到过吗？

> 遇到过。工单流转场景，告警触发创建工单和人工处理工单回写告警，两个操作访问告警表和工单表的顺序不一致，偶发死锁。统一成先操作告警表再操作工单表后解决。

## 九、Kafka 核心机制

### 达标答法

> **Kafka 为什么吞吐量高？四个设计：**
>
> 1. **顺序写磁盘。** 追加写入，不做随机写
> 2. **Page Cache。** 利用操作系统页缓存
> 3. **零拷贝。** sendfile 系统调用，数据不经过用户态
> 4. **批量和压缩。** 攒 batch 再发，snappy/lz4 压缩
>
> **不丢消息三环节：**
> - Producer：`acks=all` + retries
> - Broker：`replication.factor ≥ 3`，`min.insync.replicas = 2`
> - Consumer：关闭自动提交，处理成功后手动提交 offset
>
> **ISR 机制：** 和 Leader 保持同步的副本集合。落后 30 秒踢出，追上重新加入。Leader 挂了从 ISR 选新 Leader。

### 追问：Kafka 和 RocketMQ 怎么选？

> 大数据用 Kafka（百万级吞吐，Flink/Spark 生态），业务消息用 RocketMQ（延迟消息、事务消息、消息查询）。我们电网项目选 RocketMQ 因为需要事务消息和消息查询。

## 十、分布式事务

### 达标答法

> **四种方案按常用程度排：**
>
> **第一，本地消息表。** 业务操作和写消息表放同一个本地事务，定时任务扫描发送。我的 Feign 失败补偿就是这个思路。
>
> **第二，RocketMQ 事务消息。** 先发半消息，执行本地事务，成功提交失败回滚，超时 Broker 回查。
>
> **第三，TCC。** Try 预留、Confirm 确认、Cancel 释放。一致性强但每个参与方要写三个接口。
>
> **第四，Seata AT 模式。** 自动生成反向 SQL 回滚，侵入小但需要额外部署。
>
> **我们项目不用强一致**：Feign 补偿重试（类似本地消息表）+ MQ 两层幂等，最终一致够了。

### 追问：Feign 补偿和本地消息表的区别？

> 本质思路一样。区别是本地消息表需要业务主动写消息记录，我的 Feign 补偿是 AOP 自动拦截失败调用，自动持久化调用上下文，对业务无侵入。

## 十一、Spring Bean 生命周期 + 循环依赖

### 达标答法

> **四个阶段：**
> 1. **实例化**：反射创建空对象
> 2. **属性注入**：@Autowired、@Value
> 3. **初始化**：@PostConstruct → afterPropertiesSet → init-method，有 AOP 则生成代理
> 4. **销毁**：@PreDestroy → destroy → destroy-method
>
> **循环依赖用三级缓存解决：**
> - 一级缓存：完整初始化好的 Bean
> - 二级缓存：已实例化但未初始化的早期 Bean
> - 三级缓存：Bean 的 ObjectFactory 工厂
>
> 创建 A 依赖 B → A 的工厂放三级缓存 → 创建 B 依赖 A → 从三级缓存拿 A 的工厂创建早期引用放二级 → B 完成初始化 → A 完成初始化。
>
> **构造器注入的循环依赖解决不了**，因为对象都没创建出来没法放缓存。

### 追问：为什么要三级，两级不行吗？

> 两级能解决简单循环依赖，但有 AOP 代理就不行了。三级缓存的 ObjectFactory 可以在需要时生成代理对象，保证 B 拿到的是 A 的代理而不是原始对象。

## 十二、回归决策（当前 6 分 → 目标 7）

### 面试官问题

> 改了 prompt，68 条里 3 条从对变错了，5 条从错变对了，要不要上？

### 达标答法

> 不能只看总数。分三步：
>
> 第一步，看变错的 3 条是什么。核心高频场景变错了就不能上。
>
> 第二步，分析变错原因。是 prompt 影响了意图识别？还是检索召回变了？
>
> 第三步，如果变错的都是边缘场景，变对的覆盖了更重要的业务场景，可以上。同时把 3 条加入回归基线，确保下次不退化。
>
> 准确率是门槛，但最终决策要看变错的 case 影响面多大。
