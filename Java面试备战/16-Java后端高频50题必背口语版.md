# Java 后端高频 50 题必背口语版

> 目标不是背标准答案，而是给你一版能直接开口说的面试话术。
> 每题尽量控制在 30 到 60 秒，先说结论，再说原理，最后结合项目。

## 一、Java 基础与集合

### 1. `==` 和 `equals` 有什么区别？

`==` 比较的是地址，基本类型比较的是值；`equals` 默认也是比较地址，但很多类比如 `String` 重写后比较的是内容。面试里我会补一句，包装类型还要注意自动装箱缓存，比如 `Integer` 在 `-128~127` 有缓存。

### 2. `String`、`StringBuilder`、`StringBuffer` 的区别？

`String` 不可变，适合读多写少；`StringBuilder` 可变但线程不安全，单线程拼接优先用它；`StringBuffer` 线程安全但有锁，性能低一些，业务里现在用得少。

### 3. `HashMap` 底层原理是什么？

JDK 8 里是数组加链表加红黑树。key 先算 hash 定位桶，桶里冲突少时走链表，链表过长且容量达到阈值会树化成红黑树，减少极端情况下的查找时间。

### 4. `HashMap` 为什么线程不安全？

因为它没有并发控制，多线程同时 put 可能导致数据覆盖、链表结构异常、size 不准。JDK 7 扩容时还可能形成环。并发场景我会用 `ConcurrentHashMap`。

### 5. `ConcurrentHashMap` 为什么比 `Hashtable` 好？

`Hashtable` 基本是整表大锁，并发性能差；`ConcurrentHashMap` 在 JDK 8 里用的是 CAS 加 `synchronized` 的桶级控制，锁粒度更细，读性能也更好。

### 6. `ArrayList` 和 `LinkedList` 怎么选？

大多数业务场景我默认选 `ArrayList`。因为它连续内存，随机访问快，遍历也友好。`LinkedList` 只有在频繁头尾插入删除时才可能占优，但真实业务里并不常见。

### 7. 为什么重写 `equals` 一般也要重写 `hashCode`？

因为哈希容器先用 `hashCode` 定位桶，再用 `equals` 比较内容。如果两个对象逻辑上相等，但 hashCode 不一样，就会出现在 `HashMap`、`HashSet` 里表现异常。

### 8. 异常分哪几类？

主要分 `Error` 和 `Exception`。`Exception` 里又分受检异常和运行时异常。业务开发里最常见的是运行时异常，Spring 事务默认也是对 `RuntimeException` 回滚。

## 二、JVM 与并发

### 9. JVM 运行时内存区域有哪些？

我一般会答五块：程序计数器、虚拟机栈、本地方法栈、堆、方法区。业务里最常打交道的是堆、栈和元空间，因为 OOM、Full GC、类加载问题基本都围绕这几块。

### 10. 怎么判断对象是否存活？

主流不是引用计数，而是可达性分析。就是从 GC Roots 出发往下找，能连上的对象算存活，连不上的才可能被回收。这样能解决循环引用问题。

### 11. G1 的特点是什么？

G1 把堆分成很多 Region，不再要求物理上固定新生代和老年代。它会优先回收收益高的 Region，可以设停顿目标，比较适合大堆和对停顿有要求的场景。

### 12. 你怎么排查 OOM？

先看是哪种 OOM，再拿 dump。堆溢出我会开 `HeapDumpOnOutOfMemoryError`，用 MAT 看 Dominator Tree 和 GC Roots。你自己的项目案例就可以讲“全量跑批改成分批处理，内存峰值明显下降”。

### 13. CPU 飙升怎么排查？

我一般按五步排查。先用 `top` 看整体 CPU，定位占用高的进程；如果机器上有多个 Java 进程，再用 `jps -l` 对一下具体是哪个应用。然后用 `top -H -p <pid>` 找出占用 CPU 最高的线程，把线程 ID 转成十六进制，再用 `jstack <pid>` 去查对应线程栈。最后结合线程栈判断是死循环、频繁 GC、锁竞争，还是某段业务代码热点。  
如果看到是 GC 线程打满，我会继续看 GC 日志和堆使用；如果是业务线程，就沿着栈信息定位到具体方法和代码行；如果是锁相关问题，就再结合线程状态看有没有大量 `BLOCKED` 或反复竞争。

### 14. `synchronized` 和 `ReentrantLock` 区别是什么？

`synchronized` 是 JVM 内置锁，语法简单，异常还能自动释放；`ReentrantLock` 基于 AQS，支持可中断、超时获取、公平锁和多个 Condition，能力更强。简单互斥我优先 `synchronized`，复杂控制再上 `ReentrantLock`。

### 15. `volatile` 能保证什么？不能保证什么？

它能保证可见性和一定的有序性，但不能保证复合操作的原子性，比如 `count++` 还是不安全。要做计数我会用 `AtomicInteger` 或加锁。

### 16. 线程池核心参数有哪些？

核心线程数、最大线程数、空闲存活时间、阻塞队列、线程工厂、拒绝策略。面试里我会强调，线程池设计本质上是在平衡吞吐、延迟和资源上限。

### 17. 为什么不建议直接用 `Executors`？

因为它很多工厂方法默认参数太激进。比如 `newFixedThreadPool` 的队列基本无界，可能堆积任务导致 OOM；`newCachedThreadPool` 最大线程数很大，可能把机器打满。更稳妥的是手动 new `ThreadPoolExecutor`。

### 18. 常见拒绝策略怎么选？

默认是 `AbortPolicy` 直接抛异常；如果想让上游感知背压，我更常用 `CallerRunsPolicy`，让调用方自己执行，系统会自然慢下来；静默丢弃一般要非常谨慎。

### 19. `ThreadLocal` 有什么用？有什么坑？

它适合存线程上下文，比如 traceId、用户上下文、任务上下文。最大的坑是线程池复用导致脏数据，所以用完一定要 `remove()`，不然容易内存泄漏和串数据。

### 20. 什么是 CAS？有什么问题？

CAS 是比较并交换，乐观锁思想，很多原子类都基于它。优点是无锁高性能，问题是会有 ABA、自旋开销大、只能保证单变量原子性。

## 三、Spring / Spring Boot / MyBatis

### 21. 什么是 IoC 和 DI？

IoC 就是对象控制权交给 Spring 容器，DI 是容器把依赖注入进来。对业务开发来说，本质意义是解耦，让对象创建、装配、生命周期都统一管理。

### 22. Bean 生命周期大概是怎样的？

实例化、属性注入、Aware 回调、前置处理、初始化、后置处理、放入容器、容器关闭时销毁。面试官如果追问，我会补一句，AOP 代理通常是在初始化后阶段织进去的。

### 23. Spring 怎么解决循环依赖？

针对单例 Bean 的 setter 或字段注入，Spring 用三级缓存解决。核心思想是先暴露一个早期引用，让依赖方先拿到，再补全初始化。构造器循环依赖就没法这么解。

### 24. AOP 是什么？项目里怎么用？

AOP 就是把日志、事务、监控这种横切逻辑从业务代码里抽出来统一处理。你项目里就很适合讲定时任务监控切面和 Feign 失败补偿切面。

### 25. Spring AOP 用的是动态代理还是 CGLIB？

都有。目标类实现了接口时可以走 JDK 动态代理；没有接口时一般走 CGLIB。Spring Boot 2.x 默认更偏向类代理。

### 26. Spring Boot 自动装配原理是什么？

核心就是启动时加载自动配置类，再通过一堆 `@ConditionalOnXxx` 按条件生效。面试里如果结合你项目，就可以直接讲你们自研 Starter 是怎么通过自动装配接入业务服务的。

### 27. 什么时候会自己写 Starter？

当某个能力需要在多个服务里复用，而且希望低侵入接入时很适合写 Starter。你的定时任务治理平台就是典型例子，它不是一个类库，而是一套统一治理能力。

### 28. `# {}` 和 `${}` 的区别？

`#{}` 走预编译参数，占位安全，能防 SQL 注入；`${}` 是字符串拼接，只适合动态表名、动态列名这类特殊场景，普通查询尽量别用。

### 29. `@Transactional` 常见失效场景有哪些？

最常见三个：同类内部调用绕过代理、异常被你自己 catch 住了、抛的是受检异常但没配 `rollbackFor`。另外也尽量把事务放在 `public` 方法上讲，最稳。

### 30. 事务传播行为里最常用的是哪几个？

`REQUIRED` 最常用，外面有事务就加入，没有就新开；`REQUIRES_NEW` 适合我要把某段逻辑独立提交，比如操作日志、补偿记录；`NESTED` 更依赖底层数据库保存点能力。

## 四、MySQL 与 SQL 优化

### 31. 为什么 MySQL 索引用 B+ 树？

因为磁盘场景最关心 IO 次数。B+ 树非叶子只存索引，单节点能放更多 key，树更矮；叶子节点天然有序，范围查询和排序都友好，所以很适合数据库索引。

### 32. 聚簇索引和非聚簇索引有什么区别？

InnoDB 主键索引是聚簇索引，叶子节点直接存整行；二级索引叶子节点存的是主键值，所以很多查询还要回表。能做覆盖索引就尽量做，能少一次回表。

### 33. 什么是最左前缀原则？

复合索引从左往右匹配，比如 `(a,b,c)`，你查 `a` 或 `a,b` 都能很好利用索引；如果直接跳过 `a` 查 `b`，基本就用不上。面试里我会强调一句，范围条件后面的列利用率也会变差。

### 34. 索引为什么会失效？

常见原因有对索引列做函数、隐式类型转换、左模糊、`or` 一侧没索引、优化器判断全表扫描更便宜。真正优化 SQL 时，不能只看“建没建索引”，要看执行计划。

### 35. `EXPLAIN` 重点看什么？

我重点看 `type`、`key`、`rows`、`Extra`。尤其是 `type` 有没有掉到 `ALL`，`key` 有没有命中预期索引，`Extra` 里有没有 `Using filesort`、`Using temporary`。

### 36. 什么是覆盖索引？

查询需要的字段都在索引里，MySQL 只扫索引树，不用回表。这个优化在高频读场景里非常实用，往往比单纯“多建几个索引”更有效。

### 37. MySQL 事务四大特性是什么？

原子性、一致性、隔离性、持久性。面试我一般不会只背定义，还会补一句：undo log 主要支撑回滚和 MVCC，redo log 主要支撑持久化和崩溃恢复。

### 38. MySQL 隔离级别有哪些？默认是什么？

四种：读未提交、读已提交、可重复读、串行化。MySQL InnoDB 默认是可重复读。你要记住的不只是名字，还要知道它分别解决了什么问题。

### 39. 什么是 MVCC？

MVCC 就是多版本并发控制，核心是通过 undo log 和 Read View 让读写不互相强阻塞。RC 每次读都会生成新视图，RR 一般复用第一次读的视图。

### 40. MySQL 怎么解决幻读？

这个问题要分开答。快照读主要靠 MVCC，在 RR 下通常不会看到幻读；当前读比如 `for update`，主要靠 Next-Key Lock，也就是记录锁加间隙锁，防止别人往这个范围插入新行。

### 41. 慢 SQL 你怎么排查和优化？

先开慢查询日志定位 SQL，再用 `EXPLAIN` 看执行计划，然后针对性优化：改索引、改 SQL 写法、减少回表、避免 `select *`、把大查询拆批。你银行项目的跑批优化就很适合拿来讲。

## 五、Redis / MQ / 微服务

### 42. Redis 为什么快？

首先是内存访问快，其次是单线程命令执行避免了锁竞争，再加上 IO 多路复用把网络层效率做上去。面试里我会补一句，Redis 6 以后是网络 IO 多线程，不是命令执行多线程。

### 43. 缓存穿透、击穿、雪崩分别是什么？怎么解决？

穿透是查根本不存在的数据，解决靠布隆过滤器或空值缓存；击穿是热点 key 失效瞬间把数据库打爆，解决靠互斥锁或逻辑过期；雪崩是大量 key 同时失效，解决靠过期时间打散、多级缓存、限流降级。

### 44. Redis 分布式锁怎么实现？

最基础是 `SET key value NX PX expire`，value 要是唯一 token，释放时要校验 token 再删，最好用 Lua 脚本原子释放。面试里我还会主动说，锁不是万能的，最终还要靠业务幂等兜底。

### 45. Redis 里为什么很多原子操作喜欢用 Lua？

因为 Lua 脚本在 Redis 服务端一次执行完，没有多次网络往返，也不会出现“先查再改”被别人插队的问题。像分布式锁释放、令牌桶限流都很适合用 Lua。

### 46. Redis 主从复制流程是怎样的？

首次一般走全量同步，主节点生成 RDB 发给从节点，再把期间写命令补过去；后续断线重连优先走增量同步，根据 offset 补缺失数据。如果 backlog 里找不到了，才退回全量。

### 47. RocketMQ 和 Kafka 怎么选？

业务系统我更偏向 RocketMQ，因为延迟消息、事务消息、消息查询这些能力更完整，可靠性也更强；大数据日志流我更偏 Kafka，因为吞吐量高，生态跟 Flink、Spark 更贴。

### 48. 消息为什么会重复消费？怎么解决？

因为 MQ 至少一次投递语义下，只要消费确认丢了、消费者重启了、网络抖动了，都可能重投。真正解法不是追求“绝不重复”，而是消费端做幂等，比如业务唯一键、去重表、Redis 短时去重。

### 49. 消息丢失怎么分析？

我会按三段说：Producer 到 Broker、Broker 存储、Broker 到 Consumer。分别对应发送重试、刷盘和副本、消费确认和重试机制。这样答会比只说一个环节更完整。

### 50. 消息积压怎么处理？

先判断是消费者挂了还是消费太慢。应急手段是恢复消费者、扩容实例、必要时临时跳过非核心逻辑；长期优化要看消费端是不是单条写库、远程调用太慢、分区数不够。
