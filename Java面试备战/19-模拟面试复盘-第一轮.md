## 二、任务扫描注册流程（得分：4/10 → 目标：7）

### 面试官问题

> 你的 Starter 启动的时候，任务是怎么被发现和注册的？具体流程是什么？

### 我的回答

> 我写了注解，只要定时任务上加上这个注解，项目启动的时候就会发现和注册。

### 问题

- 等于什么都没说
- 关键环节全丢：谁去扫描、扫描完做什么、元数据存哪
- 会让面试官怀疑你没写过这个东西

### 达标答法

> 我定义了一个自定义注解，上面会标任务分组、cron 表达式、负责人、最大调用次数这些治理信息。
>
> 启动时有一个扫描器，实现了 Spring 的 CommandLineRunner，容器初始化完成后自动触发。它会扫描所有 Bean 上标了这个注解的方法，把注解里的元数据提取出来。
>
> 提取完之后，跟数据库里已有的任务配置做比对——新任务就插入，已有的就更新 cron 和分组。这样每次启动，数据库里的任务清单就和代码里的注解保持同步了。后续动态启停、执行监控都是基于这张配置表来驱动的。

## 三、动态调度感知（得分：5/10 → 目标：7）

### 面试官问题

> 任务实际调度是怎么触发的？是 Spring 的 @Scheduled 在跑，还是你自己接管了调度？数据库里改了 cron，运行中的应用能感知到吗？

### 我的回答

> 我直接接管了调度，可以感知到，组件会开一个线程每分钟去查一次。

### 问题

- "接管了调度"怎么接管的没说
- 查完之后发现变化了怎么处理没说
- 动态启停怎么做没说

### 达标答法

> 我没有用 Spring 的 @Scheduled，而是自己接管了调度。启动时扫描器把任务注册完之后，会用 Spring 的 TaskScheduler 按数据库里的 cron 表达式注册调度，拿到的 ScheduledFuture 会缓存在内存 Map 里。
>
> 然后组件会开一个刷新线程，每分钟从数据库重新拉一次任务配置。如果发现某个任务的 cron 变了，就先 cancel 掉内存里旧的 ScheduledFuture，再按新 cron 重新注册一个。如果发现任务被禁用了，就直接 cancel，不再触发。
>
> 所以运维人员改数据库配置后，最多一分钟就能生效，不需要重启应用。

## 四、分布式锁实现（得分：8/10 ✓ 已达标）

### 面试官问题

> 两个实例同时触发同一个任务，怎么保证只有一个在跑？分布式锁具体怎么实现的？

### 我的回答

> 定时任务启动的时候，会基于自身生成一个 token，然后用 setIfAbsent 方法获取锁并设置 TTL 防止任务失败锁一直没有释放。获取到锁之后就会启动自动续时任务，主要任务还在执行就一直续期，防止长任务的时候锁被释放。任务执行完或者异常则基于 Redis 的 Lua 结合 token 释放锁，保证锁不会被别的线程释放。

### 可以更好

- 补上数字：TTL 120 秒，每 30 秒续租一次
- 主动提对比：业务仓库里的基础分布式锁解锁是先读后删不是原子的，但调度平台里做了 Lua 原子释放

### 加分补充

> TTL 默认 120 秒，续租线程每 30 秒跑一次，用 Lua 脚本先比对 token 再 expire，保证续租也是原子的。
>
> 另外说一句，业务仓库里还有一个基础的分布式锁工具类，那个解锁是先读再删，不是原子的。我知道这个差距，调度平台里的锁要求更严谨所以做了升级。

## 五、MQ 消费端怎么保证不重复消费？消费失败怎么处理？（得分：5/10 → 目标：7）

### 面试官常见问法

> - "你们消费端幂等怎么做的？"
> - "消息重复消费怎么防？"
> - "消费失败了怎么办？"

### 我的回答

> 收到消息后先用 Redis 短时内去重，MySQL 唯一索引兜底，数据进入数据库之后再提交 offset。

### 问题

- Redis 去重的 key 是什么、窗口多长没说
- 处理失败的链路完全没讲
- "提交 offset"说法不准确，RocketMQ 是返回 CONSUME_SUCCESS，不是手动提交 offset
- 副作用控制（通知不重复发）没提

### 达标答法

> 我做了两层幂等，再加失败兜底：
>
> **第一层，Redis 快速去重。** 消息进来后，拿消息里的唯一 ID 往 Redis 写一个处理中的标记，设 3 分钟过期。如果标记已存在，说明正在处理或刚处理完，直接跳过。处理成功后把过期时间延长到 10 分钟，防止刚处理完标记就过期又被当成新消息。
>
> **第二层，数据库业务幂等。** Redis 去重只是快速拦截，真正的业务幂等在 service 层按业务唯一键做，比如设备 ID + 事件时间建唯一索引。insert 冲突就说明已处理过。
>
> **副作用隔离。** 落库是幂等的，但发通知不是。所以只有 insert 返回影响行数为 1（真正新插入）才触发异步通知，重试进来发现数据已存在就不会重复发。
>
> **消费失败处理：** 清掉 Redis 的处理中标记，抛异常让 MQ 重新投递。最多重试 3 次加上初始那次总共 4 次机会，超了就不再重试，避免一条坏消息把消费者卡死。
>
> 举个具体场景，比如故障跳闸消息进来，要落主记录、子事件和后果记录，还要发短信通知。落库这一步靠唯一索引保证幂等，短信只对新插入的数据发，重试不会重复通知。

### 追问：你怎么判断数据是新插入的还是已存在的？

> insert 返回影响行数。返回 1 是新插入，返回 0 或唯一索引冲突是已存在。只对返回 1 的触发异步通知。本质就是把落库当判断依据，通知当后置动作。
>
> **设计原则：先幂等写入，再根据写入结果决定是否触发副作用。**

### 表达原则

- 讲机制，不念方法名。比如不要说"调 tryAcquireByMessage"，要说"往 Redis 写一个处理中标记"
- 面试官不知道你代码里的方法叫什么，他只关心你做了什么、为什么这么做
- Spring / Redis / MQ 的标准术语可以说，你自己项目里的类名和方法名不要说

## 六、Feign 补偿幂等（得分：5/10 → 目标：7）

### 面试官问题

> 补偿任务重试的时候，怎么保证被调用方不会重复处理？第一次调成功了但网络超时没收到响应，你以为失败了落库准备补偿，被调方岂不是处理两次？

### 我的回答

> 被调用方是单线程执行的，并且通过业务键进行去重。

### 问题

- "单线程执行"说法不准确，Spring Boot 默认多线程处理请求
- "业务键去重"太笼统，怎么去重没说

### 达标答法

> 每次 Feign 调用都会带一个业务唯一 ID，比如工单号、设备 ID 加时间戳的组合。被调方处理的时候，按这个业务 ID 做幂等校验——最直接的就是数据库唯一索引，第二次进来 insert 冲突，直接返回成功。
>
> 补偿任务重试时，用的是跟第一次完全一样的参数和业务 ID，被调方发现已经处理过就直接返回。
>
> 这也是为什么落库时要保留完整的原始参数，而不是重新生成——就是为了保证重试和首次调用的幂等基准一致。

## 七、补偿重试策略（得分：7/10 ✓ 基本达标）

### 面试官问题

> 补偿任务是无脑扫表重试，还是有什么策略？目标服务挂了你还一直重试？

### 我的回答

> 定时器每 5 分钟发起一次重试，重试之前先检查服务的心跳，判断服务是否正常，正常才发起重试。

### 可以更好

- 有没有按服务分组？A 挂了 B 正常，是不是只跳过 A？
- 重试次数有上限吗？
- 面试时不要重复同一句话

### 加分版

> 补偿任务每 5 分钟跑一次，但不是无脑扫全表重试。
>
> 第一步，按目标服务分组，把待重试的记录按服务名归类。
>
> 第二步，逐个服务做健康检查，查目标服务有没有可用实例。整个服务不可用的话，这一批直接跳过。
>
> 第三步，才发起真正的重试，用落库时保留的原始参数重放。
>
> 第四步，有上限兜底。每条记录有最大重试次数，超过标记为最终失败。
>
> 这样补偿机制不会在服务故障时变成压力放大器。

## 九、Slot 识别链路（得分：8/10 ✓ 已达标）

### 面试官问题

> "110kV 及以上断路器 SF6 气体泄漏年超过 50mL 且运行年限超过 15 年的设备生成预警"，这句话你的系统怎么处理的？

### 我的回答

> 先用正则表达式抽取结构化信息，分词器进行分词，通过 AC 自动机提取设备、场景、指标。如果缺少 slot 先尝试根据已有信息反推，比如根据设备和指标可以推出场景。如果存在多候选对象则根据语义距离、共现模式、实体特异性筛选最合适候选人。无法推断则触发澄清。把得到的结果打包发给 LLM 获取结果。

### 可以更好

- 主动指出"运行年限超过 15 年"容易被误识别成时间窗口的 bad case
- 后半段打分融合和前面链路之间的过渡要说清楚
- 补上 DSL 校验环节

### 要主动补的 bad case

> 这句话有个典型难点——"运行年限超过 15 年"，大模型很容易把它当成时间窗口来处理，但它其实是设备属性过滤条件，应该放在设备筛选条件里而不是指标条件里。我们在 DSL 校验阶段会检查这种归类错误，发现位置不对会自动纠正。

## 十、混合检索（得分：6.5/10 → 目标：7）

### 面试官问题

> 为什么不直接用向量检索？加 BM25 解决了什么问题？两路结果怎么融合的？

### 我的回答中有两个硬伤

| 我说的 | 正确的 |
|---|---|
| 系数 0.6 | k 参数 60，不是权重，是排名平滑因子 |
| 相同含义语义距离远需要关键词检索 | 说反了，BM25 优势是精确关键词匹配 |

### 达标答法

> 电网领域两种情况都有：
>
> **纯向量的问题**：SF6 泄漏和油气泄漏语义相近，但业务含义完全不同，向量检索会搞混。BM25 靠精确关键词匹配，能区分开。
>
> **纯 BM25 的问题**："超过额定值"和"大于标准值"关键词不同但含义一致，BM25 搜不到。向量检索靠语义相似性能找到。
>
> 所以两路互补。融合用的是 RRF 算法，核心思想是两路检索分别给每个文档一个排名，两路都排前面的文档总分最高。k 取 60，是排名平滑参数，在 20 到 100 之间调过，60 在我们 300 条规则语料上效果最好。

### 必须记牢

- BM25 赢在精确匹配行业术语
- 向量赢在语义相似但用词不同
- RRF 的 k=60，不是权重系数

## 十一、回归决策（得分：6/10 → 目标：7）

### 面试官问题

> 改了一版 prompt，68 条里 3 条从对变错了，5 条从错变对了，你怎么决定要不要上？

### 我的回答

> 要上，我们跑回归集看的是准确率，低于就重新调，高于就可以发版。

### 问题

- 只看总准确率太粗
- 变错的 3 条可能比变对的 5 条更重要

### 达标答法

> 不能只看总数。分三步判断：
>
> 第一步，先看变错的那 3 条是什么。如果是核心高频场景，比如断路器、主变这类主力设备，哪怕整体准确率提升了也不能直接上，得先修。
>
> 第二步，分析变错的原因。是 prompt 改动影响了意图识别？还是检索召回的范文变了？搞清原因才知道能不能定向修复。
>
> 第三步，如果变错的都是边缘场景，变对的覆盖了更重要的业务场景，那可以上。同时把那 3 条加入回归基线，确保下次不再退化。
>
> 准确率是门槛，但最终决策要看变错的 case 影响面有多大。

## 十二、HashMap put 流程（得分：4/10 → 目标：7）

### 面试官问题

> HashMap 在 JDK 1.8 里，put 一个元素的完整流程讲一下。

### 我的回答中的硬伤

| 我说的 | 正确的 |
|---|---|
| 桶中节点大于 12 个或总节点大于 64 触发扩容 | 元素总数 > 容量 × 0.75 时扩容；链表 ≥ 8 且容量 ≥ 64 时树化 |
| 没提 hash 扰动 | 高 16 位和低 16 位异或 |
| 没提定位方式 | (n-1) & hash 位运算 |
| 没提 key 相同覆盖 | 相同 key 覆盖 value |

### 达标答法

> **第一步，算 hash。** 拿 key 的 hashCode，高 16 位和低 16 位异或，让高位参与运算减少冲突。
>
> **第二步，定位桶。** 用 (n-1) & hash 算桶下标，位运算代替取模，所以容量必须是 2 的幂。
>
> **第三步，放元素：**
> - 桶是空的，直接放
> - 桶里第一个节点 key 相同，覆盖 value
> - 桶里是链表，遍历到尾部追加，中间发现相同 key 就覆盖
> - 桶里是红黑树，走树的插入逻辑
>
> **第四步，判断树化。** 链表追加后长度 ≥ 8，再看数组容量是否 ≥ 64。都满足才转红黑树，容量不够就先扩容。
>
> **第五步，判断扩容。** 元素总数超过容量 × 0.75，触发扩容，数组翻倍，所有元素重新定位。

### 关键数字

| 参数 | 值 |
|---|---|
| 默认容量 | 16 |
| 负载因子 | 0.75 |
| 树化阈值 | 链表 ≥ 8 且容量 ≥ 64 |
| 退化阈值 | 红黑树节点 ≤ 6 |

## 十三、OOM 排查（得分：5/10 → 目标：7）

### 面试官问题

> 讲一个你实际遇到的线上 OOM。

### 问题

- 讲的是教科书流程，不是具体故事
- "Full GC 后下降就扩充老年代"判断太草率
- 原来准备的是银行项目的故事，太久远了，换成电网项目更好

### 达标答法（讲电网项目数据同步 OOM）

> 在变电 3.0 项目里遇到过一次数据同步任务 OOM。
>
> **现象**：有一个定时任务负责拉取设备在线监测数据并入库，某天任务跑到一半服务直接挂了，日志抛 OutOfMemoryError: Java heap space。
>
> **定位**：启动参数配了 -XX:+HeapDumpOnOutOfMemoryError，自动 dump 了堆快照。用 MAT 打开 Dominator Tree，发现一个 ArrayList 占了大量内存，顺着引用链往上查，定位到数据同步 Service 里——它一次性把几万台设备的监测数据全查出来放在内存里做处理。
>
> **原因**：设备规模增长后，单次全量拉取的数据量远超预期，内存直接撑爆。
>
> **解决**：改成设备 ID 分批拉取，每批 100 台设备，拉完处理完再拉下一批。写库也分批，每批 1000 条 insert。处理完一批就释放引用，不会一直持有全量数据。
>
> **效果**：内存峰值从接近堆上限降到正常水平，任务稳定运行。后来我还把这个分批策略作为任务开发规范推广到了其他同步任务里。

### 备选故事（银行项目，如果面试官追问早期经验）

> 银行项目里也遇到过类似问题——财务报表统计一次性加载全量交易流水到内存做汇总，改成分页查询 + 分批处理解决的。

### 要点

- 讲排查问题用 STAR 结构：现象 → 定位 → 原因 → 解决 → 效果
- 优先讲最近项目的故事，面试官更感兴趣
- 不要讲通用流程图，要讲"我遇到了什么、我怎么查的、最后怎么改的"

## 十三补充、CPU 飙高排查

### 面试官问题

> 你遇到过 CPU 飙高的问题吗？怎么排查的？

### 达标答法

> 遇到过。在电网项目里有一次定时任务高峰期，监控告警 CPU 使用率持续 90% 以上。
>
> **排查过程：**
>
> 第一步，先用 `top` 看整机 CPU，确认是哪个 Java 进程占用高。机器上跑了多个服务，用 `jps -l` 确认是数据处理服务。
>
> 第二步，用 `top -H -p <pid>` 找出这个进程里 CPU 最高的线程，记下线程 ID。
>
> 第三步，把线程 ID 转成十六进制，再用 `jstack` 导出线程栈，搜这个十六进制 ID 对应的线程。
>
> 第四步，看线程栈发现是某个数据清洗逻辑里有一段循环在做大量字符串拼接和正则匹配，数据量一大就变成了 CPU 热点。
>
> **解决**：把字符串拼接改成 StringBuilder，正则表达式预编译复用，同时对这段逻辑做了分批处理，避免单次处理数据量过大。
>
> **效果**：CPU 使用率从 90% 降到 40% 左右。

### 追问：如果不是代码热点，还可能是什么原因？

> 还有几种常见情况：
>
> - **频繁 Full GC**：GC 线程一直在跑，可以通过 GC 日志确认。解决思路是查内存泄漏或者调大堆。
> - **死循环**：线程栈里会看到同一个方法反复出现。
> - **锁竞争严重**：大量线程卡在 BLOCKED 状态等锁，实际上 CPU 花在线程调度上了。

### 要点

- 排查步骤要记牢：top → jps → top -H → 线程 ID 转十六进制 → jstack
- 最后一定要落到"具体是什么代码、怎么改的"，不能只停在"找到线程"

## 十四、Feign 补偿 vs MQ 重试为什么不统一（得分：6/10 → 目标：7）

### 面试官问题

> 一个用落库补偿，一个用 MQ 自带重试，为什么不统一成一种方案？

### 我的回答

> Feign 调用是系统微服务之间的调用，MQ 是外部系统和我们系统之间的调用，并不能统一。

### 达标答法

> 这两种重试解决的是不同层面的问题。
>
> Feign 补偿解决的是同步调用失败后的恢复。我需要保留完整的调用上下文——哪个服务、哪个方法、什么参数——后面才能按原样重放。如果硬把每次 Feign 失败都发消息到 MQ 再消费重试，多引入一层中间件依赖，链路更复杂更难排查。
>
> MQ 自带重试解决的是消息消费失败后的恢复。消息本身就在 Broker 里，天然支持重投，不需要额外落库。而且 MQ 的重试有退避策略，间隔越来越长，这个能力自带就有。
>
> 简单说：Feign 失败时没有现成的重试基础设施，所以我要自己建；MQ 天然有重试能力，直接用就行。

## 十六、Redis Lua 令牌桶限流（第一轮遗漏，补充）

### 面试官可能怎么问

> 你简历上写了 Redis Lua 限流，具体用的什么算法？怎么实现的？为什么用 Lua？

### 达标答法

> 用的是令牌桶算法，通过 Redis Lua 脚本保证原子性。
>
> **核心流程：** Redis 里用一个 Hash 存两个值——当前令牌数和上次补充时间。每次请求进来，Lua 脚本会先算时间差，按速率补充令牌，但不超过容量上限。然后看令牌够不够，够就扣减放行，不够就拒绝。整个过程在 Redis 侧一次执行完，不存在并发问题。
>
> **为什么用 Lua 而不是 Java 侧加锁？** 因为 Lua 脚本在 Redis 里是原子执行的，一次网络往返就搞定，不需要额外的分布式锁，性能更高。如果用 Java 实现，读令牌数、判断、扣减这几步之间会有并发窗口，要么加锁要么多次访问 Redis，都不如 Lua 干净。
>
> **两个值得主动提的设计细节：**
>
> 第一，时间戳不是让 Redis 自己取的，而是由 Java 侧把当前微秒时间作为参数传进 Lua，这样时间口径统一，不依赖 Redis 服务器时钟。
>
> 第二，限流器异常时默认放行，也就是 fail-open。为什么？因为限流是保护手段，不是核心业务。如果限流组件自己挂了，不应该连带把核心业务也打挂。日志里会记录异常，但请求照样放过去。

### 追问：令牌桶和漏桶有什么区别？

> **令牌桶**允许突发流量——桶里攒了令牌的话，短时间内可以消费多个，适合有突发峰值但整体要限速的场景。
>
> **漏桶**是严格匀速出水，不管进来多少请求，处理速率是固定的，适合需要严格平滑流量的场景。
>
> 我们选令牌桶是因为业务上会有短时突发（比如设备批量上报），需要允许一定的突发通过，而不是严格削平。

### 追问：这个限流用在什么场景？

> 主要用在两个地方：
>
> 第一，定时任务里控制对外部 API 的调用频率。任务注解上有个最大调用次数配置，每次发远程请求前会做原子计数，超了就中断任务，状态落成"调用次数超限"。
>
> 第二，业务接口层面的通用限流，防止上游系统突发大量请求把下游打崩。

### 关键数字

| 参数 | 示例值 | 含义 |
|---|---|---|
| capacity | 20 | 桶容量，最多攒 20 个令牌 |
| rate | 10 | 每秒补充 10 个令牌 |
| requested | 1 | 每次请求消耗 1 个令牌 |

## 十七、MQ 复杂场景——设备级熔断 + 消息过滤转发 + 保护下游（第一轮遗漏，补充）

> 基于真实代码：隔离开关位置信息消费者。这个场景同时涉及去重、设备级限流熔断、消息过滤转发、下游保护，非常适合面试展开。

### 面试官可能怎么问

> 你们消费者有没有遇到过某个设备疯狂上报消息、把系统打爆的情况？怎么处理的？

或者：

> 消费者消费完一条消息后，还要转发给另一个 Topic，这种链路你怎么设计的？

### 达标答法（完整链路）

> 有一个很典型的场景，是隔离开关的位置信息和变位信息消费。这个消费者做了四层处理：
>
> **第一层，设备级熔断检查。** 消息进来后先解析出设备 ID，去 Redis 查这个设备有没有被熔断。如果已经熔断了，直接跳过，不做任何处理。
>
> **第二层，消息去重。** 用消息里的 UUID 做 Redis 短时去重，3 分钟窗口，防止同一条消息重复消费。
>
> **第三层，设备日计数 + 自动熔断。** 去重通过后，用 Redis 原子自增给这个设备的日消息计数加 1。如果单设备单天超过 700 条，就触发熔断——往 Redis 写一个熔断标记，有效期 3 天。后续这个设备的所有消息在第一层就会被拦住。
>
> **第四层，消息过滤 + 异步转发。** 不是所有消息都需要转发，只有特定类型的消息才异步发到另一个 Topic 做进一步处理。过滤条件很轻量，直接用字符串匹配判断，不做完整 JSON 反序列化，减少性能开销。
>
> 最后才调 Service 层做真正的业务处理。

### 追问：为什么要做设备级熔断？

> 因为现场设备可能出现异常——比如某个隔离开关的传感器抖动，一天能上报几千条甚至上万条消息。如果不做限制，这一台设备就能把整个消费链路拖慢，影响其他正常设备的消息处理。
>
> 所以我做了两层保护：单日计数限制 + 超限后自动熔断 3 天。熔断期间这个设备的消息在最前面就被拦掉了，不会进入去重、计数和业务处理环节，保护了整条链路的吞吐。
>
> 另外熔断检查和计数异常时都是 fail-open，也就是 Redis 出问题了就当没熔断继续处理，不会因为保护机制本身的故障把正常消息也拦住。

### 追问：这个设计有什么可以优化的地方？

> 我觉得有几个点可以继续优化：
>
> **第一，异步转发失败没有补偿。** 现在转发失败只打了日志，消息就丢了。可以加一个失败落库机制，由后台补偿任务重新转发。
>
> **第二，JSON 解析重复了。** 消费者里解析了一次设备 ID，Service 层又会完整解析一次。可以在消费者层解析一次后把结果传给 Service，避免重复反序列化。
>
> **第三，异常处理太宽泛。** 现在所有异常都被 catch 吞掉了，等于处理失败也算消费成功，MQ 不会重投。可以区分一下：业务异常（比如数据格式不对）直接跳过；系统异常（比如数据库不可用）应该抛出去让 MQ 重试。
>
> **第四，熔断只有设备维度，没有全局维度。** 如果同时几百台设备都在大量上报，单设备没超限但总量已经很大了，可以加一个全局限流兜底。

### 场景补充：两条消息配对等待（遥信1 + 遥信2 配对推送工单）

> 这个消费者还涉及一个消息等待配对的场景。隔离开关会上报两种信号——遥信1（位置信息）和遥信2（变位信息），它们是同一个设备在差不多同一时间产生的两条独立消息。只有两条都到了并且配对成功，才能生成工单推送给下游。
>
> **问题是：** 两条消息到达时间不确定，可能间隔几百毫秒甚至几秒，不能假设它们一定同时到。
>
> **当前实现：** 每条消息先入库，然后异步检查配对。拿到遥信1后去数据库查同一设备、同一时间窗口内（10 秒）有没有遥信2；反过来也一样。查不到就短暂等待再查一次。同一设备的配对逻辑用设备级别的内存锁串行执行，防止并发竞态。配对成功后保存一条 pair 记录，再触发工单推送。
>
> **可以优化的点：** 当前等待配对用的是 Thread.sleep 轮询数据库，不够优雅而且浪费资源。更好的方案是用 Redis 做轻量级配对缓存：
>
> - 消息入库后，把自己的 ID 和信号类型写到 Redis 里，key 是设备 ID + 时间窗口，过期时间设 30 秒
> - 写入的同时检查 Redis 里有没有另一条信号的记录
> - 如果有，说明配对成功，直接拿出来做配对处理
> - 如果没有，就不等了，让后到的那条消息来触发配对
>
> 这样不需要 sleep，不需要轮询数据库，后到的消息天然会触发配对逻辑。Redis 过期时间兜底，超时没配上的自动清理。

### 追问：如果消息积压了怎么办？

> 先判断是消费者挂了还是消费太慢。
>
> **短期应急**：加消费者实例横向扩容，但不能超过 Queue 数，超了也没用。
>
> **如果是消费逻辑太重**：把消费者拆成两步——先快速落库或者写 Redis，返回消费成功；再由另一个任务异步做重逻辑处理。这样消费端永远是轻量的，不会因为下游慢而积压。
>
> **长期优化**：消费端批量处理、异步 IO、增加 Queue 数提高并行度。

## 十八、线程池（第一轮遗漏，补充）

### 面试官可能怎么问

> 线程池的核心参数有哪些？你项目里怎么用的？

或者：

> 为什么不推荐用 Executors 创建线程池？

### 达标答法：七个核心参数

> 线程池有七个核心参数：
>
> 1. **corePoolSize**：核心线程数，即使空闲也不会回收
> 2. **maximumPoolSize**：最大线程数，核心线程不够时最多能创建多少线程
> 3. **keepAliveTime**：非核心线程空闲多久后回收
> 4. **TimeUnit**：存活时间的单位
> 5. **workQueue**：任务等待队列，核心线程满了之后任务先进队列
> 6. **threadFactory**：线程工厂，可以自定义线程名方便排查
> 7. **handler**：拒绝策略，队列也满了之后怎么处理新任务

### 达标答法：执行流程

> 任务提交后的执行流程是这样的：
>
> 第一步，看核心线程有没有空闲的，有就直接执行。
>
> 第二步，核心线程都在忙，任务进等待队列排队。
>
> 第三步，队列也满了，创建非核心线程来处理，但不超过最大线程数。
>
> 第四步，线程数已经到最大、队列也满了，触发拒绝策略。
>
> **注意顺序：不是先创建到最大线程数再入队列，而是先入队列再创建非核心线程。** 这个顺序很多人搞反。

### 达标答法：为什么不用 Executors

> 因为它的默认参数太危险。
>
> `newFixedThreadPool` 和 `newSingleThreadExecutor` 用的是无界队列，任务堆积起来能把内存撑爆导致 OOM。`newCachedThreadPool` 最大线程数是 Integer.MAX_VALUE，等于不限制，可能创建几万个线程把系统打垮。
>
> 所以实际项目中都是手动创建 ThreadPoolExecutor，明确指定每个参数，队列一定用有界的。

### 达标答法：你项目里怎么用的

> 我项目里有两个典型场景用到了线程池：
>
> **第一个，数据同步任务里的并发拉取。** 比如遥测数据同步，要拉几万台设备的数据，不可能串行一台台拉。我把设备 ID 分批，每批 100 台，扔到线程池并发执行，用 Future 收集结果，主线程统一等待。线程池参数按 IO 密集型配置，核心线程数设为 CPU 核数的 2 倍，队列有界，拒绝策略用 CallerRunsPolicy，满了就让提交线程自己执行，起到自然反压的效果。
>
> **第二个，消息配对里的异步检查。** 隔离开关的遥信配对逻辑用了 @Async 异步执行，底层就是 Spring 的线程池。这样消费者不会被配对逻辑阻塞，能继续消费下一条消息。

### 追问：参数怎么定的？CPU 密集和 IO 密集有什么区别？

> **CPU 密集型**（计算多、IO 少）：核心线程数 = CPU 核数 + 1，再多线程也没用，CPU 已经跑满了。
>
> **IO 密集型**（数据库、网络调用多）：核心线程数 = CPU 核数 × 2，因为线程大部分时间在等 IO，多开一些线程能提高 CPU 利用率。更精确的公式是 `线程数 = CPU 核数 × (1 + IO 耗时 / CPU 耗时)`。
>
> 我的数据同步任务是典型 IO 密集型——大量远程调用和数据库写入，所以按 CPU 核数 × 2 配的。

### 追问：四种拒绝策略分别是什么？你选了哪个？

> - **AbortPolicy**：直接抛异常，默认策略
> - **CallerRunsPolicy**：让提交任务的线程自己执行，起到限流反压效果
> - **DiscardPolicy**：静默丢弃，不抛异常
> - **DiscardOldestPolicy**：丢弃队列里最老的任务，把新的放进去
>
> 我选的是 CallerRunsPolicy。因为对于数据同步来说，宁可慢一点也不能丢任务。让调用线程自己执行，相当于在过载时自动降速，而不是丢数据或者抛异常。

### 关键数字

| 场景 | 核心线程数 | 队列 | 拒绝策略 |
|---|---|---|---|
| CPU 密集 | CPU 核数 + 1 | 有界 | 按业务选 |
| IO 密集 | CPU 核数 × 2 | 有界 | 推荐 CallerRunsPolicy |
| 绝对不能用 | Executors 工厂方法 | 无界队列 | 会 OOM |

## 十九、高吞吐数据同步链路你是怎么做的？（得分：未考 → 目标：7）

### 目标答案

> 我们有一批定时任务负责从外部系统拉取设备遥测数据，写入本地数据库。数据量比较大，单次任务要处理几千台设备、几十万条记录。
>
> **核心设计有三个点：**
>
> **第一，时间窗口滚动拉取。** 不是每次全量拉，而是按最近 30 分钟的窗口增量拉取，减少重复数据和接口压力。
>
> **第二，并发控制。** 设备 ID 分批后通过线程池并发执行，子任务用 Future 收集结果，主线程统一等待。线程池配的是 IO 密集型参数，核心线程数按 CPU 核数 × 2。
>
> **第三，可中断。** 子任务执行过程中会检查中断状态，如果任务平台发出停止指令或应用要停机，能及时中断。Future.get 抛出中断异常时会显式恢复中断标志，不会吞掉。
>
> 分批和 OOM 治理的细节我在 OOM 排查那个问题里讲过——早期全量拉撑爆内存，改成每批 100 台设备、每批 1000 条入库之后解决的。

### 追问：为什么不用 MQ 做数据同步？

> 因为数据源是外部系统的 HTTP 接口，不是消息推送。我们是主动拉取的模式，这种场景更适合定时任务 + 批量入库。拉回来的数据如果要触发下游业务联动（比如告警判定），这一步才会发 MQ 消息异步处理。

### 追问：数据重复怎么处理？

> 入库时有设备 ID + 时间戳的联合唯一索引，批量插入用 `INSERT IGNORE`，数据库层面兜底。

## 二十、任务启动恢复是怎么做的？（得分：未考 → 目标：7）

### 目标答案

> 生产上任务执行到一半的时候应用可能会重启，比如发版或者机器故障。这时候任务日志表里会留下一些状态是"运行中"的脏数据，Redis 里的调用计数也可能和数据库不一致。
>
> 我们在应用启动时挂了一个恢复程序，用的是 Spring 的 `CommandLineRunner`，它会做四件事：
>
> **第一，** 查出当前应用下所有遗留的"运行中"状态的任务日志，把它们改成"系统关闭中断"，这样运维看日志就知道这些任务不是正常结束的。
>
> **第二，** 从 Redis 里恢复这些任务上次的调用计数和 URL 调用明细，同步到数据库里，保证统计数据不丢。
>
> **第三，** 更新任务配置表里的最近运行状态、耗时、日/月累计次数，让管理界面显示的数据和实际一致。
>
> **第四，** 清理 Redis 里的临时状态，避免残留数据影响下一次执行。
>
> **核心思路就是：** 不依赖 finally 收尾，因为进程被 kill 的时候 finally 不一定能执行。而是在下一次启动时做一次状态修复，保证日志表、配置表和 Redis 最终一致。

### 追问：如果启动恢复本身也失败了呢？

> 恢复程序本身是幂等的，跑多次结果一样。如果恢复过程中抛异常，会记录错误日志但不影响应用启动。最坏情况就是某几条日志状态没修复，后面可以人工补。不能因为恢复失败就让整个应用起不来。

## 二十一、MySQL 索引和 B+ 树（得分：未考 → 目标：7）

### 目标答案

> **B+ 树是 InnoDB 的默认索引结构，** 它和 B 树最大的区别有两个：
>
> **第一，非叶子节点只存 key 不存数据。** 这样一个节点能放更多索引项，树更矮，查询时磁盘 IO 次数更少。一般 3-4 层的 B+ 树就能存几千万条数据。
>
> **第二，叶子节点之间有双向链表。** 范围查询的时候可以直接沿着链表遍历，不需要回到上层节点，这对 SQL 里的 BETWEEN、ORDER BY 这些操作非常友好。
>
> **聚簇索引和非聚簇索引的区别：** 主键索引是聚簇索引，叶子节点存的是完整行数据。普通索引是非聚簇索引，叶子节点只存主键值，查完还要拿着主键去主键索引再查一次，这就是回表。要避免回表就用覆盖索引，让查询字段都在索引里。

### 追问：联合索引的最左前缀原则是什么？

> 联合索引 `(a, b, c)` 是按 a → b → c 的顺序排列的。查询时必须从最左列开始匹配，才能用上索引。
>
> - `WHERE a = 1 AND b = 2` → 能用上 a 和 b
> - `WHERE b = 2` → 用不上索引，因为跳过了 a
> - `WHERE a = 1 AND c = 3` → 只能用上 a，c 用不上（中间断了 b）
> - `WHERE a > 1 AND b = 2` → a 能用上，但 a 用了范围查询后 b 就用不上了

### 追问：索引什么时候会失效？

> 常见的有六种：
> 1. 左模糊 `LIKE '%xxx'`，因为不知道前缀，没法走 B+ 树
> 2. 对索引列用了函数或计算，比如 `WHERE YEAR(create_time) = 2024`
> 3. 隐式类型转换，比如 varchar 列用数字查 `WHERE phone = 13800138000`
> 4. OR 条件中有一边没索引
> 5. 违反最左前缀原则
> 6. 优化器判断全表扫描比走索引更快（数据量小或区分度低时）

## 二十二、Redis 缓存穿透、击穿、雪崩（得分：未考 → 目标：7）

### 目标答案

> **缓存穿透：** 查的数据在数据库里根本不存在，缓存里自然也没有，每次请求都打到数据库。
>
> 解决方案有两个：一是布隆过滤器，在缓存前加一层拦截，快速判断数据是否存在；二是缓存空值，查到不存在就缓存一个空结果，设个短过期时间比如 2-5 分钟，防止恶意请求反复打数据库。
>
> **缓存击穿：** 某个热点 key 刚好过期，大量并发请求同时打到数据库。
>
> 解决方案：一是用互斥锁重建缓存，第一个请求用 `SET NX PX` 拿到锁去查数据库、更新缓存，其他请求等一下或者返回旧值；二是逻辑过期，key 永不过期，但 value 里带一个过期时间字段，发现过期了异步更新，请求先返回旧数据。
>
> **缓存雪崩：** 大量 key 同时过期，或者 Redis 整体宕机，请求全打到数据库。
>
> 解决方案：第一，设置过期时间时加随机偏移，避免同一时间集中过期；第二，热点数据可以不设过期时间，由后台任务定期更新；第三，如果是 Redis 宕机，就要做好降级和限流，比如本地缓存兜底。

### 追问：你项目里哪里用到了这些方案？

> 我的项目里用 Redis 做短时去重和设备级计数（每日消息上限 700），这些 key 都设了 TTL。为了防止 Redis 异常导致所有消息被拦截或全部放行，我做了 fail-open 设计——Redis 不可用时默认放行消息，宁可重复处理也不能丢消息。这其实就是缓存雪崩场景下的一种降级策略。

## 二十三、Spring 事务传播机制和失效场景（得分：未考 → 目标：7）

### 目标答案

> **@Transactional 的默认传播行为是 REQUIRED，** 意思是如果当前已经有事务了就加入，没有就新建一个。这是最常用的。
>
> **最常考的传播行为有三种：**
>
> - **REQUIRED（默认）：** 加入当前事务，没有就新建。大多数场景用这个。
> - **REQUIRES_NEW：** 不管当前有没有事务，都新建一个独立事务。比如记操作日志，不想因为主业务回滚把日志也回滚了。
> - **NESTED：** 在当前事务内创建一个保存点，内层回滚只回到保存点，不影响外层事务。
>
> **@Transactional 什么时候会失效？五种常见情况：**
>
> 1. **方法不是 public 的。** Spring 事务基于 AOP 代理，非 public 方法代理拦截不到。
> 2. **同类自调用。** 比如 A 方法调 B 方法，两个方法在同一个类里，B 上加了 @Transactional 也不生效，因为走的是 this 调用不是代理对象。
> 3. **异常被 catch 吞掉了。** Spring 默认只对 RuntimeException 和 Error 回滚，如果你 catch 了异常没往外抛，事务不会回滚。
> 4. **数据库引擎不支持事务。** 比如 MyISAM。
> 5. **传播行为设置不当。** 比如内层事务用了 NOT_SUPPORTED，它会挂起当前事务以非事务方式执行。

### 追问：你项目里怎么用事务的？

> 比如故障跳闸消息消费的场景，落主记录和落子事件是在同一个事务里的，保证要么都成功要么都回滚。但是发短信通知是异步的，放在事务外面，不会因为通知失败导致整个事务回滚。这就是把"主事务"和"副作用"拆开的思路。

## 二十四、分布式事务怎么保证一致性？（得分：未考 → 目标：7）

### 目标答案

> 分布式事务的核心问题是：跨服务调用时，一个操作成功另一个失败，怎么保证数据一致。
>
> **常见方案有四种，我按实际用的多少来排：**
>
> **第一，本地消息表。** 最常用。业务操作和写消息表放在同一个本地事务里，然后定时任务扫描消息表，把消息发出去。下游消费成功就标记完成，失败就重试。我的 Feign 失败补偿其实就是这个思路——调用失败的上下文落到本地数据库，后台任务定期重放。
>
> **第二，RocketMQ 事务消息。** 先发半消息，执行本地事务，成功就提交消息，失败就回滚。如果超时没回复，Broker 会定期回查 Producer。适合异步场景，比如下单后通知扣库存。
>
> **第三，TCC（Try-Confirm-Cancel）。** 分成三步：Try 预留资源，Confirm 确认执行，Cancel 释放资源。一致性最强但代码复杂，每个参与方都要写三个接口。
>
> **第四，Seata AT 模式。** 阿里开源的框架，对业务代码侵入小，自动生成反向 SQL 做回滚。但需要额外部署 Seata Server。
>
> **实际选型：** 我们项目里没用 Seata 或 TCC，因为场景不需要强一致。跨服务调用用的是 Feign 补偿重试（类似本地消息表），异步消息用的是 RocketMQ 两层幂等。只要最终一致就够了。

### 追问：Feign 补偿和本地消息表的区别是什么？

> 本质思路一样，都是"本地落库 + 后台重放"。区别在于本地消息表一般是业务主动写消息记录，而我的 Feign 补偿是 AOP 自动拦截失败调用，把完整的调用上下文（服务名、方法名、参数、异常信息）自动持久化，不需要业务代码手动写。好处是对业务无侵入，坏处是只能处理同步调用失败的场景。

## 二十五、Spring Bean 生命周期和循环依赖（得分：未考 → 目标：7）

### 目标答案

> **Bean 生命周期四个阶段：**
>
> **第一，实例化。** Spring 通过反射创建 Bean 对象，这时候只是一个空对象，属性还没注入。
>
> **第二，属性注入。** 把 @Autowired、@Value 这些依赖注入进去。
>
> **第三，初始化。** 先调 @PostConstruct 标注的方法，再调 InitializingBean 的 afterPropertiesSet，最后调 init-method。如果有 AOP，这一步会生成代理对象。
>
> **第四，销毁。** 容器关闭时先调 @PreDestroy，再调 DisposableBean 的 destroy，最后调 destroy-method。
>
> **循环依赖怎么解决：** Spring 用三级缓存。
>
> - **一级缓存：** 存完整的、初始化好的 Bean
> - **二级缓存：** 存早期暴露的 Bean（已实例化但未初始化）
> - **三级缓存：** 存 Bean 的工厂对象（ObjectFactory）
>
> 流程是这样的：创建 A 时发现依赖 B，先把 A 的工厂放进三级缓存，然后去创建 B。创建 B 时发现依赖 A，从三级缓存拿到 A 的工厂，创建 A 的早期引用放到二级缓存。B 拿到 A 的早期引用就能完成初始化。最后 A 也能完成初始化。
>
> **注意：** 构造器注入的循环依赖解决不了，因为对象都没创建出来就没法放缓存。只有 setter 注入或 @Autowired 字段注入的才能解决。

### 追问：为什么要三级缓存，两级不行吗？

> 两级缓存能解决简单的循环依赖，但如果有 AOP 代理就不行了。三级缓存存的是 ObjectFactory，它可以在需要时生成代理对象。这样即使 A 被 AOP 增强了，B 拿到的也是 A 的代理对象，而不是原始对象。如果只有两级缓存，没法在早期暴露阶段就决定是否要生成代理。

## 二十六、慢查询排查你是怎么做的？（得分：未考 → 目标：7）

### 目标答案

> 我之前在电网项目里碰到过一个典型的慢查询，用 STAR 结构讲：
>
> **背景：** 有个设备台账查询接口，联表查了设备基本信息、在线监测数据和最近的试验记录，页面加载有时候要 3-4 秒。
>
> **定位过程：**
>
> 第一步，开慢查询日志，找到耗时最长的 SQL。
>
> 第二步，用 EXPLAIN 分析，发现两个问题：一个是 type 是 ALL，全表扫描；另一个是 Extra 里有 Using filesort，说明排序也没走索引。
>
> 第三步，看了表结构和索引，发现查询条件用的是设备编号 + 时间范围，但联合索引的顺序不对，导致索引失效。
>
> **解决：**
>
> 1. 调整联合索引顺序，把区分度高的字段放前面
> 2. 把 `SELECT *` 改成只查需要的字段，部分查询走了覆盖索引不用回表
> 3. 大偏移量的分页改成 `WHERE id > last_id LIMIT N`，避免深分页扫描大量数据
>
> 优化后接口从 3-4 秒降到 200ms 以内。

### 追问：EXPLAIN 你重点看哪几个字段？

> 三个：
> - **type：** 看访问类型，ALL 就是全表扫描必须优化，至少要到 ref 或 range
> - **key：** 看有没有命中索引，NULL 就是没走索引
> - **Extra：** 看有没有 Using filesort（额外排序）和 Using temporary（临时表），有的话要优化。如果是 Using index 就是覆盖索引，这是好的

## 二十七、Kafka 核心机制（得分：未考 → 目标：7）

### 目标答案

> 我了解 Kafka，虽然当前项目用的是 RocketMQ，但目标公司用到了 Kafka，核心机制我比较清楚。
>
> **Kafka 为什么吞吐量高？四个设计：**
>
> 1. **顺序写磁盘。** 消息追加写入 Partition 日志文件，不做随机写，顺序写速度接近内存。
> 2. **Page Cache。** 不自己管缓存，利用操作系统页缓存，写先到页缓存，读命中就不走磁盘。
> 3. **零拷贝。** 消费者拉消息时用 sendfile 系统调用，数据从磁盘到网卡不经过用户态。
> 4. **批量和压缩。** Producer 攒 batch 再发，支持 snappy/lz4 压缩。
>
> **怎么保证不丢消息？三个环节：**
>
> - **Producer：** `acks=all`，等 ISR 所有副本确认 + retries 重试
> - **Broker：** `replication.factor ≥ 3`，`min.insync.replicas = 2`
> - **Consumer：** 关闭自动提交 offset，处理成功后手动提交
>
> **ISR 机制：** 和 Leader 保持同步的副本集合。Follower 落后太久（30 秒内没追上）被踢出 ISR，追上后重新加入。Leader 挂了从 ISR 里选新 Leader。

### 追问：Kafka 和 RocketMQ 怎么选？

> 一句话：大数据场景用 Kafka，业务消息用 RocketMQ。
>
> Kafka 吞吐量百万级，和 Flink/Spark 生态集成好，适合日志采集、埋点、流计算。RocketMQ 吞吐量十万级但功能更丰富，原生支持延迟消息、事务消息、按 key 查询，适合订单、支付、告警这种对可靠性要求高的业务场景。
>
> 我们电网项目选 RocketMQ 就是因为需要事务消息和消息查询，吞吐量十万级够用了。

## 二十八、数据库死锁怎么排查？（得分：未考 → 目标：7）

### 面试官常见问法

> - "你遇到过数据库死锁吗？怎么排查的？"
> - "死锁产生的条件是什么？怎么避免？"

### 目标答案

> **死锁四个必要条件：** 互斥、占有且等待、不可抢占、循环等待。四个同时满足才会死锁。
>
> **排查流程：**
>
> 第一步，看日志。InnoDB 检测到死锁会自动回滚其中一个事务，错误日志里会有 `Deadlock found when trying to get lock` 的记录。
>
> 第二步，用 `SHOW ENGINE INNODB STATUS\G` 查看最近一次死锁的详细信息，能看到两个事务分别持有什么锁、在等什么锁、执行的 SQL 是什么。
>
> 第三步，分析加锁顺序。死锁的根本原因几乎都是两个事务以相反的顺序加锁。比如事务 A 先锁订单表再锁库存表，事务 B 先锁库存表再锁订单表，就会互相等待。
>
> **解决和预防：**
>
> 1. **统一加锁顺序。** 所有事务都按固定顺序访问表和行，比如都先锁订单再锁库存
> 2. **缩短事务。** 事务越短持锁时间越短，死锁概率越低。不要在事务里做远程调用或重计算
> 3. **合理加索引。** 没有索引的 UPDATE/DELETE 会锁全表而不是锁行，大幅增加死锁概率
> 4. **InnoDB 自动检测。** `innodb_deadlock_detect = ON`（默认开启），检测到死锁后自动回滚代价更小的那个事务

### 追问：你项目里遇到过死锁吗？

> 遇到过。在工单流转的场景里，有两个并发操作——一个是告警触发创建工单并更新告警状态，另一个是人工处理工单并回写告警记录。两个操作访问告警表和工单表的顺序不一致，偶发死锁。
>
> 排查出来之后，把两边的 SQL 都改成先操作告警表再操作工单表，统一了加锁顺序，死锁就没再出现过。同时把事务范围缩小，不必要的查询移到事务外面。
