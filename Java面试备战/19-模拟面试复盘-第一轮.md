## 二、任务扫描注册流程（得分：4/10 → 目标：7）

### 面试官问题

> 你的 Starter 启动的时候，任务是怎么被发现和注册的？具体流程是什么？

### 我的回答

> 我写了注解，只要定时任务上加上这个注解，项目启动的时候就会发现和注册。

### 问题

- 等于什么都没说
- 关键环节全丢：谁去扫描、扫描完做什么、元数据存哪
- 会让面试官怀疑你没写过这个东西

### 达标答法

> 我定义了一个自定义注解，上面会标任务分组、cron 表达式、负责人、最大调用次数这些治理信息。
>
> 启动时有一个扫描器，实现了 Spring 的 CommandLineRunner，容器初始化完成后自动触发。它会扫描所有 Bean 上标了这个注解的方法，把注解里的元数据提取出来。
>
> 提取完之后，跟数据库里已有的任务配置做比对——新任务就插入，已有的就更新 cron 和分组。这样每次启动，数据库里的任务清单就和代码里的注解保持同步了。后续动态启停、执行监控都是基于这张配置表来驱动的。

## 三、动态调度感知（得分：5/10 → 目标：7）

### 面试官问题

> 任务实际调度是怎么触发的？是 Spring 的 @Scheduled 在跑，还是你自己接管了调度？数据库里改了 cron，运行中的应用能感知到吗？

### 我的回答

> 我直接接管了调度，可以感知到，组件会开一个线程每分钟去查一次。

### 问题

- "接管了调度"怎么接管的没说
- 查完之后发现变化了怎么处理没说
- 动态启停怎么做没说

### 达标答法

> 我没有用 Spring 的 @Scheduled，而是自己接管了调度。启动时扫描器把任务注册完之后，会用 Spring 的 TaskScheduler 按数据库里的 cron 表达式注册调度，拿到的 ScheduledFuture 会缓存在内存 Map 里。
>
> 然后组件会开一个刷新线程，每分钟从数据库重新拉一次任务配置。如果发现某个任务的 cron 变了，就先 cancel 掉内存里旧的 ScheduledFuture，再按新 cron 重新注册一个。如果发现任务被禁用了，就直接 cancel，不再触发。
>
> 所以运维人员改数据库配置后，最多一分钟就能生效，不需要重启应用。

## 四、分布式锁实现（得分：8/10 ✓ 已达标）

### 面试官问题

> 两个实例同时触发同一个任务，怎么保证只有一个在跑？分布式锁具体怎么实现的？

### 我的回答

> 定时任务启动的时候，会基于自身生成一个 token，然后用 setIfAbsent 方法获取锁并设置 TTL 防止任务失败锁一直没有释放。获取到锁之后就会启动自动续时任务，主要任务还在执行就一直续期，防止长任务的时候锁被释放。任务执行完或者异常则基于 Redis 的 Lua 结合 token 释放锁，保证锁不会被别的线程释放。

### 可以更好

- 补上数字：TTL 120 秒，每 30 秒续租一次
- 主动提对比：业务仓库里的基础分布式锁解锁是先读后删不是原子的，但调度平台里做了 Lua 原子释放

### 加分补充

> TTL 默认 120 秒，续租线程每 30 秒跑一次，用 Lua 脚本先比对 token 再 expire，保证续租也是原子的。
>
> 另外说一句，业务仓库里还有一个基础的分布式锁工具类，那个解锁是先读再删，不是原子的。我知道这个差距，调度平台里的锁要求更严谨所以做了升级。

## 五、RocketMQ 故障跳闸消费流程（得分：5/10 → 目标：7）

### 面试官问题

> 一条故障跳闸消息进来，Consumer 从收到消息到处理完成，中间经过了哪些步骤？重点讲怎么防重复消费的，处理失败会发生什么？

### 我的回答

> 收到消息后先用 Redis 短时内去重，MySQL 唯一索引兜底，数据进入数据库之后再提交 offset。

### 问题

- Redis 去重的 key 是什么、窗口多长没说
- 处理失败的链路完全没讲
- "提交 offset"说法不准确，RocketMQ 是返回 CONSUME_SUCCESS，不是手动提交 offset
- 故障跳闸的业务处理（子事件、短信通知）没提

### 达标答法

> 消息进来后分几步：
>
> **第一步，Redis 快速去重。** 拿消息里的唯一 ID 往 Redis 写一个处理中的标记，设 3 分钟过期。如果这个标记已经存在，说明这条消息正在被处理或者刚处理完，直接跳过不处理。
>
> **第二步，业务处理。** 把故障跳闸主记录落库，同时落子事件和后果记录。这里有个细节——只有本次真正新插入的数据才会触发异步短信通知，重试进来的不会重复发。
>
> **第三步，处理成功后延长标记。** 把 Redis 里那个标记的过期时间延长到 10 分钟。为什么？因为处理窗口是 3 分钟，成功标记是 10 分钟，这样刚处理完的消息不会因为标记过期又被当成新消息。
>
> **第四步，业务层兜底幂等。** Redis 去重只是第一层快速拦截，真正的业务幂等在 service 层按业务键做，比如按设备 ID 加故障时间建唯一索引。
>
> **如果处理失败：** 清掉 Redis 的处理中标记，抛异常让 MQ 重新投递。但最多重试 3 次，加上初始那次总共 4 次机会，超了就不再重试，避免一条坏消息把消费者卡死。

### 表达原则

- 讲机制，不念方法名。比如不要说"调 tryAcquireByMessage"，要说"往 Redis 写一个处理中标记"
- 面试官不知道你代码里的方法叫什么，他只关心你做了什么、为什么这么做
- Spring / Redis / MQ 的标准术语可以说，你自己项目里的类名和方法名不要说

## 六、副作用控制——怎么判断数据是新插入的（得分：未答出）

### 面试官问题

> 你说"只对新插入的数据才发短信，重试不会重复发"。你怎么判断这条数据是本次新插入的，还是之前已经存在的？

### 考点

- 幂等和副作用控制的意识
- 落库是幂等的（唯一索引），但发短信是副作用，不受唯一索引保护

### 达标答法

> 落库的时候，数据库 insert 会返回影响行数。如果返回 1，说明是本次新插入的；如果返回 0 或者抛唯一索引冲突，说明之前已经插入过了。
>
> 我只对真正新插入成功的记录才触发异步短信。这样即使消息重试进来，走到落库这一步发现数据已存在，就不会再发短信。
>
> 本质上就是把落库当判断依据，把通知当后置动作，而不是两件事一起做。

### 设计原则

**先幂等写入，再根据写入结果决定是否触发副作用。**

## 七、Feign 补偿幂等（得分：5/10 → 目标：7）

### 面试官问题

> 补偿任务重试的时候，怎么保证被调用方不会重复处理？第一次调成功了但网络超时没收到响应，你以为失败了落库准备补偿，被调方岂不是处理两次？

### 我的回答

> 被调用方是单线程执行的，并且通过业务键进行去重。

### 问题

- "单线程执行"说法不准确，Spring Boot 默认多线程处理请求
- "业务键去重"太笼统，怎么去重没说

### 达标答法

> 每次 Feign 调用都会带一个业务唯一 ID，比如工单号、设备 ID 加时间戳的组合。被调方处理的时候，按这个业务 ID 做幂等校验——最直接的就是数据库唯一索引，第二次进来 insert 冲突，直接返回成功。
>
> 补偿任务重试时，用的是跟第一次完全一样的参数和业务 ID，被调方发现已经处理过就直接返回。
>
> 这也是为什么落库时要保留完整的原始参数，而不是重新生成——就是为了保证重试和首次调用的幂等基准一致。

## 八、补偿重试策略（得分：7/10 ✓ 基本达标）

### 面试官问题

> 补偿任务是无脑扫表重试，还是有什么策略？目标服务挂了你还一直重试？

### 我的回答

> 定时器每 5 分钟发起一次重试，重试之前先检查服务的心跳，判断服务是否正常，正常才发起重试。

### 可以更好

- 有没有按服务分组？A 挂了 B 正常，是不是只跳过 A？
- 重试次数有上限吗？
- 面试时不要重复同一句话

### 加分版

> 补偿任务每 5 分钟跑一次，但不是无脑扫全表重试。
>
> 第一步，按目标服务分组，把待重试的记录按服务名归类。
>
> 第二步，逐个服务做健康检查，查目标服务有没有可用实例。整个服务不可用的话，这一批直接跳过。
>
> 第三步，才发起真正的重试，用落库时保留的原始参数重放。
>
> 第四步，有上限兜底。每条记录有最大重试次数，超过标记为最终失败。
>
> 这样补偿机制不会在服务故障时变成压力放大器。

## 十、Slot 识别链路（得分：8/10 ✓ 已达标）

### 面试官问题

> "110kV 及以上断路器 SF6 气体泄漏年超过 50mL 且运行年限超过 15 年的设备生成预警"，这句话你的系统怎么处理的？

### 我的回答

> 先用正则表达式抽取结构化信息，分词器进行分词，通过 AC 自动机提取设备、场景、指标。如果缺少 slot 先尝试根据已有信息反推，比如根据设备和指标可以推出场景。如果存在多候选对象则根据语义距离、共现模式、实体特异性筛选最合适候选人。无法推断则触发澄清。把得到的结果打包发给 LLM 获取结果。

### 可以更好

- 主动指出"运行年限超过 15 年"容易被误识别成时间窗口的 bad case
- 后半段打分融合和前面链路之间的过渡要说清楚
- 补上 DSL 校验环节

### 要主动补的 bad case

> 这句话有个典型难点——"运行年限超过 15 年"，大模型很容易把它当成时间窗口来处理，但它其实是设备属性过滤条件，应该放在设备筛选条件里而不是指标条件里。我们在 DSL 校验阶段会检查这种归类错误，发现位置不对会自动纠正。

## 十一、混合检索（得分：6.5/10 → 目标：7）

### 面试官问题

> 为什么不直接用向量检索？加 BM25 解决了什么问题？两路结果怎么融合的？

### 我的回答中有两个硬伤

| 我说的 | 正确的 |
|---|---|
| 系数 0.6 | k 参数 60，不是权重，是排名平滑因子 |
| 相同含义语义距离远需要关键词检索 | 说反了，BM25 优势是精确关键词匹配 |

### 达标答法

> 电网领域两种情况都有：
>
> **纯向量的问题**：SF6 泄漏和油气泄漏语义相近，但业务含义完全不同，向量检索会搞混。BM25 靠精确关键词匹配，能区分开。
>
> **纯 BM25 的问题**："超过额定值"和"大于标准值"关键词不同但含义一致，BM25 搜不到。向量检索靠语义相似性能找到。
>
> 所以两路互补。融合用的是 RRF 算法，核心思想是两路检索分别给每个文档一个排名，两路都排前面的文档总分最高。k 取 60，是排名平滑参数，在 20 到 100 之间调过，60 在我们 300 条规则语料上效果最好。

### 必须记牢

- BM25 赢在精确匹配行业术语
- 向量赢在语义相似但用词不同
- RRF 的 k=60，不是权重系数

## 十二、回归决策（得分：6/10 → 目标：7）

### 面试官问题

> 改了一版 prompt，68 条里 3 条从对变错了，5 条从错变对了，你怎么决定要不要上？

### 我的回答

> 要上，我们跑回归集看的是准确率，低于就重新调，高于就可以发版。

### 问题

- 只看总准确率太粗
- 变错的 3 条可能比变对的 5 条更重要

### 达标答法

> 不能只看总数。分三步判断：
>
> 第一步，先看变错的那 3 条是什么。如果是核心高频场景，比如断路器、主变这类主力设备，哪怕整体准确率提升了也不能直接上，得先修。
>
> 第二步，分析变错的原因。是 prompt 改动影响了意图识别？还是检索召回的范文变了？搞清原因才知道能不能定向修复。
>
> 第三步，如果变错的都是边缘场景，变对的覆盖了更重要的业务场景，那可以上。同时把那 3 条加入回归基线，确保下次不再退化。
>
> 准确率是门槛，但最终决策要看变错的 case 影响面有多大。

## 十三、HashMap put 流程（得分：4/10 → 目标：7）

### 面试官问题

> HashMap 在 JDK 1.8 里，put 一个元素的完整流程讲一下。

### 我的回答中的硬伤

| 我说的 | 正确的 |
|---|---|
| 桶中节点大于 12 个或总节点大于 64 触发扩容 | 元素总数 > 容量 × 0.75 时扩容；链表 ≥ 8 且容量 ≥ 64 时树化 |
| 没提 hash 扰动 | 高 16 位和低 16 位异或 |
| 没提定位方式 | (n-1) & hash 位运算 |
| 没提 key 相同覆盖 | 相同 key 覆盖 value |

### 达标答法

> **第一步，算 hash。** 拿 key 的 hashCode，高 16 位和低 16 位异或，让高位参与运算减少冲突。
>
> **第二步，定位桶。** 用 (n-1) & hash 算桶下标，位运算代替取模，所以容量必须是 2 的幂。
>
> **第三步，放元素：**
> - 桶是空的，直接放
> - 桶里第一个节点 key 相同，覆盖 value
> - 桶里是链表，遍历到尾部追加，中间发现相同 key 就覆盖
> - 桶里是红黑树，走树的插入逻辑
>
> **第四步，判断树化。** 链表追加后长度 ≥ 8，再看数组容量是否 ≥ 64。都满足才转红黑树，容量不够就先扩容。
>
> **第五步，判断扩容。** 元素总数超过容量 × 0.75，触发扩容，数组翻倍，所有元素重新定位。

### 关键数字

| 参数 | 值 |
|---|---|
| 默认容量 | 16 |
| 负载因子 | 0.75 |
| 树化阈值 | 链表 ≥ 8 且容量 ≥ 64 |
| 退化阈值 | 红黑树节点 ≤ 6 |

## 十四、OOM 排查（得分：5/10 → 目标：7）

### 面试官问题

> 讲一个你实际遇到的线上 OOM。

### 问题

- 讲的是教科书流程，不是具体故事
- "Full GC 后下降就扩充老年代"判断太草率
- 原来准备的是银行项目的故事，太久远了，换成电网项目更好

### 达标答法（讲电网项目数据同步 OOM）

> 在变电 3.0 项目里遇到过一次数据同步任务 OOM。
>
> **现象**：有一个定时任务负责拉取设备在线监测数据并入库，某天任务跑到一半服务直接挂了，日志抛 OutOfMemoryError: Java heap space。
>
> **定位**：启动参数配了 -XX:+HeapDumpOnOutOfMemoryError，自动 dump 了堆快照。用 MAT 打开 Dominator Tree，发现一个 ArrayList 占了大量内存，顺着引用链往上查，定位到数据同步 Service 里——它一次性把几万台设备的监测数据全查出来放在内存里做处理。
>
> **原因**：设备规模增长后，单次全量拉取的数据量远超预期，内存直接撑爆。
>
> **解决**：改成设备 ID 分批拉取，每批 100 台设备，拉完处理完再拉下一批。写库也分批，每批 1000 条 insert。处理完一批就释放引用，不会一直持有全量数据。
>
> **效果**：内存峰值从接近堆上限降到正常水平，任务稳定运行。后来我还把这个分批策略作为任务开发规范推广到了其他同步任务里。

### 备选故事（银行项目，如果面试官追问早期经验）

> 银行项目里也遇到过类似问题——财务报表统计一次性加载全量交易流水到内存做汇总，改成分页查询 + 分批处理解决的。

### 要点

- 讲排查问题用 STAR 结构：现象 → 定位 → 原因 → 解决 → 效果
- 优先讲最近项目的故事，面试官更感兴趣
- 不要讲通用流程图，要讲"我遇到了什么、我怎么查的、最后怎么改的"

## 十四补充、CPU 飙高排查

### 面试官问题

> 你遇到过 CPU 飙高的问题吗？怎么排查的？

### 达标答法

> 遇到过。在电网项目里有一次定时任务高峰期，监控告警 CPU 使用率持续 90% 以上。
>
> **排查过程：**
>
> 第一步，先用 `top` 看整机 CPU，确认是哪个 Java 进程占用高。机器上跑了多个服务，用 `jps -l` 确认是数据处理服务。
>
> 第二步，用 `top -H -p <pid>` 找出这个进程里 CPU 最高的线程，记下线程 ID。
>
> 第三步，把线程 ID 转成十六进制，再用 `jstack` 导出线程栈，搜这个十六进制 ID 对应的线程。
>
> 第四步，看线程栈发现是某个数据清洗逻辑里有一段循环在做大量字符串拼接和正则匹配，数据量一大就变成了 CPU 热点。
>
> **解决**：把字符串拼接改成 StringBuilder，正则表达式预编译复用，同时对这段逻辑做了分批处理，避免单次处理数据量过大。
>
> **效果**：CPU 使用率从 90% 降到 40% 左右。

### 追问：如果不是代码热点，还可能是什么原因？

> 还有几种常见情况：
>
> - **频繁 Full GC**：GC 线程一直在跑，可以通过 GC 日志确认。解决思路是查内存泄漏或者调大堆。
> - **死循环**：线程栈里会看到同一个方法反复出现。
> - **锁竞争严重**：大量线程卡在 BLOCKED 状态等锁，实际上 CPU 花在线程调度上了。

### 要点

- 排查步骤要记牢：top → jps → top -H → 线程 ID 转十六进制 → jstack
- 最后一定要落到"具体是什么代码、怎么改的"，不能只停在"找到线程"

## 十五、Feign 补偿 vs MQ 重试为什么不统一（得分：6/10 → 目标：7）

### 面试官问题

> 一个用落库补偿，一个用 MQ 自带重试，为什么不统一成一种方案？

### 我的回答

> Feign 调用是系统微服务之间的调用，MQ 是外部系统和我们系统之间的调用，并不能统一。

### 达标答法

> 这两种重试解决的是不同层面的问题。
>
> Feign 补偿解决的是同步调用失败后的恢复。我需要保留完整的调用上下文——哪个服务、哪个方法、什么参数——后面才能按原样重放。如果硬把每次 Feign 失败都发消息到 MQ 再消费重试，多引入一层中间件依赖，链路更复杂更难排查。
>
> MQ 自带重试解决的是消息消费失败后的恢复。消息本身就在 Broker 里，天然支持重投，不需要额外落库。而且 MQ 的重试有退避策略，间隔越来越长，这个能力自带就有。
>
> 简单说：Feign 失败时没有现成的重试基础设施，所以我要自己建；MQ 天然有重试能力，直接用就行。

## 十六、场景设计——订单下游通知（得分：7/10 ✓ 基本达标）

### 面试官问题

> 下单后要通知库存扣减、支付创建预支付单、风控校验，怎么设计？

### 我的回答

> 库存扣减和支付创建用同步调用保证一致性，风控校验可以异步来做。

### 可以更好

- 同步调用如果库存成功但支付失败怎么办？
- 异步用什么方式？

### 加分版

> 分两类：
>
> **同步链路：库存 + 支付。** 直接影响下单结果，必须同步。库存成功但支付失败要回滚库存。用本地事务加 Feign 调用，失败走补偿回滚，不用重量级分布式事务。
>
> **异步链路：风控。** 不阻塞主流程，发消息到 MQ，风控服务异步消费。发现问题再反过来冻结订单。
>
> 主流程响应快，风控又不拖慢用户体验。

## 十七、Redis Lua 令牌桶限流（第一轮遗漏，补充）

### 面试官可能怎么问

> 你简历上写了 Redis Lua 限流，具体用的什么算法？怎么实现的？为什么用 Lua？

### 达标答法

> 用的是令牌桶算法，通过 Redis Lua 脚本保证原子性。
>
> **核心流程：** Redis 里用一个 Hash 存两个值——当前令牌数和上次补充时间。每次请求进来，Lua 脚本会先算时间差，按速率补充令牌，但不超过容量上限。然后看令牌够不够，够就扣减放行，不够就拒绝。整个过程在 Redis 侧一次执行完，不存在并发问题。
>
> **为什么用 Lua 而不是 Java 侧加锁？** 因为 Lua 脚本在 Redis 里是原子执行的，一次网络往返就搞定，不需要额外的分布式锁，性能更高。如果用 Java 实现，读令牌数、判断、扣减这几步之间会有并发窗口，要么加锁要么多次访问 Redis，都不如 Lua 干净。
>
> **两个值得主动提的设计细节：**
>
> 第一，时间戳不是让 Redis 自己取的，而是由 Java 侧把当前微秒时间作为参数传进 Lua，这样时间口径统一，不依赖 Redis 服务器时钟。
>
> 第二，限流器异常时默认放行，也就是 fail-open。为什么？因为限流是保护手段，不是核心业务。如果限流组件自己挂了，不应该连带把核心业务也打挂。日志里会记录异常，但请求照样放过去。

### 追问：令牌桶和漏桶有什么区别？

> **令牌桶**允许突发流量——桶里攒了令牌的话，短时间内可以消费多个，适合有突发峰值但整体要限速的场景。
>
> **漏桶**是严格匀速出水，不管进来多少请求，处理速率是固定的，适合需要严格平滑流量的场景。
>
> 我们选令牌桶是因为业务上会有短时突发（比如设备批量上报），需要允许一定的突发通过，而不是严格削平。

### 追问：这个限流用在什么场景？

> 主要用在两个地方：
>
> 第一，定时任务里控制对外部 API 的调用频率。任务注解上有个最大调用次数配置，每次发远程请求前会做原子计数，超了就中断任务，状态落成"调用次数超限"。
>
> 第二，业务接口层面的通用限流，防止上游系统突发大量请求把下游打崩。

### 关键数字

| 参数 | 示例值 | 含义 |
|---|---|---|
| capacity | 20 | 桶容量，最多攒 20 个令牌 |
| rate | 10 | 每秒补充 10 个令牌 |
| requested | 1 | 每次请求消耗 1 个令牌 |

## 十八、MQ 复杂场景——设备级熔断 + 消息过滤转发 + 保护下游（第一轮遗漏，补充）

> 基于真实代码：隔离开关位置信息消费者。这个场景同时涉及去重、设备级限流熔断、消息过滤转发、下游保护，非常适合面试展开。

### 面试官可能怎么问

> 你们消费者有没有遇到过某个设备疯狂上报消息、把系统打爆的情况？怎么处理的？

或者：

> 消费者消费完一条消息后，还要转发给另一个 Topic，这种链路你怎么设计的？

### 达标答法（完整链路）

> 有一个很典型的场景，是隔离开关的位置信息和变位信息消费。这个消费者做了四层处理：
>
> **第一层，设备级熔断检查。** 消息进来后先解析出设备 ID，去 Redis 查这个设备有没有被熔断。如果已经熔断了，直接跳过，不做任何处理。
>
> **第二层，消息去重。** 用消息里的 UUID 做 Redis 短时去重，3 分钟窗口，防止同一条消息重复消费。
>
> **第三层，设备日计数 + 自动熔断。** 去重通过后，用 Redis 原子自增给这个设备的日消息计数加 1。如果单设备单天超过 700 条，就触发熔断——往 Redis 写一个熔断标记，有效期 3 天。后续这个设备的所有消息在第一层就会被拦住。
>
> **第四层，消息过滤 + 异步转发。** 不是所有消息都需要转发，只有特定类型的消息才异步发到另一个 Topic 做进一步处理。过滤条件很轻量，直接用字符串匹配判断，不做完整 JSON 反序列化，减少性能开销。
>
> 最后才调 Service 层做真正的业务处理。

### 追问：为什么要做设备级熔断？

> 因为现场设备可能出现异常——比如某个隔离开关的传感器抖动，一天能上报几千条甚至上万条消息。如果不做限制，这一台设备就能把整个消费链路拖慢，影响其他正常设备的消息处理。
>
> 所以我做了两层保护：单日计数限制 + 超限后自动熔断 3 天。熔断期间这个设备的消息在最前面就被拦掉了，不会进入去重、计数和业务处理环节，保护了整条链路的吞吐。
>
> 另外熔断检查和计数异常时都是 fail-open，也就是 Redis 出问题了就当没熔断继续处理，不会因为保护机制本身的故障把正常消息也拦住。

### 追问：这个设计有什么可以优化的地方？

> 我觉得有几个点可以继续优化：
>
> **第一，异步转发失败没有补偿。** 现在转发失败只打了日志，消息就丢了。可以加一个失败落库机制，由后台补偿任务重新转发。
>
> **第二，JSON 解析重复了。** 消费者里解析了一次设备 ID，Service 层又会完整解析一次。可以在消费者层解析一次后把结果传给 Service，避免重复反序列化。
>
> **第三，异常处理太宽泛。** 现在所有异常都被 catch 吞掉了，等于处理失败也算消费成功，MQ 不会重投。可以区分一下：业务异常（比如数据格式不对）直接跳过；系统异常（比如数据库不可用）应该抛出去让 MQ 重试。
>
> **第四，熔断只有设备维度，没有全局维度。** 如果同时几百台设备都在大量上报，单设备没超限但总量已经很大了，可以加一个全局限流兜底。

### 追问：如果消息积压了怎么办？

> 先判断是消费者挂了还是消费太慢。
>
> **短期应急**：加消费者实例横向扩容，但不能超过 Queue 数，超了也没用。
>
> **如果是消费逻辑太重**：把消费者拆成两步——先快速落库或者写 Redis，返回消费成功；再由另一个任务异步做重逻辑处理。这样消费端永远是轻量的，不会因为下游慢而积压。
>
> **长期优化**：消费端批量处理、异步 IO、增加 Queue 数提高并行度。
