# 专题三：消息队列 + 下游保护

> 这条线的逻辑：MQ 幂等怎么做 → 复杂场景（设备熔断 + 消息配对） → 令牌桶限流 → Kafka 基础
> 面试官问 MQ 相关的任何问题，都能从这条线上找到入口。

---

## 入口问题："你们消费端幂等怎么做的？消费失败怎么处理？"

### 完整话术（直接背）

> 我做了两层幂等加上失败兜底。
>
> 第一层是 Redis 快速去重。消息进来后，拿消息里的唯一 ID 往 Redis 写一个处理中的标记，用 SET NX 加 3 分钟过期。如果标记已经存在，说明这条消息正在被处理或者刚处理完，直接跳过返回消费成功。处理成功后，我会把这个标记的过期时间延长到 10 分钟。为什么要延长？因为处理窗口只有 3 分钟，但成功标记需要保持更久，防止刚处理完标记就过期，MQ 重投进来又被当成新消息。
>
> 第二层是数据库业务幂等。Redis 去重只是快速拦截，真正的业务幂等在 service 层按业务唯一键做，比如设备 ID 加故障时间建联合唯一索引。insert 的时候如果冲突，说明已经处理过了。
>
> 这里有一个副作用控制的细节。落库是幂等的，但发短信通知不是。所以我只有在 insert 返回影响行数为 1 的时候——也就是本次真正新插入了数据——才触发异步短信。重试进来发现数据已存在，影响行数是 0，就不会重复发短信。原则就是先幂等写入，再根据写入结果决定是否触发副作用。
>
> 消费失败的处理：清掉 Redis 里的处理中标记，抛异常让 MQ 重新投递。但我设了最多重试 3 次，加上初始那次总共 4 次机会。超了就不再重试，避免一条坏消息把整个消费者卡死。

## 追问："有没有遇到过某个设备疯狂上报把系统打爆的情况？"

### 完整话术（设备级熔断）

> 遇到过，这也是我做设备级熔断的原因。
>
> 现场设备可能出现传感器抖动，比如某个隔离开关一天能上报几千条甚至上万条消息。如果不做限制，这一台设备就能把整个消费链路拖慢，影响其他正常设备。
>
> 我在消费者里做了四层处理。第一层是熔断检查，消息进来先解析出设备 ID，去 Redis 查这个设备有没有被熔断，有就直接跳过。第二层是 UUID 去重，3 分钟窗口防重复消费。第三层是设备日计数加自动熔断，用 Redis 原子自增给这个设备的日消息计数加 1，如果单设备单天超过 700 条，就往 Redis 写一个熔断标记，有效期 3 天。后续这个设备的消息在第一层就被拦住了。第四层是消息过滤加异步转发，只有特定类型的消息才异步发到另一个 Topic 做进一步处理。
>
> 这里有两个设计细节值得说。第一，熔断检查和计数的 Redis 操作如果异常，我做的是 fail-open，也就是 Redis 挂了就当没熔断继续处理。不能因为保护机制本身的故障把正常消息也拦住。第二，过滤条件用字符串匹配判断，不做完整 JSON 反序列化，尽量减少性能开销。

## 追问："你说还有消息配对？这是什么场景？"

### 完整话术（消息等待与配对）

> 隔离开关会上报两种信号——遥信 1 是位置信息，遥信 2 是变位信息。它们是同一个设备在差不多同一时间产生的两条独立消息，只有两条都到了并且配对成功，才能生成工单推送下游。
>
> 问题是两条消息到达时间不确定，可能间隔几百毫秒甚至几秒，不能假设它们同时到。
>
> 当前实现是这样的：每条消息先入库，然后异步检查配对。拿到遥信 1 后去数据库查同一设备、10 秒时间窗口内有没有遥信 2，反过来也一样。同一设备的配对逻辑用设备级别的内存锁串行执行，防止并发竞态。配对成功后保存一条 pair 记录，再触发工单推送。
>
> 这个实现能用，但有优化空间。当前等待配对用的是 Thread.sleep 轮询数据库，我有一个改进方案是用 Redis 做轻量级配对缓存：消息入库后把自己的 ID 和信号类型写到 Redis，key 是设备 ID 加时间窗口，过期时间 30 秒。写入的同时检查 Redis 里有没有另一条信号的记录，有就配对成功直接处理，没有就不等了，让后到的那条消息来触发配对。这样不需要 sleep，不需要轮询数据库，后到的消息天然触发配对逻辑。

## 追问："你简历上写了 Redis Lua 限流，具体怎么做的？"

### 完整话术（令牌桶——基于项目真实代码）

> 我在项目的公共模块里写了一个 RateLimiterUtil，用令牌桶算法加 Redis Lua 脚本实现的。
>
> 数据结构是 Redis 的 Hash，存两个字段：当前令牌数和上次补充时间。每次请求进来，Lua 脚本先算距离上次补充过了多长时间，按速率补充令牌但不超过容量上限，然后看令牌够不够，够就扣减放行返回 1，不够就拒绝返回 0。整个过程在 Redis 侧一次执行完，不存在并发问题。
>
> 为什么用 Lua 不用 Java 侧加锁？因为 Lua 在 Redis 里是原子执行的，一次网络往返搞定。如果用 Java，读令牌、判断、扣减之间有并发窗口，要么加分布式锁要么多次访问 Redis，都不如 Lua 干净。
>
> 这个工具类封装了几个场景化的方法。一个是 Feign 远程调用限流，容量 20、每秒补充 10 个令牌，防止某个定时任务在批量调用外部接口时把下游打崩。另一个是数据库操作限流，容量 100、每秒补充 50 个，用在批量写库的场景防止数据库连接池被打满。还有隔离开关相关的限流，有专门的管理接口可以查看当前令牌数、重置限流器。
>
> 三个值得说的设计细节。第一，时间戳用的是微秒精度，由 Java 侧传入 Lua 脚本，不依赖 Redis 服务器时钟，保证时间口径统一。第二，每个 key 都设了过期时间防止 Redis 内存泄漏，过期时间是补充间隔的 2 倍。第三，限流器异常时默认放行，也就是 fail-open。限流是保护手段不是核心业务，它自己挂了不应该连带把业务打挂。
>
> 令牌桶和漏桶的区别是：令牌桶允许突发，桶里攒了令牌可以一次消费多个；漏桶是严格匀速出水。选令牌桶是因为设备批量上报有突发峰值，需要允许一定突发通过。

## 追问："Kafka 了解吗？和 RocketMQ 怎么选？"

### 完整话术

> 了解，目标公司用到了 Kafka。
>
> Kafka 吞吐量高主要靠四个设计：第一是顺序写磁盘，消息追加写入 Partition 日志文件不做随机写；第二是 Page Cache，利用操作系统页缓存，不自己管内存；第三是零拷贝，消费者拉消息时用 sendfile 系统调用，数据从磁盘到网卡不经过用户态；第四是批量和压缩，Producer 攒 batch 再发，支持 snappy 和 lz4。
>
> 保证不丢消息从三个环节做：Producer 端设 acks=all 等 ISR 所有副本确认，加上 retries 重试；Broker 端副本因子至少 3，最小同步副本数设 2；Consumer 端关闭自动提交 offset，处理成功后手动提交。
>
> ISR 是和 Leader 保持同步的副本集合，Follower 如果 30 秒内没追上 Leader 就被踢出 ISR，追上了再重新加入。Leader 挂了从 ISR 里选新 Leader。
>
> 和 RocketMQ 怎么选？一句话：大数据场景用 Kafka，业务消息用 RocketMQ。Kafka 吞吐量百万级，和 Flink、Spark 生态集成好，适合日志采集、埋点、流计算。RocketMQ 吞吐量十万级但功能更丰富，原生支持延迟消息、事务消息、按 key 查询，适合订单、支付、告警这种对可靠性要求高的业务场景。我们电网项目选 RocketMQ 就是因为需要事务消息和消息查询，吞吐量十万级够用了。

## 追问："消息积压了怎么办？"

### 完整话术

> 先判断原因。消费者挂了就先恢复消费者，消费太慢就优化消费逻辑，流量突增就扩容。
>
> 短期应急的话，第一是加消费者实例横向扩容，但不能超过 Partition 数或 Queue 数，超了多余的消费者会空闲。第二是如果消费逻辑太重，把消费者拆成两步——先快速落库或者写 Redis 返回消费成功，再由另一个任务异步做重逻辑。这样消费端永远是轻量的。
>
> 长期优化就是消费端批量处理、异步 IO、增加 Queue 或 Partition 数提高并行度。
