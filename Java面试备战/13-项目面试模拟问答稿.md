# 项目面试模拟问答稿

> 这份文档不是知识点总结，而是给你直接拿来“张嘴练”的。
> 建议你边看边说，尽量把每道题控制在 1-2 分钟内，再根据追问往下展开。

---

## 一、怎么使用这份稿子

建议按下面节奏练：

1. 先只看题目，不看答案，自己说一遍
2. 卡壳了再看“建议答法”
3. 说顺之后，再练“继续追问怎么接”
4. 最后把每题压到 30 秒、1 分钟、2 分钟三个版本

答题通用结构：

- 先说结论
- 再说方案或链路
- 最后说效果、取舍或风险

## 二、模拟一：AI 岗 / RAG 项目主问答

### Q1：你先介绍一下这个 RAG 项目

**建议答法：**

这个项目本质上不是通用问答，而是电网预警规则的意图解析引擎。核心目标是把业务人员写的自然语言规则，转成结构化 Business DSL，后面再交给执行层映射成真实查询和告警动作。

我负责的是后端主链路，包括文本标准化、Slot 抽取、混合检索、DSL 生成、多轮澄清、DSL 校验和回译确认。后端主编排入口是 `PhaseAOrchestrator`。技术栈是 Python + FastAPI + PostgreSQL(pgvector)，整个系统已经做成了独立部署的 AI 后端服务，不是单次调模型的 demo。

**继续追问可以接：**

- 为什么不是普通 ChatBot
- 为什么选 Python / FastAPI
- 为什么要做多轮澄清

### Q2：这个项目最核心的技术挑战是什么？

**建议答法：**

最核心的挑战不是“调 LLM”，而是让自然语言规则稳定落成可执行 DSL。因为电网场景里别名很多、指标和实体之间有强约束，而且用户一句话里经常会同时出现设备范围、时间窗口、阈值和动作意图。

所以我没有把识别全扔给大模型，而是拆成四层：

1. 正则抽结构化值，比如电压、次数、时间窗口、阈值
2. 别名词典扫实体、场景、动作提示词
3. 消歧器处理多候选实体
4. 最后再让 `LLMSlotAssembler` 做语义汇总

这样做的好处是准确率更稳、延迟更低，而且 LLM 超时还能 fallback。

### Q3：为什么要用 pgvector + BM25 混合检索？

**建议答法：**

因为电网领域既有强关键词，又有强语义相似。像 `SF6`、`GIS`、`CT` 这种词，BM25 很有优势；但像“超过额定值”和“大于标准值”这种表达差异，向量检索更有优势。

所以我用 BM25 和 pgvector 两路召回，再通过 `HybridRanker` 做 RRF 融合。这样两路都排前面的范文会优先被拿来做 few-shot，Top-3 命中率比单路更稳。

**继续追问可以接：**

- 为什么不用纯向量
- 为什么 RRF 的 k 取 60
- 为什么不用 Milvus

### Q4：这个系统怎么保证不是一改就回退？

**建议答法：**

我把它做成了有明确回归口径的后端系统。当前项目里不是随手测几个 case，而是有 `42` 条 existing cases 加 `26` 条 markdown cases 汇成的 `68` 条基线，再配合 `backend/tests/` 下 `49` 个 pytest 文件做单测、集成和 smoke 兜底。

日常会跑 `make slot-regression-bundle`、`make slot-dsl-smoke` 这些命令。这样每次改动之后，我能明确知道是“修复了一个 bad case”还是“顺手把别的分支搞坏了”。

### Q5：多轮澄清是怎么做的？

**建议答法：**

多轮澄清靠的是 `SessionContext`。里面会保存 `pending_clarification`、`negative_examples`、`negative_constraints`、`deprioritized_corpus_ids`、`retry_count` 这些字段。

实际流程是这样的：

- 如果当前输入信息不完整，比如实体不明确、指标缺失、能力不支持，就先返回澄清问题
- 用户下一轮回复时，先走 Step 0 消费澄清，而不是重新从零识别
- 如果用户拒绝当前 DSL，就把这次结果放进 `negative_examples`，同时把相关范文 ID 加入 `deprioritized_corpus_ids`

所以这个系统不是“单轮 prompt 调用”，而是有状态的。

### Q6：你怎么保证 DSL 一定合法？

**建议答法：**

我做了两层控制。

第一层是生成前的 Slot 校验，确保实体、场景、指标组合是成立的。
第二层是生成后的 `DSLValidator`，会校验：

- 顶层字段是否合法
- entity 和 target_entities 是否冲突
- metrics 是否存在、是否适用于当前实体和场景
- rules 里的 metric 引用是否真实存在
- scope_filters 是否落在白名单字段里

如果只是命名不标准，会尝试自动修复；如果结构不合法，就直接打回。

### Q7：如果面试官说“这听起来还是挺像 prompt 工程”，你怎么回？

**建议答法：**

我会强调它的工程属性：

- 编排链是代码驱动，不是一个大 prompt 包打天下
- 有明确的会话状态管理
- 有健康检查，探活 DB、LLM、Embedding 三个依赖
- 有回归体系和固定基线
- 有清晰的降级策略，比如 LLM 超时 fallback、向量检索降级到 BM25

所以这个项目更接近“AI 后端系统”，不是 demo。

### Q8：这个项目里最难的 bad case 你会怎么讲？

**建议答法：**

我会讲“别名多 + 条件嵌套”的 case，比如：

> 110kV及以上断路器 SF6 气体泄漏年超过 50mL 且运行年限超过 15 年的设备生成预警

这个输入里同时有：

- 电压范围
- 实体
- 场景
- 两个条件

最难的是“运行年限超过 15 年”容易被模型错认成时间窗口，而不是设备属性过滤。

我的处理方式是：

1. 正则先抽范围和数值
2. 别名扫描锁定实体和场景
3. LLM 汇总时明确区分“指标阈值”和“设备属性过滤”
4. DSL 校验阶段发现位置不对就纠正

### Q9：如果让你继续优化这个 RAG 项目，你会怎么做？

**建议答法：**

我会优先做四件事：

1. 引入 rerank 模型，替代当前纯 RRF 排序
2. 让 DSL 生成走流式输出，提升用户体感
3. 强化领域词典和 tokenizer，继续优化 BM25 召回
4. 把回归结果做成更清晰的自动化看板

## 三、模拟二：Java 岗 / 变电3.0 项目主问答

### Q1：你先介绍一下变电 3.0 项目

**建议答法：**

这是一个变电站生产管控平台，核心是把遥测、气象、试验、巡视、遥信等多源数据做同步、清洗、统计、规则判断和工单生成。技术上不是单一服务，而是一个 7 模块 Maven 聚合工程，技术栈是 Java 8 + Spring Boot 2.3.12 + Spring Cloud Alibaba + Redis + RocketMQ + MyBatis。

我负责过的重点是任务治理、异步消息处理、OpenFeign 失败补偿，以及部分告警链路优化。

### Q2：你们为什么要自己做定时任务 Starter？

**建议答法：**

因为我们需要的不是“能定时执行”这么简单，而是统一治理。

电网内网环境下，如果为了定时任务单独引 XXL-Job，还要部署调度中心、运维审批、额外接入，成本比较高。而我们的核心诉求其实是：

- 自动扫描注册任务
- 动态启停
- 记录执行日志
- 限制任务外部调用次数
- 支持手动触发和停止
- 支持异常通知和启动恢复

这些更适合做成业务内聚的 Starter。

### Q3：这套 Starter 你能讲到源码级吗？

**建议答法：**

可以，但我不会背类名，我会按治理链路讲：

1. 启动时自动发现任务并完成注册
2. 运行时按数据库配置动态启停
3. 执行过程中统一做日志、锁、计数和异常通知
4. 平台还支持手动触发、人工停止和启动恢复

这样既能体现我看过源码，也不会把回答讲成“源码目录介绍”。

### Q4：任务平台怎么防止一个任务在多个实例上重复跑？

**建议答法：**

它做了两层防护：

- 单实例内防止重复执行
- 多实例之间再加一层 Redis 分布式锁

这把调度锁默认有 120 秒 TTL，每 30 秒续租一次，释放时也是原子操作，不是那种简单先读再删的实现。

所以这一块我会说，我们在调度平台里用的是更严谨的锁方案。

### Q5：手动执行和停止任务怎么做的？

**建议答法：**

平台支持后台手动执行和停止任务。

手动执行时不会占用当前请求线程，而是异步触发，并且会绕过自动任务的禁用检查，方便排障和补数。停止任务时，会先定位当前运行记录，再中断真实执行线程，而不是只改一个数据库状态。

### Q6：你们怎么监控任务执行过程？

**建议答法：**

执行监控做得比较完整：

- 执行前就创建运行日志
- 执行中会统计调用次数和耗时
- 异常场景会触发通知
- 停止、停机、调用超限这些场景会落成不同状态

状态不是简单成功失败，而是区分：

- `SUCCESS`
- `ERROR`
- `HAND_SHUTTING_DOWN`
- `SYSTEM_SHUTTING_DOWN`
- `REQUEST_LIMIT_EXCEEDED`

### Q7：你们怎么控制任务不要把外部 API 打爆？

**建议答法：**

这个是 Starter 里比较有亮点的一点。每次发远程请求前，平台都会先做原子计数。

如果超过任务配置里的 `maxCallLimit`，就会中断任务，并把最终状态落成 `REQUEST_LIMIT_EXCEEDED`。同时还会保留 URL 维度的调用明细，后面一起回填到日志。

### Q8：OpenFeign 失败补偿机制是怎么做的？

**建议答法：**

我做的是 AOP 拦截失败调用并落库，后面再由后台补偿任务重放。

重点不是“记一条失败日志”，而是把服务标识、方法名、原始参数、参数类型、重试上限、优先级和错误信息完整保留下来。这样后面重试时能按原始上下文重放，不需要人工猜怎么补。

### Q9：RocketMQ 这块你怎么讲才像生产经验？

**建议答法：**

我会拿 `FaultTripRecordConsumer` 举例，不会只泛泛说“用了 MQ”。

它做了两层幂等：

- Consumer 侧基于 Redis 做短时间去重，处理窗口 3 分钟，成功标记 10 分钟
- 真正的业务幂等在 service 层按业务键兜底

另外它的重试边界也很清晰：

- 失败就抛异常给 RocketMQ 重试
- `reconsumeTimes >= 3` 时停止继续抛异常
- 等于总共最多 4 次机会

### Q10：这个项目里你会主动承认什么不足？

**建议答法：**

我会主动说业务仓库里的 `RedisLockUtil` 还不是最严谨的实现，因为它的解锁是先读后删，不是原子 compare-and-delete。

但我也会顺势说明：

- 我知道这个实现的边界
- 我在自研 `scheduledTask` 调度锁里已经用了 Lua 原子释放和自动续租
- 如果要进一步统一，我会推动业务侧锁也升级到同样标准或者直接切 Redisson

这种回答会让人觉得你不是回避问题，而是知道怎么把系统往前推。

## 四、模拟三：混合压力追问

### Q1：你这两个项目里，哪一个更能体现你的架构能力？

**建议答法：**

如果投 AI 岗，我会选 RAG 项目，因为它体现的是“把 LLM 能力做成可回归、可降级、可多轮交互的后端系统”。

如果投 Java 后端岗，我会选变电 3.0，因为它体现的是“任务治理、消息可靠性、失败补偿、规则引擎和多模块协同”。

### Q2：你做过最像平台能力的东西是什么？

**建议答法：**

我会选 `scheduledTask` 这个 Starter。因为它不是一个业务功能点，而是一套可复用能力，包括自动装配、任务注册、动态调度、执行监控、调用限流、手动控制和启动恢复。

### Q3：你做过最难排查的问题是什么？

**建议答法：**

可以从两个方向选一个讲：

- RAG：某些复杂规则被误识别成错误实体或错误约束，最后靠 bad case 回归和状态化澄清体系收敛
- 变电：任务中断、应用重启、日志残留 `RUNNING` 状态不一致，最后通过 `TaskStateCleanupRunner` 做启动恢复收敛

### Q4：如果面试官质疑“这些是不是你自己一个人做的”？

**建议答法：**

不要抢功，但要讲清自己主导的部分：

- 哪些是你做方案设计
- 哪些是你做核心链路实现
- 哪些是你推动落地和优化
- 哪些是团队协作完成

最稳的说法是：

> 这个项目是团队一起做的，但我主导的是某某能力的设计和核心实现，尤其是某某类、某某链路、某某回归机制是我重点负责的。

## 五、压缩版速答

### 5.1 RAG 项目 30 秒版

这是一个把电网自然语言规则转成 Business DSL 的 AI 后端系统。我负责后端主链路，包括 Slot 抽取、混合检索、DSL 生成、多轮澄清和校验。核心不是单次 prompt 调用，而是 `PhaseAOrchestrator` 驱动的完整链路，加上 `68` 条基线回归和多依赖健康检查。

### 5.2 变电 3.0 30 秒版

这是一个 7 模块的变电站生产管控平台，我重点负责过任务治理、RocketMQ 消息处理和 Feign 失败补偿。比较有代表性的能力是自研 `scheduledTask` Starter，它把自动装配、启动扫描、动态调度、执行日志、调用限流、手动控制和启动恢复做成了一套可复用平台。

### 5.3 RAG 项目 1 分钟版

这个项目做的是电网预警规则意图解析，不是普通问答。我们要把业务人员写的自然语言规则转成结构化 Business DSL，所以难点在于实体、场景、指标、阈值、时间窗口和动作的稳定识别。我用的是正则、别名词典、消歧器和 LLM 汇总的分层识别方式，再通过 BM25 + pgvector 混合检索选 few-shot 范文，最后用 DSL 校验和回译确认兜底。整个链路由 `PhaseAOrchestrator` 编排，并且有固定回归基线，不是凭感觉调 prompt。

### 5.4 变电 3.0 1 分钟版

这个项目是变电站生产管控平台，数据来源多、任务多、消息多、规则复杂，所以我重点做的是治理能力。我参与的比较核心的是自研 `scheduledTask` Starter：启动后自动扫描任务注解并入库，运行中每分钟刷新动态调度，执行时通过切面统一记录日志、限制外部调用次数、支持手动执行和停止，同时用 Redis 分布式锁避免多实例重复执行。另外微服务之间的 Feign 调用失败也不是简单报错，而是落库后由定时任务补偿重试。

## 六、最后的练习建议

你可以按这个顺序练：

1. 先把 `5.1` 到 `5.4` 背熟
2. 再练第二部分和第三部分的主问答
3. 最后让别人随机抽第四部分的压力追问

如果你发现某题一说就散，说明不是不会，而是还没压缩成“面试话术”。这种题优先多练。
