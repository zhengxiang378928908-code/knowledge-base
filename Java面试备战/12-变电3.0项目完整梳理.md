## 一、30 秒项目介绍（开场白）

> 这是一个变电站生产管控平台，核心职责是把遥测、气象、巡视、试验报告、遥信等多源数据做同步、清洗、统计、规则判断和工单生成。我在里面负责过任务治理、异步消息处理、远程调用补偿和部分告警工单链路优化。技术栈是 Java 8 + Spring Boot 2.3.12 + Spring Cloud Alibaba + MyBatis + Redis + RocketMQ，整个仓库是 7 模块 Maven 聚合工程。

## 三、自研任务治理平台怎么讲

### 3.1 为什么不用 XXL-Job？

**参考回答：**

主要有三个原因：

1. **内网部署成本高**：电网环境新增一套调度中心会带来额外部署和审批成本
2. **我们更需要治理能力，而不是复杂编排能力**：核心诉求是任务统一注册、动态启停、执行监控和限流
3. **需要和业务元数据深度绑定**：比如任务归属人、调用上限、生产环境保护、人工触发控制，这些都更贴近业务定制

### 3.2 仓库里有什么证据说明这套任务平台真实在用？

**参考回答：**

有几个非常直观的证据：

- `scheduledTask` 的 `PmsAutoConfiguration` 里直接做了 `@ComponentScan("com.ysh.pms")` 和 `@MapperScan("com.ysh.pms.mapper")`
- `PmsRedisAutoConfiguration` 在 Spring 官方 Redis 自动配置之后提供统一的 `RedisTemplate<String, Object>`
- `ScheduledTaskScanner` 实现了 `CommandLineRunner`，应用启动后会遍历 Spring Bean，借助 `AopUtils.getTargetClass` 和 `AnnotationUtils.findAnnotation` 扫描所有 `@MonitoredScheduledTask`
- `DynamicTaskConfigurer` 实现 `SchedulingConfigurer`，每分钟刷新一次数据库里的启用任务列表，并用 `ConcurrentHashMap<String, ScheduledFuture<?>>` 持有当前活跃任务
- `ScheduledTaskInvoker` 负责真正反射调用任务方法，而且只允许两种签名：无参，或者单个 `ScheduledTaskContext` 参数
- 很多任务方法还叠加了 `@ProdOnly`，说明任务平台不仅管调度，还管环境边界
- `TelemetryDataTask` 这样的任务，把 `taskGroup`、`taskName`、`cron`、`functionOwner`、`maxCallLimit` 都显式写在注解里，治理维度很完整

### 3.3 Starter 的源码级运行链

面试里我建议你直接按这条链路讲：

1. `spring.factories` 自动装配 `PmsAutoConfiguration` 和 `PmsRedisAutoConfiguration`
2. `ScheduledTaskScanner` 在应用启动完成后做一次全量扫描，把注解里的任务元数据同步到 `scheduled_task_config`
3. `YshScheduledTaskServiceImpl.syncTaskFromAnnotation()` 会补齐默认值，比如 `is_enabled=1`、cron 描述、负责人、应用名；遇到唯一索引冲突会捕获 `DuplicateKeyException` 并复用已有记录
4. `DynamicTaskConfigurer` 每分钟拉一次当前应用的启用任务，新增就注册、禁用就 `future.cancel(true)` 停掉
5. 真正执行时不直接反射字符串，而是走 `ScheduledTaskInvoker`：
   - 先按类名找到 Spring Bean
   - 再找到兼容方法
   - 如果存在多个兼容重载直接报错，避免误调
6. 任务执行过程再交给 `ScheduledTaskAspect` 做监控、日志、限流、异常通知和停机收尾

这套说法的价值是：你不是只会说“我们做了任务平台”，而是能把 Starter 的完整生命周期讲清楚。

### 3.4 用一个具体任务说明你的设计思路

`TelemetryDataTask` 很适合拿来讲：

- `@ProdOnly("遥测数据获取仅在生产环境执行")`
- `@MonitoredScheduledTask(taskGroup = "遥测数据定时任务", cron = "0 25 */1 * * ?", functionOwner = "韩岩崎", maxCallLimit = 10000)`
- 每小时 25 分执行一次，但业务上拉的是最近 30 分钟窗口的数据
- 设备 ID 分批处理，`BATCH_SIZE = 100`
- 数据写库再分批，`INSERT_BATCH_SIZE = 1000`
- 子任务通过 `Future<?>` 收集，主线程统一 `future.get()`，中断时显式恢复中断状态

**这段回答的价值在于：**

- 你可以从“有任务平台”讲到“平台怎么承载一个高耗时、高吞吐任务”
- 能自然引出并发控制、批处理、优雅中断、环境保护这些后端工程点

### 3.5 执行监控和优雅停止是怎么落地的

这部分现在有源码可以讲得非常具体：

- `ScheduledTaskAspect` 在任务执行前会先查配置，自动执行时如果任务被禁用就直接跳过
- 手动执行场景通过 `ManualExecutionContext.IS_MANUAL_EXECUTION` 这个 `ThreadLocal` 绕过禁用检查
- 同一实例内用 `TaskExecutionRegistry.RUNNING_TASKS` 防止重复执行
- 跨实例用 `ScheduledTaskLockManager` 抢 Redis 锁，锁 key 是 `scheduled_task:lock:{application}:{taskId}`
- 锁默认 TTL 是 120 秒，每 30 秒续租一次
- 续租和释放都用了 Lua compare-and-update / compare-and-delete，而不是简单 `GET` + `DEL`
- 执行时会先插一条 `RUNNING` 日志，再把调用次数 Redis key 和 URL 调用明细 Hash key 建起来
- 任务方法如果声明了 `ScheduledTaskContext` 参数，切面会把上下文对象自动注入进去

**状态流转也值得背：**

- 正常完成：`SUCCESS`
- 业务异常：`ERROR`
- 人工停止：`HAND_SHUTTING_DOWN`
- 应用关闭时被中断：先保留 `RUNNING`，等下次启动由清理程序改成 `SYSTEM_SHUTTING_DOWN`
- 如果任务因为远程调用次数超限被打断，还会落成 `REQUEST_LIMIT_EXCEEDED`

### 3.6 手动执行和停止任务怎么做

`TaskControlController` 暴露的是内部控制接口：

- `POST /{application}/tasks/{taskId}/execute`
- `POST /{application}/tasks/{taskId}/stop`

Service 层的做法也很适合拿来讲：

- `manualExecute()` 不会占当前请求线程，而是扔到 `manualTaskExecutor` 这个专用线程池里异步执行
- 执行前把 `ManualExecutionContext` 置为 `true`，执行后再清理
- `stopTask()` 先查最新 `RUNNING` 日志，再从 `TaskExecutionRegistry` 里拿到真实线程，最后调用 `thread.interrupt()`

这说明你的平台不仅能“自动跑”，还能“远程管”。

## 四、Feign 调用失败补偿怎么讲

### 4.1 核心设计

这是一个比较有辨识度的设计点，核心是 **AOP 拦截失败调用并落库，后续再由定时任务重放**。

链路如下：

1. `@FallbackToDatabase` 标记需要补偿的 Feign 方法
2. `FeignFallbackAspect` 环绕这些调用
3. Feign 调用失败时，把调用上下文落到 `FeignFallbackTask`
4. `FeignFallbackRetryTask` 每分钟扫描失败记录并重试

### 4.2 真正落库了哪些信息？

`FeignFallbackAspect` 里保存的信息比较完整：

- `beanName`：从 `@FeignClient` 里解析服务名
- `methodName`
- `methodParams`：参数序列化后的 JSON
- `paramTypes`
- `description`
- `status`
- `retryCount`
- `maxRetryCount`
- `priority`
- `errorMessage`

**面试时可以这样回答：**

> 我们不是简单记一条“某某接口失败了”的日志，而是把“哪个服务、哪个方法、原始参数、参数类型、优先级、最大重试次数、错误信息”完整落库，这样后续重试任务才能真正重放原调用，而不是人工猜怎么补。

### 4.3 仓库里哪里能看出它不是摆设？

- `SubstationFeignClient` 上有多处 `@FallbackToDatabase`
- `ConfigOrderClient`、`StrategyAnalysisClient` 也用了同样注解
- `FeignFallbackRetryTask` 的 cron 是 `0 * * * * ?`，也就是每分钟扫一次失败任务

## 五、RocketMQ 的可靠性和幂等怎么讲

### 5.1 当前项目里消息规模怎么描述更稳？

从当前仓库里能直接搜到 **30 个生效的 `@RocketMQMessageListener`**，主要分布在事件调度服务和数据处理服务里，覆盖断路器、GIS、电压异常、故障跳闸、量测同步等场景。

这样讲的好处是：

- 有具体数字支撑
- 不会说成“我们用了 MQ”这种泛泛描述

### 5.2 典型消费者怎么处理重复消费？

最值得讲的是 `FaultTripRecordConsumer`：

- 收到消息先从消息体里提取 UUID
- 调 `messageDedupUtil.tryAcquireByMessage(...)` 做 3 分钟处理窗口去重
- 处理成功后再打一个 10 分钟成功标记
- 如果业务失败，会清理 processing key 并抛异常触发 RocketMQ 重试
- 当 `reconsumeTimes >= 3` 时停止继续抛异常，也就是总共最多 4 次机会

**要主动补的一句：**

Consumer 这里只是拦“相同消息的短时重复”，真正的业务幂等还在 service 层按业务键做。代码注释里也明确写了这一点。

### 5.3 这段设计为什么值得拿来讲？

因为它把消息可靠性拆成了两层：

- 第一层：Consumer 用 Redis 处理“短时间内的快速重复投递”
- 第二层：业务 service 用业务键兜住“同一业务事件反复到达”

这比单纯说“我用 Redis 去重”更像真实生产方案。

## 六、Redis 在这个项目里不只是缓存

### 6.1 Lua 限流怎么讲

`RateLimiterUtil` 用的是 Redis Lua 令牌桶：

- Redis Hash 里保存 `tokens` 和 `last_refill`
- Java 把当前时间按“微秒”传给 Lua
- Lua 内部计算补充令牌数
- 够就扣减，不够就拒绝
- 整个过程在 Redis 侧原子执行

代码里还有两个很适合加分的点：

- 提供了 `tryAcquireForRemoteCall()` 和 `tryAcquireForDatabase()` 两套常用预设
- 限流器异常时默认放行，也就是 **fail-open**，避免核心业务被限流组件拖垮

另外自研 `scheduledTask` Starter 里还有一层和任务治理强相关的“调用次数控制”：

- `ExternalApiCallService` 每次远程调用前都会从 `ScheduledTaskAspect.taskContextHolder` 拿上下文
- 然后对 `redisCallCountKey` 做原子自增
- 一旦超过任务注解里的 `maxCallLimit`，就抛 `InterruptedException`
- 同时把每个 URL 的调用次数记录到 `scheduled_task:url_details:{logId}` 这个 Redis Hash

这意味着 `maxCallLimit` 不是写着好看，而是真的能约束高频外部调用。

### 6.2 分布式锁怎么讲更诚实也更加分？

当前 `RedisLockUtil` 的实现是：

- `tryLock`：`setIfAbsent(key, token, expire, TimeUnit.MILLISECONDS)`
- `lock`：每 50ms 轮询，超时放弃
- token 用 UUID

但这里我建议你在面试时 **主动指出一个真实风险**：

`unlock(key, token)` 现在是先 `GET` 再 `DELETE`，不是 Lua compare-and-delete 的原子解锁。

所以你可以这样回答：

> 当前实现已经具备 token 校验和超时释放这些基本能力，但如果我要把它升级成更严谨的生产级分布式锁，我会把解锁改成 Lua 原子脚本，或者直接换成 Redisson，再结合业务幂等一起兜底。

这类“我知道当前实现到哪一步，也知道下一步怎么改”的回答，面试里很加分。

这里你还可以顺手补一句对比：

> 业务仓库里的 `RedisLockUtil` 偏基础互斥；但我自研的 `scheduledTask` 调度锁已经做了 Lua 原子释放和自动续租，说明我知道不同场景对锁可靠性的要求是不一样的。

## 八、启动恢复和异常收尾怎么讲

这部分很容易被忽略，但其实很加分。

`TaskStateCleanupRunner` 也是 `CommandLineRunner`，它会在应用启动时：

1. 找出当前应用下所有遗留的 `RUNNING` 日志
2. 把这些日志改成 `SYSTEM_SHUTTING_DOWN`
3. 尝试从 Redis 恢复调用次数和 URL 调用明细
4. 同步更新 `scheduled_task_config` 里的最近运行状态、耗时、调用次数、日/月累计次数

**面试时你可以这样说：**

> 我们考虑过“任务执行中应用突然重启”这种情况，所以不是简单依赖 finally 收尾，而是在下次启动时再跑一遍状态修复程序，保证日志表、配置表和 Redis 临时状态最终能收敛一致。

## 九、面试高频追问 & 参考答案

### Q1：这个项目里你最能体现架构能力的点是什么？

> 我一般会讲三件事：任务治理、Feign 失败补偿、消息可靠性。因为这三块都不是单点 CRUD，而是要考虑可观测、幂等、重试、限流、优雅关闭和多模块协同。

### Q2：为什么要做自研任务治理，不直接全部写死在 `@Scheduled`？

> 因为任务一多，真正难的不是“能定时执行”，而是任务元数据治理、动态启停、调用频次限制、环境隔离、责任人归属和执行留痕。这些能力如果没有统一平台，后面维护成本会非常高。

### Q3：Feign 调用失败为什么不只靠 Feign 自己重试？

> Feign 自带重试更适合短暂网络抖动，但不适合处理“服务暂时不可用、参数需要留档、后续人工可追溯”的场景。落库补偿的价值在于可以保留原始调用上下文，并让补偿链路独立运行。

### Q4：RocketMQ 去重为什么还要 service 层幂等？

> 因为 Consumer 侧去重更像“短时间内的快速拦截”，它基于消息内容和 TTL，不等于完整的业务语义幂等。真正稳妥的是 service 层再按业务唯一键做一次兜底。

### Q5：这个项目里你会主动承认什么不足？

> 我会主动提 Redis 锁解锁还不是原子 compare-and-delete，这说明我知道当前实现的边界；同时我也会给出明确改进方案，比如 Lua 原子脚本或 Redisson。

### Q6：如果让你继续优化，你会做什么？

> 我会优先做四件事：
> 1. 分布式锁改成 Lua 原子解锁或 Redisson
> 2. Feign 多次补偿失败后增加告警通知，而不是只留在库里
> 3. 对任务平台补充分片和更细粒度的并发控制
> 4. 把消费者幂等、去重、重试策略做成统一组件，减少重复实现

## 十、考前最值得背的 6 句话

1. 这是一个 7 模块的变电站生产管控平台，不是单体 CRUD，核心是任务、消息、规则和工单联动。
2. 自研任务治理 Starter 是独立项目，核心链路是自动装配、启动扫描、动态刷新、反射调用、AOP 监控和启动恢复。
3. Feign 失败补偿不是简单重试，而是 AOP 落库原始调用上下文，再由定时任务重放。
4. RocketMQ 可靠性做了两层：Consumer 短时去重 + service 业务幂等。
5. 业务仓库里的基础 Redis 锁我知道边界，但调度平台里的锁已经做了 Lua 原子释放和自动续租。
6. 任务平台的价值不在“能定时执行”，而在统一元数据、动态启停、生产保护、调用限流、异常通知和最终一致性修复。
