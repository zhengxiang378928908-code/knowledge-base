## 一、30 秒项目介绍（开场白）

> 这是一个变电站生产管控平台，核心职责是把遥测、气象、巡视、试验报告、遥信等多源数据做同步、清洗、统计、规则判断和工单生成。我在里面负责过任务治理、异步消息处理、远程调用补偿和部分告警工单链路优化。技术栈是 Java 8 + Spring Boot 2.3.12 + Spring Cloud Alibaba + MyBatis + Redis + RocketMQ，整个仓库是 7 模块 Maven 聚合工程。

## 三、自研任务治理平台怎么讲

### 3.1 为什么不用 XXL-Job？

**参考回答：**

主要有三个原因：

1. **内网部署成本高**：电网环境新增一套调度中心会带来额外部署和审批成本
2. **我们更需要治理能力，而不是复杂编排能力**：核心诉求是任务统一注册、动态启停、执行监控和限流
3. **需要和业务元数据深度绑定**：比如任务归属人、调用上限、生产环境保护、人工触发控制，这些都更贴近业务定制

### 3.2 仓库里有什么证据说明这套任务平台真实在用？

**参考回答：**

有几个非常直观的证据：

- 任务平台有独立 starter 源码，不是散落在业务代码里的几段工具类
- 启动后会自动发现并同步任务元数据，说明任务是被统一管理的
- 运行期支持按数据库配置动态启停，不需要改代码重发版
- 执行过程中会统一记录日志、统计调用次数、做异常通知和停机收尾
- 很多任务方法还叠加了 `@ProdOnly`，说明任务平台不仅管调度，还管环境边界
- `TelemetryDataTask` 这样的任务，把 `taskGroup`、`taskName`、`cron`、`functionOwner`、`maxCallLimit` 都显式写在注解里，治理维度很完整

### 3.3 面试里怎么概括这套任务平台

面试里我建议直接按“能力链路”讲，不用背源码清单：

1. 启动时自动发现任务并完成元数据注册
2. 运行时按数据库配置动态启停任务
3. 执行中统一做日志、限流、防重复和异常通知
4. 人工可以触发和停止任务，停机和重启时还能把状态收敛回来

这套说法的价值是：你不是只会说“我们做了任务平台”，而是能把平台到底解决了哪些治理问题讲清楚。

### 3.4 用一个具体任务说明你的设计思路

`TelemetryDataTask` 很适合拿来讲：

- `@ProdOnly("遥测数据获取仅在生产环境执行")`
- `@MonitoredScheduledTask(taskGroup = "遥测数据定时任务", cron = "0 25 */1 * * ?", functionOwner = "韩岩崎", maxCallLimit = 10000)`
- 每小时 25 分执行一次，但业务上拉的是最近 30 分钟窗口的数据
- 设备 ID 分批处理，`BATCH_SIZE = 100`
- 数据写库再分批，`INSERT_BATCH_SIZE = 1000`
- 子任务通过 `Future<?>` 收集，主线程统一 `future.get()`，中断时显式恢复中断状态

**这段回答的价值在于：**

- 你可以从“有任务平台”讲到“平台怎么承载一个高耗时、高吞吐任务”
- 能自然引出并发控制、批处理、优雅中断、环境保护这些后端工程点

### 3.5 执行监控和优雅停止是怎么落地的

这部分我建议讲成“治理能力”，不要讲成“类图背诵”：

- 执行前会先检查启停状态，自动执行和人工执行走不同控制逻辑
- 单实例内会做防重复执行，多实例间再加一层 Redis 分布式锁
- 调度锁有超时和续租机制，释放也是原子操作，比普通业务锁更严谨
- 执行时会统一落运行日志、统计调用次数，并在异常或中断时收敛成明确状态

**状态流转也值得背：**

- 正常完成：`SUCCESS`
- 业务异常：`ERROR`
- 人工停止：`HAND_SHUTTING_DOWN`
- 应用关闭时被中断：先保留 `RUNNING`，等下次启动由清理程序改成 `SYSTEM_SHUTTING_DOWN`
- 如果任务因为远程调用次数超限被打断，还会落成 `REQUEST_LIMIT_EXCEEDED`

### 3.6 手动执行和停止任务怎么做

这块面试里可以直接说成“平台具备后台控制能力”：

- 手动执行不会占住当前请求线程，而是异步触发，避免控制动作把业务线程拖死
- 手动执行会绕过自动任务的禁用判断，保证排障和补数场景可用
- 停止任务时不是简单改状态，而是会定位当前运行记录并中断真实执行线程

这说明你的平台不仅能“自动跑”，还能“被运维和业务安全地接管”。

## 四、Feign 调用失败补偿怎么讲

### 4.1 核心设计

这是一个比较有辨识度的设计点，核心是 **AOP 拦截失败调用并落库，后续再由后台补偿任务重放**。

你可以直接讲成三步：

1. 用注解标记需要补偿的远程调用
2. 调用失败时把原始上下文完整落库
3. 后台补偿任务按原始参数重试，而不是人工补单

### 4.2 真正落库了哪些信息？

落库的信息比较完整：

- 服务标识和方法名
- 原始参数和参数类型
- 重试次数上限、优先级和错误信息
- 当前补偿状态

**面试时可以这样回答：**

> 我们不是简单记一条“某某接口失败了”的日志，而是把“哪个服务、哪个方法、原始参数、参数类型、优先级、最大重试次数、错误信息”完整落库，这样后续重试任务才能真正重放原调用，而不是人工猜怎么补。

### 4.3 仓库里哪里能看出它不是摆设？

- 多个核心 Feign 调用都接入了同一套补偿机制
- 后台会周期性扫描失败记录并重试，不依赖人工兜底

## 五、RocketMQ 的可靠性和幂等怎么讲

### 5.1 当前项目里消息规模怎么描述更稳？

从当前仓库里能直接搜到 **30 个生效的 `@RocketMQMessageListener`**，主要分布在事件调度服务和数据处理服务里，覆盖断路器、GIS、电压异常、故障跳闸、量测同步等场景。

这样讲的好处是：

- 有具体数字支撑
- 不会说成“我们用了 MQ”这种泛泛描述

### 5.2 典型消费者怎么处理重复消费？

最值得讲的是 `FaultTripRecordConsumer`：

- 收到消息先从消息体里提取 UUID
- 调 `messageDedupUtil.tryAcquireByMessage(...)` 做 3 分钟处理窗口去重
- 处理成功后再打一个 10 分钟成功标记
- 如果业务失败，会清理 processing key 并抛异常触发 RocketMQ 重试
- 当 `reconsumeTimes >= 3` 时停止继续抛异常，也就是总共最多 4 次机会

**要主动补的一句：**

Consumer 这里只是拦“相同消息的短时重复”，真正的业务幂等还在 service 层按业务键做。代码注释里也明确写了这一点。

### 5.3 这段设计为什么值得拿来讲？

因为它把消息可靠性拆成了两层：

- 第一层：Consumer 用 Redis 处理“短时间内的快速重复投递”
- 第二层：业务 service 用业务键兜住“同一业务事件反复到达”

这比单纯说“我用 Redis 去重”更像真实生产方案。

## 六、Redis 在这个项目里不只是缓存

### 6.1 Lua 限流怎么讲

`RateLimiterUtil` 用的是 Redis Lua 令牌桶：

- Redis Hash 里保存 `tokens` 和 `last_refill`
- Java 把当前时间按“微秒”传给 Lua
- Lua 内部计算补充令牌数
- 够就扣减，不够就拒绝
- 整个过程在 Redis 侧原子执行

代码里还有两个很适合加分的点：

- 提供了 `tryAcquireForRemoteCall()` 和 `tryAcquireForDatabase()` 两套常用预设
- 限流器异常时默认放行，也就是 **fail-open**，避免核心业务被限流组件拖垮

另外自研 `scheduledTask` Starter 里还有一层和任务治理强相关的“调用次数控制”：

- 每次远程调用前都会做原子计数
- 一旦超过任务配置里的 `maxCallLimit`，就会中断任务
- 同时还会把 URL 维度的调用明细回填到执行日志里

这意味着 `maxCallLimit` 不是写着好看，而是真的能约束高频外部调用。

### 6.2 分布式锁怎么讲更诚实也更加分？

当前 `RedisLockUtil` 的实现是：

- `tryLock`：`setIfAbsent(key, token, expire, TimeUnit.MILLISECONDS)`
- `lock`：每 50ms 轮询，超时放弃
- token 用 UUID

但这里我建议你在面试时 **主动指出一个真实风险**：

`unlock(key, token)` 现在是先读后删，不是 Lua compare-and-delete 的原子解锁。

所以你可以这样回答：

> 当前实现已经具备 token 校验和超时释放这些基本能力，但如果我要把它升级成更严谨的生产级分布式锁，我会把解锁改成 Lua 原子脚本，或者直接换成 Redisson，再结合业务幂等一起兜底。

这类“我知道当前实现到哪一步，也知道下一步怎么改”的回答，面试里很加分。

这里你还可以顺手补一句对比：

> 业务仓库里的 `RedisLockUtil` 偏基础互斥；但我自研的 `scheduledTask` 调度锁已经做了 Lua 原子释放和自动续租，说明我知道不同场景对锁可靠性的要求是不一样的。

## 八、启动恢复和异常收尾怎么讲

这部分很容易被忽略，但其实很加分。

`TaskStateCleanupRunner` 也是 `CommandLineRunner`，它会在应用启动时：

1. 找出当前应用下所有遗留的 `RUNNING` 日志
2. 把这些日志改成 `SYSTEM_SHUTTING_DOWN`
3. 尝试从 Redis 恢复调用次数和 URL 调用明细
4. 同步更新 `scheduled_task_config` 里的最近运行状态、耗时、调用次数、日/月累计次数

**面试时你可以这样说：**

> 我们考虑过“任务执行中应用突然重启”这种情况，所以不是简单依赖 finally 收尾，而是在下次启动时再跑一遍状态修复程序，保证日志表、配置表和 Redis 临时状态最终能收敛一致。

## 九、面试高频追问 & 参考答案

### Q1：这个项目里你最能体现架构能力的点是什么？

> 我一般会讲三件事：任务治理、Feign 失败补偿、消息可靠性。因为这三块都不是单点 CRUD，而是要考虑可观测、幂等、重试、限流、优雅关闭和多模块协同。

### Q2：为什么要做自研任务治理，不直接全部写死在 `@Scheduled`？

> 因为任务一多，真正难的不是“能定时执行”，而是任务元数据治理、动态启停、调用频次限制、环境隔离、责任人归属和执行留痕。这些能力如果没有统一平台，后面维护成本会非常高。

### Q3：Feign 调用失败为什么不只靠 Feign 自己重试？

> Feign 自带重试更适合短暂网络抖动，但不适合处理“服务暂时不可用、参数需要留档、后续人工可追溯”的场景。落库补偿的价值在于可以保留原始调用上下文，并让补偿链路独立运行。

### Q4：RocketMQ 去重为什么还要 service 层幂等？

> 因为 Consumer 侧去重更像“短时间内的快速拦截”，它基于消息内容和 TTL，不等于完整的业务语义幂等。真正稳妥的是 service 层再按业务唯一键做一次兜底。

### Q5：这个项目里你会主动承认什么不足？

> 我会主动提 Redis 锁解锁还不是原子 compare-and-delete，这说明我知道当前实现的边界；同时我也会给出明确改进方案，比如 Lua 原子脚本或 Redisson。

### Q6：如果让你继续优化，你会做什么？

> 我会优先做四件事：
> 1. 分布式锁改成 Lua 原子解锁或 Redisson
> 2. Feign 多次补偿失败后增加告警通知，而不是只留在库里
> 3. 对任务平台补充分片和更细粒度的并发控制
> 4. 把消费者幂等、去重、重试策略做成统一组件，减少重复实现

## 十、考前最值得背的 6 句话

1. 这是一个 7 模块的变电站生产管控平台，不是单体 CRUD，核心是任务、消息、规则和工单联动。
2. 自研任务治理 Starter 是独立项目，核心链路是自动装配、启动扫描、动态刷新、反射调用、AOP 监控和启动恢复。
3. Feign 失败补偿不是简单重试，而是 AOP 落库原始调用上下文，再由定时任务重放。
4. RocketMQ 可靠性做了两层：Consumer 短时去重 + service 业务幂等。
5. 业务仓库里的基础 Redis 锁我知道边界，但调度平台里的锁已经做了 Lua 原子释放和自动续租。
6. 任务平台的价值不在“能定时执行”，而在统一元数据、动态启停、生产保护、调用限流、异常通知和最终一致性修复。
