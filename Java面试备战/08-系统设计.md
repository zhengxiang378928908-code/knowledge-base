## 一、设计一个分布式定时任务平台

> 你就是做过这个的，把你的设计讲出来即可。

### 1.1 需求分析

- 支持动态添加/启停定时任务
- 任务执行状态追踪和日志
- 支持手动触发
- 防止重复执行
- 优雅关闭
- 外部 API 调用次数监控

### 1.2 整体架构

```
┌─────────────┐     ┌──────────────┐
│ 管理控制台    │────→│  REST API    │
│ （手动触发/停止）│     │ /tasks/{id}/ │
└─────────────┘     └──────┬───────┘
                           │
        ┌──────────────────┼──────────────────┐
        ↓                  ↓                   ↓
┌──────────────┐  ┌──────────────┐   ┌──────────────┐
│ Service A    │  │ Service B    │   │ Service C    │
│ (Starter)    │  │ (Starter)    │   │ (Starter)    │
│              │  │              │   │              │
│ @Monitored   │  │ @Monitored   │   │ @Monitored   │
│ TaskScanner  │  │ TaskScanner  │   │ TaskScanner  │
│ TaskAspect   │  │ TaskAspect   │   │ TaskAspect   │
│ DynamicConf  │  │ DynamicConf  │   │ DynamicConf  │
└──────┬───────┘  └──────┬───────┘   └──────┬───────┘
       │                 │                   │
       ↓                 ↓                   ↓
┌─────────────────────────────────────────────────┐
│                    MySQL                         │
│  task_config（任务配置）  task_log（执行日志）      │
└─────────────────────────────────────────────────┘
       │
       ↓
┌─────────────────────────────────────────────────┐
│                    Redis                         │
│  call_count（调用计数）  url_details（URL 统计）    │
└─────────────────────────────────────────────────┘
```

### 1.3 核心设计点

**任务发现与注册：**
- 自定义 `@MonitoredScheduledTask` 注解
- 启动时 `CommandLineRunner` 扫描 Bean → 反射找注解 → 同步到 DB

**动态调度：**
- `SchedulingConfigurer` 每分钟拉取 DB 中已启用的任务
- `ConcurrentHashMap<taskKey, ScheduledFuture>` 管理活跃任务
- 新增注册、删除取消

**执行监控（AOP）：**
- `@Around` 拦截任务方法
- 记录开始时间、状态、调用次数
- Redis 原子计数器追踪 API 调用次数

**并发控制：**
- `TaskExecutionRegistry`（ConcurrentHashMap）记录运行中的任务和线程
- 同一任务不能并发执行

**手动触发 & 停止：**
- 手动触发：`ManualExecutionContext` ThreadLocal 标记，绕过 isEnabled 检查
- 停止：`thread.interrupt()` → AOP 捕获中断异常

**优雅关闭：**
- `ApplicationListener<ContextClosedEvent>` → 设置全局标志
- AOP 的 finally 块检测到关闭标志 → 跳过清理，保持 RUNNING 状态
- 下次启动：`TaskStateCleanupRunner` 清理残留日志

## 二、设计一个预警规则引擎

### 2.1 需求

- 自然语言 → 结构化规则
- 支持复杂条件组合（AND/OR/嵌套）
- 规则可编辑、可回溯
- 支持多种设备类型和场景

### 2.2 方案

```
用户输入
   ↓
意图解析服务（你的 DSL 引擎）
   ↓
结构化 DSL
   ↓
规则执行引擎（Drools / 自研）
   ↓
预警结果
   ↓
通知/工单
```

### 2.3 DSL 设计

```json
{
  "entity": "breaker",
  "scenario": "sf6_test",
  "metrics": [
    {"id": "m1", "name": "SF6泄漏量", "stat": "latest", "time": {"range": "1y"}}
  ],
  "scope_filters": [
    {"field": "电压等级", "op": "gte", "value": "110kV"}
  ],
  "rules": [
    {
      "id": "r1",
      "when": {
        "op": "and",
        "args": [
          {"metric": "m1", "op": "gt", "value": 50},
          {"field": "运行状态", "op": "eq", "value": "在运"}
        ]
      },
      "then": {
        "action": "generate_warning",
        "message": "SF6泄漏量超标"
      }
    }
  ]
}
```

### 2.4 为什么用 DSL 而不是直接让 LLM 生成 SQL？

- SQL 直接暴露数据库结构，安全风险
- DSL 是业务层面的抽象，可以做校验
- DSL 到 SQL 的转换是确定性的（不需要 LLM）
- DSL 更易于展示给业务用户确认

## 三、设计一个高并发审核系统

> 结合邮储银行风控中台经验

### 3.1 需求

- 客户准入审核，支持高并发提交
- 黑白名单实时拦截
- 审批流程流转
- 数据统计

### 3.2 架构

```
客户端
   ↓
API Gateway（限流、鉴权）
   ↓
审核提交服务
   ├── Redis：黑白名单缓存（实时拦截）
   ├── RocketMQ：审核任务异步处理
   └── MySQL：持久化
   ↓
审核处理服务（消费者）
   ├── 风控规则引擎
   ├── 人工审批队列
   └── 结果通知
   ↓
统计分析服务
   └── 异步聚合，定时报表
```

### 3.3 关键设计

**黑白名单：**
- 全量名单加载到 Redis Set（定时同步）
- 查询用 `SISMEMBER`，O(1) 复杂度
- 批量校验用 Pipeline

**高并发处理：**
- 前端限流（令牌桶）
- 异步解耦：提交后立即返回，审核结果通过 MQ 通知
- 数据库写入批量化

**跑批优化：**
- 分批查询（每批 1000 条）
- 避免一次性加载全量到内存
- 多线程并行处理不同批次

## 五、微服务设计模式

### 5.1 服务拆分原则

- 单一职责
- 按业务领域拆分（DDD）
- 服务间松耦合
- 数据库独立（每个服务有自己的数据库）

### 5.2 服务通信

| 方式 | 场景 | 示例 |
|------|------|------|
| 同步（OpenFeign/gRPC） | 需要实时响应 | 查询设备信息 |
| 异步（RocketMQ） | 不需要实时、需要解耦 | 告警处理、统计 |
| 事件驱动 | 发布-订阅 | 设备状态变更通知 |

### 5.3 你项目中的微服务架构

```
变电 3.0 微服务体系（7 个模块）：
├── pms-substation-service      # 主业务服务（设备台账、预警规则）
├── pms-event-scheduler-service # 事件调度（RocketMQ 消费者）
├── pms-substation-data-handler # 数据处理（多数据源 MySQL + Oracle）
├── pms-scheduler-service       # 定时任务执行器
├── pms-safety-report-service   # 安全报告生成
├── pms-substation-entity       # 统一实体
└── pms-substation-common       # 公共组件（Redis、分页、Swagger）
```

### 5.4 分布式事务

**你项目中的做法：**
- 最终一致性：RocketMQ + 本地事务表
- 补偿机制：Feign Fallback + 定时重试

**常见方案对比：**

| 方案 | 一致性 | 性能 | 复杂度 |
|------|--------|------|--------|
| 2PC（两阶段提交） | 强一致 | 低 | 高 |
| TCC | 强一致 | 中 | 高 |
| Saga | 最终一致 | 高 | 中 |
| 本地消息表 | 最终一致 | 高 | 低 |
| RocketMQ 事务消息 | 最终一致 | 高 | 中 |
