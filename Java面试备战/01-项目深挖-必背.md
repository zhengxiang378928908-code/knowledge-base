## 一、电网预警规则解析平台 / DSL 意图引擎（2024.10 - 至今）

### 1.1 请介绍一下这个项目

**参考回答：**

这个项目是在变电站生产管控平台的基础上扩展的 AI 能力。核心目标是把电网业务人员用自然语言描述的预警规则，转换成结构化的 DSL（领域特定语言），用于支撑后续设备预警映射和执行。

整个处理流程是一个 8 步 Pipeline：
1. **文本标准化**：全角转半角、去除噪声
2. **Slot 识别**：多源抽取（正则 + 别名词典 AC 自动机 + LLM）
3. **LLM Slot 组装**：把正则和词典的命中结果交给 LLM 做最终槽位判定
4. **Slot 校验**：检查实体/场景/指标是否完整，不完整触发澄清
5. **混合检索**：BM25 + pgvector 混合召回相似规则范文
6. **DSL 生成**：用检索到的范文作为 few-shot 示例，LLM 生成业务 DSL
7. **DSL 校验**：结构校验、语义字段白名单、指标引用验证
8. **回译确认**：把 DSL 翻译成自然语言让用户确认

技术栈用的是 Python + FastAPI + PostgreSQL(pgvector) + BM25，部署用 Docker 4 容器（API 服务、数据库、前端、Nginx）。

---

### 1.2 为什么选 pgvector + BM25 混合召回，而不是纯向量检索？

**参考回答：**

电网领域有大量专业术语和缩写（比如 SF6、GIS、CT、PT），这些术语在语义空间上可能距离很远但关键词完全匹配。

- **纯向量检索的问题**：语义相似但关键词不匹配的情况下容易召回错误范文。比如"SF6泄漏"和"油气泄漏"语义相近，但业务含义完全不同。
- **纯 BM25 的问题**：无法捕获语义层面的相似性，比如"超过额定值"和"大于标准值"关键词不同但含义一致。
- **混合方案**：用 Reciprocal Rank Fusion（RRF）算法融合两路结果，公式是 `score(d) = Σ 1/(k + rank + 1)`，k 取 60 做平滑。两路都排名靠前的文档得分最高。

实际效果上，混合方案比单一方案在 top-3 命中率上更稳定，尤其是在电网行业名词别名多的场景下优势明显。

**追问：为什么 k 取 60？**

k 越大，排名差异的影响越平滑。k=60 是经验值，意味着即使一个文档在某一路排名 50+，在另一路排名前 3，综合得分依然可观。实际调参时在 20-100 之间试过，60 在我们的语料规模（约 300 条规则范文）上表现最好。

### 1.3 Slot 抽取具体怎么做的？

**参考回答：**

我们用的是多源抽取 + LLM 汇总的方案：

**第一层：正则抽取（RegexExtractor）**
- 电压等级：`110kV及以上` → `{base: "110kV", bound: "gte"}`
- 数值阈值：`大于等于50℃` → `{op: "大于等于", value: 50, unit: "℃"}`
- 时间窗口：`近3年` → `{value: 3, unit: "y", qualifier: "近"}`
- 次数/倍数：`3次`、`2倍`

**第二层：别名词典扫描（AliasScanner）**
- 基于 AC 自动机做多模式匹配
- 实体别名：`{"断路器", "开关", "SF6开关"}` → entity: breaker
- 场景提示：`{"SF6试验", "缺陷查询"}` → scenario: sf6_test
- 动作别名：`{"告警", "预警", "闭环"}` → action type

**第三层：歧义消解（EntityDisambiguator）**
- 当多个候选实体/场景命中时，根据文本距离、共现模式、实体特异性做消歧

**第四层：LLM Slot 组装（LLMSlotAssembler）**
- 把前三层的命中结果打包成 JSON 发给 LLM
- System Prompt：`"你是电力设备规则意图识别助手，输出 JSON，只保留有把握的信息"`
- temperature=0，确保确定性输出
- 超时 8 秒，超时则 fallback 到纯规则结果

**为什么不全交给 LLM？**
- 正则和词典在结构化信息抽取上准确率更高、延迟更低
- LLM 负责语义层面的综合判断，比如判断"50mL"到底修饰哪个指标
- 这样分工既保证了准确性又控制了成本

### 1.4 多轮校验（回译/逻辑校验）具体流程是什么？

**参考回答：**

校验分三层：

**Slot 校验（Step 4）：**
- 检查实体是否存在于标准知识库
- 检查场景是否合法
- 检查指标是否适用于该实体+场景组合
- 不完整时触发澄清，类型有：
  - `entity_choice`：多个候选实体，让用户选
  - `entity_multi_select`：多设备规则
  - `metric_fill`：缺少可执行指标
  - `unsupported_capability`：检测到不支持的指标（如同比增长）

**DSL 校验（Step 7）：**
- 顶层结构校验：只允许 entity/scenario/metrics/rules/output 等固定字段
- 指标引用校验：rules 中引用的 metric ID 必须在 metrics 列表中存在
- 语义字段白名单：scope_filters 只能用预定义的语义字段
- 嵌套深度限制：条件嵌套最多 3 层
- 自动修复：标准化指标名、补充 canonical_metric_name

**回译确认（Step 8）：**
- 把最终 DSL 转换成自然语言描述，比如：`"这条规则面向断路器，场景是SF6试验，包含1条规则，关注指标为SF6泄漏量"`
- 用户确认 → 保存 DSL
- 用户拒绝 → 将该 DSL 加入 negative_examples，对应的语料范文在下次排序时乘以 0.3 惩罚系数，重新生成

### 1.5 遇到的最难的 bad case 是什么，怎么解决的？

**参考回答（举例）：**

电网行业一个典型难点是**别名多 + 嵌套条件**。比如用户输入：

> "110kV及以上断路器SF6气体泄漏年超过50mL且运行年限超过15年的设备生成预警"

这里包含：
- 电压等级范围过滤（scope_filter）
- 实体（断路器）
- 场景（SF6试验）
- 两个指标条件的 AND 嵌套

难点在于 LLM 容易把"运行年限超过15年"误解为时间窗口而不是设备属性过滤。

解决方案：
1. 正则层先把"110kV及以上"、"年超过50mL"、"超过15年"分别抽出来
2. 别名扫描锁定 entity=breaker, scenario=sf6_test
3. LLM 组装时，prompt 中明确区分"指标阈值"和"设备属性过滤"
4. DSL 校验阶段，如果发现"运行年限"出现在 metrics 而不是 scope_filters 中，自动纠正

另外建立了 299 条槽位 + 68 条 slot acceptance cases 的回归测试集，每次改动都跑全量回归，保证不会改一个 fix 一个 break 一个。

### 1.6 为什么用 FastAPI 不用 Java（Spring Boot）？

**参考回答：**

这个项目是在既有 Java 后端业务体系上扩展的 AI 能力模块。选 Python/FastAPI 的原因：

1. **AI/NLP 生态**：Python 在 LLM 调用、embedding、BM25（rank_bm25 库）、文本处理方面生态更成熟
2. **开发效率**：RAG pipeline 需要大量实验和快速迭代，Python 的开发和调试效率更高
3. **团队技能**：AI 方向的工程实践中 Python 是事实标准
4. **架构隔离**：作为独立服务通过 API 与 Java 主体系交互，不影响已有的 Java 微服务架构

核心业务逻辑（设备台账、工单流转、预警执行）仍然在 Java 微服务体系中，Python 服务只负责"自然语言 → DSL"这一环节。

## 二、变电站生产管控平台 - 变电3.0（2023.08 - 2024.10）

### 2.1 自研定时任务 Starter 为什么不用 XXL-Job？

**参考回答：**

主要有三个原因：

1. **部署环境限制**：项目部署在电网内网环境，XXL-Job 需要额外部署调度中心（admin），内网部署审批流程长，增加运维复杂度
2. **轻量需求**：我们的需求是百级定时任务的统一管理，不需要分片、DAG 编排等重量级特性
3. **深度集成需求**：需要跟踪每个任务的外部 API 调用次数、支持动态启停、优雅关闭状态保持等业务定制需求

**我的方案核心设计：**

- **注解驱动发现**：自定义 `@MonitoredScheduledTask` 注解，启动时通过 `CommandLineRunner` 扫描所有 Bean 的方法，使用 AOP 代理获取真实类，反射找到注解标记的方法，同步到数据库
- **动态调度**：实现 `SchedulingConfigurer`，每分钟从数据库拉取已启用的任务列表，用 `ConcurrentHashMap<String, ScheduledFuture<?>>` 管理活跃任务，新增的注册、删除的取消
- **AOP 执行监控**：`@Around` 切面拦截任务执行，记录开始/结束时间、状态、调用次数（通过 Redis 计数器），防并发执行（`TaskExecutionRegistry` 用 ConcurrentHashMap 记录运行中的任务和线程）
- **手动触发**：通过 REST API 触发，用 `ManualExecutionContext`（ThreadLocal）标记为手动执行，绕过 isEnabled 检查
- **优雅停止**：`thread.interrupt()` + AOP 捕获 `InterruptedException`，区分手动停止（HAND_SHUTTING_DOWN）和应用关闭（SYSTEM_SHUTTING_DOWN）
- **应用关闭清理**：监听 `ContextClosedEvent` 设置全局关闭标志，下次启动时 `TaskStateCleanupRunner` 清理上次残留的 RUNNING 状态日志

### 2.2 OpenFeign 重试补偿机制的具体设计

**参考回答：**

微服务间通过 OpenFeign 调用，网络抖动会导致调用失败。我设计了一个**失败拦截 + 落库重试**的补偿机制：

**核心组件：**

1. **自定义注解 `@FallbackToDatabase`**：标记需要补偿的 Feign 方法，可配置最大重试次数

2. **AOP 切面 `FeignFallbackAspect`**：
```java
@Around("@annotation(FallbackToDatabase)")
public Object around(ProceedingJoinPoint point) {
    try {
        return point.proceed();  // 正常调用 Feign
    } catch (Exception e) {
        // 失败：序列化方法名、参数，写入 feign_fallback_task 表
        FeignFallbackTask task = new FeignFallbackTask();
        task.setBeanName(feignClient.name());
        task.setMethodName(method.getName());
        task.setMethodParams(objectMapper.writeValueAsString(args));
        task.setStatus(0);  // 待重试
        task.setMaxRetryCount(annotation.maxRetryCount());
        feignFallbackTaskService.saveTask(task);
        throw e;
    }
}
```

3. **定时重试调度**：定时任务扫描 status=0 且 retryCount < maxRetryCount 的记录，通过反射重新调用 Feign 方法

4. **幂等保证**：
   - 每个 Feign 调用带业务唯一 ID（雪花算法生成）
   - 被调用方做幂等校验（根据业务 ID 去重）
   - 重试时使用原始参数，不重新生成 ID

### 2.3 Redis Lua 限流用的什么算法？

**参考回答：**

用的是**令牌桶算法**，通过 Redis Lua 脚本保证原子性。

```java
// 调用方式
rateLimiterUtil.tryAcquire("api:limit:endpoint", capacity=20, rate=10, requested=1);
```

**Lua 脚本逻辑：**
1. 读取当前令牌数和上次补充时间
2. 计算时间差，按速率补充令牌（不超过容量上限）
3. 如果令牌够，扣减并返回 1（通过）
4. 如果不够，返回 0（拒绝）
5. 整个过程在 Redis 侧原子执行，不存在并发问题

**为什么用 Lua 而不是分布式锁 + Java 实现？**
- Lua 脚本在 Redis 内执行是原子的，不需要额外的锁
- 避免了网络往返（一次 EVAL 搞定 vs 多次 GET/SET）
- 性能更高，适合预警高峰场景

### 2.4 RocketMQ 消息丢失 / 重复消费怎么处理的？

**参考回答：**

项目中有 29 个消费者、34 个 Topic，主要用于告警和统计异步处理。

**防丢失：**
- Producer 端：同步发送 + 重试（默认 3 次）
- Broker 端：同步刷盘（电网场景数据可靠性要求高）
- Consumer 端：业务处理成功后才返回 CONSUME_SUCCESS

**防重复消费：**
- 实现了 `MessageDedupUtil`，基于 Redis 做消息去重：
```java
// 消费前检查（3分钟窗口）
if (!messageDedupUtil.tryAcquireByMessage(topic, message, 3L * 60)) {
    return;  // 跳过重复消息
}
// 处理成功后标记（10分钟窗口）
messageDedupUtil.markMessageSuccess(topic, message, 10L * 60);
```
- 消息体中包含 UUID，用 Redis SET key+TTL 做短时去重
- 处理窗口（3分钟）< 成功标记窗口（10分钟），避免短暂重复

**重试策略：**
```java
if (messageExt.getReconsumeTimes() >= 3) {
    log.error("最大重试次数已达，进入死信队列");
    return;  // 不再重试
}
throw e;  // 触发 RocketMQ 重试
```

### 2.5 请介绍一下分布式锁的实现

**参考回答：**

项目中 `RedisLockUtil` 实现了两种模式：

**非阻塞锁（tryLock）：**
```java
String token = redisLockUtil.tryLock("order:generate:lock", 20000);
// 尝试一次，成功返回 token，失败返回 null
```

**阻塞锁（lock）：**
```java
String token = redisLockUtil.lock("order:generate:lock", 20000, 5000);
// 在 5 秒内不断重试，超时返回 null
```

**关键设计：**
- 加锁：`SET key token NX PX expireTime`，token 是 UUID
- 释放：先比较 token 是否一致，一致才删除（防止误释放别人的锁）
- 自动过期：防止死锁

**为什么不用 Redisson？**
- 项目需求简单，只需要基础的互斥锁
- 不需要可重入、红锁等高级特性
- 减少依赖，电网内网环境依赖管理比较严格

## 三、邮储银行新客户管控及风控中台（2020.06 - 2023.06）

### 3.1 黑白名单验证核心模块怎么设计的？

**参考回答：**

- 黑白名单数据量大，存在 PostgreSQL 中
- 高频查询用 Redis 缓存：名单数据加载到 Redis Set/Hash 中
- 批量验证：一次请求可能校验几百个客户，用 Pipeline 批量查询 Redis
- 多维查询：支持按证件号、手机号、商户名等多个维度匹配
- SQL 优化：对高频查询字段建立复合索引，IN 查询拆分（每批不超过 1000 条）

### 3.2 OOM 问题怎么排查和解决的？

**参考回答：**

遇到过两类 OOM：

**跑批 OOM：**
- 原因：财务报表统计一次性加载全量数据到内存
- 排查：`-XX:+HeapDumpOnOutOfMemoryError` 抓 dump，MAT 分析发现大 List
- 解决：改为流式分批查询 + 处理，每批 1000 条，处理完释放引用

**慢查询导致连接池耗尽 → OOM：**
- 原因：复杂报表 SQL 没有走索引，执行时间过长，线程阻塞导致请求堆积
- 排查：监控发现 Druid 连接池活跃连接数持续增长
- 解决：优化 SQL（加索引、拆分大查询），设置连接池超时时间

### 3.3 多微服务节点下的锁争用问题

**参考回答：**

多节点部署后，跑批任务在多个节点同时触发，导致重复处理和数据不一致。

解决方案：
- 引入 Redis 分布式锁，跑批开始前先获取锁
- 加锁粒度：按业务批次 ID 加锁，不同批次可以并行
- 锁超时设置大于最大批处理时间，避免任务还在跑锁就过期了
