## 一、HashMap put 流程

### 完整话术

> HashMap put 一个元素分五步。
>
> 第一步算 hash，拿 key 的 hashCode，高 16 位和低 16 位异或，目的是让高位也参与运算减少冲突。第二步定位桶，用数组长度减 1 再和 hash 做与运算算出桶下标，位运算代替取模所以容量必须是 2 的幂。第三步放元素，桶是空的直接放；桶里第一个节点 key 相同就覆盖 value；桶里是链表就遍历到尾部追加，中间发现相同 key 就覆盖；桶里是红黑树就走树的插入。第四步判断树化，链表追加后长度大于等于 8 并且数组容量大于等于 64 才转红黑树，容量不够就先扩容不树化。第五步判断扩容，元素总数超过容量乘以 0.75 就触发扩容，数组翻倍，所有元素重新定位。

### 必记数字

> 默认容量 16，负载因子 0.75，树化阈值链表大于等于 8 且容量大于等于 64，退化阈值红黑树节点小于等于 6。

## 二、线程池

### 完整话术：七个参数和执行流程

> 线程池七个核心参数：核心线程数、最大线程数、非核心线程存活时间、时间单位、任务等待队列、线程工厂、拒绝策略。
>
> 执行流程是：任务进来先看核心线程有没有空闲的，有就直接执行；核心线程都在忙就进等待队列排队；队列也满了才创建非核心线程，但不超过最大线程数；线程数到最大队列也满了就触发拒绝策略。注意顺序是先入队列再创建非核心线程，不是先创建到最大再入队列，这个很多人搞反。
>
> 为什么不用 Executors 创建？因为 newFixedThreadPool 和 newSingleThreadExecutor 用的是无界队列，任务堆积能 OOM。newCachedThreadPool 最大线程数是 Integer.MAX_VALUE，可能创建几万个线程。实际项目都手动创建 ThreadPoolExecutor，队列一定用有界的。

### 完整话术：四种拒绝策略

> AbortPolicy 直接抛异常，是默认策略。CallerRunsPolicy 让提交任务的线程自己执行，起到限流反压效果。DiscardPolicy 静默丢弃不抛异常。DiscardOldestPolicy 丢弃队列里最老的任务放新的进去。我项目里选的 CallerRunsPolicy，因为数据同步宁可慢也不能丢。

### 完整话术：CPU 密集 vs IO 密集

> CPU 密集型核心线程数等于 CPU 核数加 1，再多线程 CPU 也跑满了没用。IO 密集型核心线程数等于 CPU 核数乘以 2，因为线程大部分时间在等 IO，多开一些能提高利用率。我的数据同步任务是典型 IO 密集型，大量远程调用和数据库写入，按 CPU 核数乘以 2 配的。

## 三、Spring 事务

### 完整话术：传播机制

> @Transactional 默认传播行为是 REQUIRED，有事务就加入没有就新建。最常考的三种：REQUIRED 是默认的加入或新建；REQUIRES_NEW 不管有没有事务都新建独立事务，比如记操作日志不想因为主业务回滚把日志也回滚了；NESTED 在当前事务内创建保存点，内层回滚只回到保存点不影响外层。

### 完整话术：五种失效场景

> 第一，方法不是 public 的，Spring 事务基于 AOP 代理非 public 拦截不到。第二，同类自调用，A 方法调 B 方法在同一个类里，走的是 this 调用不是代理对象，B 上的 @Transactional 不生效。第三，异常被 catch 吞掉了，Spring 默认只对 RuntimeException 和 Error 回滚，catch 了没往外抛事务不回滚。第四，数据库引擎不支持事务比如 MyISAM。第五，传播行为设置不当比如 NOT_SUPPORTED 会挂起当前事务。

### 项目场景

> 我项目里故障跳闸消息消费时，落主记录和落子事件在同一个事务里保证原子性，但发短信通知放在事务外面异步做，不让通知失败拖回整个事务。

## 四、Spring Bean 生命周期 + 循环依赖

### 完整话术：四个阶段

> Bean 生命周期四个阶段。第一实例化，Spring 通过反射创建空对象。第二属性注入，把 @Autowired 和 @Value 的依赖注入进去。第三初始化，先调 @PostConstruct 标注的方法，再调 InitializingBean 的 afterPropertiesSet，最后调 init-method，如果有 AOP 这一步会生成代理对象。第四销毁，容器关闭时先调 @PreDestroy 再调 destroy 再调 destroy-method。

### 完整话术：三级缓存

> 循环依赖用三级缓存解决。一级缓存存完整初始化好的 Bean，二级缓存存已实例化但未初始化的早期 Bean，三级缓存存 Bean 的 ObjectFactory 工厂对象。
>
> 流程是这样的：创建 A 发现依赖 B，先把 A 的工厂放三级缓存，然后去创建 B。创建 B 发现依赖 A，从三级缓存拿到 A 的工厂创建 A 的早期引用放二级缓存。B 拿到 A 的早期引用就能完成初始化。最后 A 也能完成初始化。
>
> 为什么要三级不是两级？因为有 AOP 代理。三级缓存的 ObjectFactory 可以在需要时生成代理对象，保证 B 拿到的是 A 的代理而不是原始对象。两级缓存没法在早期暴露阶段决定是否生成代理。
>
> 构造器注入的循环依赖解决不了，因为对象都没创建出来没法放缓存。

## 五、MySQL 索引 + B+ 树

### 完整话术：B+ 树

> InnoDB 默认索引结构是 B+ 树，和 B 树两个核心区别。第一非叶子节点只存 key 不存数据，一个节点能放更多索引项，树更矮，一般 3 到 4 层就能存几千万条数据，查询时磁盘 IO 次数少。第二叶子节点之间有双向链表，范围查询直接沿链表遍历不用回到上层，对 BETWEEN、ORDER BY 这些操作非常友好。
>
> 主键索引是聚簇索引，叶子节点存完整行数据。普通索引是非聚簇索引，叶子节点只存主键值，查完还要拿主键去聚簇索引再查一次，这叫回表。要避免回表就用覆盖索引，让查询字段都在索引里。

### 完整话术：最左前缀

> 联合索引 a b c 按 a b c 顺序排列，查询必须从最左列开始匹配。WHERE a=1 AND b=2 能用上 a 和 b。WHERE b=2 用不上因为跳过了 a。WHERE a=1 AND c=3 只能用上 a，c 用不上因为中间断了 b。WHERE a>1 AND b=2，a 能用上但 a 用了范围查询后 b 就用不上了。

### 完整话术：索引失效六种

> 左模糊 LIKE 百分号 xxx；对索引列用了函数或计算；隐式类型转换比如 varchar 列用数字查；OR 条件中有一边没索引；违反最左前缀原则；优化器判断全表扫描比走索引快。

## 六、慢查询排查

### 完整话术（STAR 结构）

> 我在电网项目里碰到过设备台账查询接口慢的问题，联表查设备信息、监测数据和试验记录，页面加载要 3 到 4 秒。
>
> 排查过程：先开慢查询日志找到耗时最长的 SQL，然后用 EXPLAIN 分析。发现两个问题，type 是 ALL 全表扫描，Extra 里有 Using filesort 说明排序没走索引。看了表结构发现联合索引顺序不对导致索引失效。
>
> 解决：调整联合索引顺序把区分度高的字段放前面；SELECT * 改成只查需要的字段走覆盖索引；大偏移分页改成 WHERE id 大于上一页最后一个 id 再 LIMIT，避免深分页。优化后从 3 到 4 秒降到 200 毫秒以内。
>
> EXPLAIN 重点看三个字段：type 不能是 ALL 至少到 ref 或 range；key 是 NULL 就是没走索引；Extra 有 Using filesort 或 Using temporary 要优化，Using index 是覆盖索引是好的。

## 七、数据库死锁排查

### 完整话术

> 死锁四个必要条件是互斥、占有且等待、不可抢占、循环等待，四个同时满足才会死锁。
>
> 排查流程：第一步看日志，InnoDB 检测到死锁会自动回滚其中一个事务，错误日志有 Deadlock found 的记录。第二步用 SHOW ENGINE INNODB STATUS 查看最近一次死锁详情，能看到两个事务分别持有什么锁、在等什么锁、执行的 SQL。第三步分析加锁顺序，死锁根本原因几乎都是两个事务以相反顺序加锁。
>
> 预防四招：第一统一加锁顺序，所有事务都按固定顺序访问表和行；第二缩短事务，持锁时间越短概率越低，不要在事务里做远程调用；第三合理加索引，没索引的 UPDATE 会锁全表大幅增加死锁概率；第四 InnoDB 自动检测默认开启，检测到死锁回滚代价小的事务。
>
> 我项目里遇到过，工单流转场景，告警触发创建工单和人工处理工单回写告警，两个操作访问告警表和工单表的顺序不一致导致偶发死锁。统一成先操作告警表再操作工单表就解决了。

## 八、Redis 缓存穿透、击穿、雪崩

### 完整话术

> 缓存穿透是查的数据在数据库里根本不存在，缓存自然也没有，每次请求都打到数据库。解决方案两个：一是布隆过滤器在缓存前拦截，快速判断数据存不存在；二是缓存空值，查到不存在就缓存一个空结果设个短过期比如 2 到 5 分钟。
>
> 缓存击穿是某个热点 key 刚好过期，大量并发请求同时打到数据库。解决方案：一是互斥锁重建缓存，第一个请求用 SET NX PX 拿锁去查数据库更新缓存，其他请求等一下或返回旧值；二是逻辑过期，key 永不过期但 value 里带过期时间字段，发现过期了异步更新。
>
> 缓存雪崩是大量 key 同时过期或 Redis 整体宕机。解决：过期时间加随机偏移避免集中过期；热点数据不设过期后台定期更新；Redis 宕机做降级限流比如本地缓存兜底。
>
> 我项目里用 Redis 做设备级计数和短时去重，这些 key 都有 TTL。为了防止 Redis 异常导致所有消息被拦截或全部放行，做了 fail-open 设计——Redis 不可用时默认放行。这就是缓存雪崩场景下的一种降级策略。

| HashMap | 默认16，负载0.75，树化≥8且容量≥64，退化≤6 |
