# 专题二：任务治理平台

> 这是你最核心的卖点，面试官大概率会围绕这个追 3-4 层。
> 这条线的逻辑：扫描注册 → 动态调度 → 分布式锁 → 高吞吐数据同步 → OOM 排查 → 启动恢复
> 面试官问到任何一个点，你都能顺着往下讲。

---

## 入口问题："你的任务平台是怎么做的？"

### 完整话术（直接背）

> 我从核心链路讲吧。
>
> **第一步是自动发现和注册。** 我定义了一个自定义注解，业务开发在定时任务方法上加这个注解，标上任务分组、cron 表达式、负责人、最大调用次数这些治理信息。应用启动的时候，有一个扫描器实现了 Spring 的 CommandLineRunner，容器初始化完成后自动触发。它会扫描所有 Bean 上标了这个注解的方法，把元数据提取出来，跟数据库里已有的任务配置做比对——新的就插入，已有的就更新。这样每次启动，数据库的任务清单就和代码保持同步了。
>
> **第二步是动态调度。** 我没有用 Spring 的 @Scheduled，而是自己接管了调度。扫描器注册完之后，会用 Spring 的 TaskScheduler 按数据库里的 cron 注册调度，返回的 ScheduledFuture 缓存在内存 Map 里。然后组件会开一个刷新线程，每分钟从数据库拉一次最新配置。发现 cron 变了就先 cancel 旧的再按新 cron 重建；发现任务被禁用了就直接 cancel。所以运维改数据库配置，最多一分钟生效，不用重启。
>
> **第三步是分布式锁。** 多个实例同时触发同一个任务时，只有一个能执行。任务启动时会生成一个随机 token，用 Redis 的 setIfAbsent 抢锁，同时设 TTL 120 秒防止死锁。拿到锁之后会启动一个续租线程，每 30 秒用 Lua 脚本先比对 token 再续期，保证续租是原子的。任务执行完或异常，也是用 Lua 脚本先比对 token 再删除，保证不会误删别人的锁。
>
> **第四步是启动恢复。** 生产上任务跑到一半应用可能重启，比如发版或者机器故障。任务日志表里会留下"运行中"的脏数据，Redis 里的计数也可能和数据库不一致。我在应用启动时挂了一个恢复程序，也是 CommandLineRunner，它会把遗留的运行中日志改成"系统关闭中断"，从 Redis 恢复调用计数同步到数据库，清理临时状态。核心思路是不依赖 finally，因为进程被 kill 的时候 finally 不一定执行，而是下次启动做状态修复，保证最终一致。

## 追问："你说高吞吐数据同步，具体怎么做的？"

### 完整话术

> 我们有一批定时任务负责从外部系统拉取设备遥测数据写入本地库，单次任务要处理几千台设备、几十万条记录。
>
> 核心设计有三个点。第一是时间窗口滚动拉取，不是全量拉，而是按最近 30 分钟的窗口增量拉，减少重复和接口压力。第二是并发控制，设备 ID 分批后扔线程池并发执行，子任务用 Future 收集结果，线程池按 IO 密集型配，核心线程数是 CPU 核数乘以 2，拒绝策略用 CallerRunsPolicy，满了让提交线程自己跑，起到自然反压。第三是可中断，子任务会检查中断状态，任务平台发停止指令或应用停机时能及时中断，Future.get 抛中断异常时会显式恢复中断标志，不会吞掉。
>
> 数据重复靠设备 ID 加时间戳的联合唯一索引，批量插入用 INSERT IGNORE，数据库层面兜底。

## 追问："有没有遇到过线上问题？"

### 完整话术（OOM 排查，STAR 结构）

> 有。在这个数据同步任务上遇到过一次 OOM。
>
> 现象是某天任务跑到一半服务直接挂了，日志抛 OutOfMemoryError Java heap space。启动参数配了 HeapDumpOnOutOfMemoryError，自动 dump 了堆快照。我用 MAT 打开看 Dominator Tree，发现一个 ArrayList 占了大量内存，顺着引用链往上查，定位到数据同步 Service 里——它一次性把几万台设备的监测数据全查出来放在内存做处理。
>
> 原因是设备规模增长后，单次全量拉取的数据量远超预期，内存直接撑爆。
>
> 解决方案是改成分批处理。设备 ID 每批 100 台，拉完处理完再拉下一批。写库也分批，每批 1000 条批量插入。处理完一批就释放引用，不会一直持有全量数据。改完之后内存峰值从接近堆上限降到正常水平，任务稳定运行。后来我把这个分批策略作为规范推广到了其他同步任务。

## 追问："CPU 飙高遇到过吗？怎么排查？"

### 完整话术

> 遇到过。有一次定时任务高峰期，监控告警 CPU 持续 90% 以上。
>
> 排查过程是这样的：先用 top 看整机 CPU，确认是哪个进程，用 jps 确认是数据处理服务。然后用 top -H -p 加上进程号，找出 CPU 最高的线程，记下线程 ID。把线程 ID 转成十六进制，用 jstack 导出线程栈，搜这个十六进制 ID 对应的线程。
>
> 看线程栈发现是数据清洗逻辑里有一段循环在做大量字符串拼接和正则匹配，数据量一大就变成 CPU 热点。
>
> 解决办法是字符串拼接改 StringBuilder，正则预编译复用，加上分批处理避免单次数据量过大。CPU 从 90% 降到 40% 左右。
>
> 补充一句，CPU 飙高除了代码热点，还可能是频繁 Full GC、死循环、或者锁竞争严重导致大量线程 BLOCKED。

## 追问："为什么不用 XXL-Job？"

### 完整话术

> 我们评估过。核心原因是我们的痛点不是复杂的任务编排，而是任务元数据的统一治理和内网环境下的轻量落地。XXL-Job 需要额外部署调度中心，在我们内网环境里多一个组件就多一份运维成本。而我们自研的 Starter 直接嵌入业务应用，不需要额外部署，就能覆盖注册、启停、监控、锁、恢复这些能力。对我们的场景来说，投入产出比更高。

---

## 追问："业务仓库里的锁和调度平台的锁有什么区别？"

### 完整话术

> 调度平台里的锁更严谨。TTL 120 秒加每 30 秒 Lua 原子续租，释放的时候也是 Lua 先比对 token 再删除。业务仓库里还有一个基础的分布式锁工具类，那个解锁是先读 token 再删 key，不是原子的，中间有窗口可能误删。我知道这个差距，但业务场景对锁的可靠性要求没那么高，调度平台才需要更严谨的实现。
