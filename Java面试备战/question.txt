Q1. Can you describe the system you worked on?

A:

A:
My main responsibility was building and maintaining backend services for device data ingestion and downstream processing.

Q3. Why did you use a message-driven architecture?

A:
We used a message-driven architecture to decouple upstream device traffic from downstream business services.

This helped protect core systems under burst traffic and improved overall system stability.

Q4. What kind of message volume did you handle?

A:
The system handled around 250 thousand events per day on average.

Traffic was not evenly distributed and could spike during certain periods.

Q5. How did you handle burst traffic?

A:
We handled burst traffic mainly through asynchronous processing and backpressure mechanisms.

Message queues helped buffer traffic spikes, and we also used throttling and consumer group isolation to prevent downstream overload.

Q6. How did you ensure message processing reliability?

A:
We implemented retry and dead-letter mechanisms to handle transient failures.

For unrecoverable cases, messages were sent to a dead-letter queue for manual inspection and recovery.

Q7. How did you handle duplicate messages?

A:
We implemented idempotent consumers.

Each message had a unique deduplication key, and we checked processing state before applying any business changes.

Q8. How did you ensure data consistency?

A:
We used a combination of idempotency, explicit state transitions, and recovery workflows.

This ensured correctness under retries, partial failures, and message reprocessing.

Q9. Can you describe a production incident you handled?

A:
One typical incident was message backlog caused by downstream database contention.

Message consumption slowed down, and messages started to accumulate in the queue.

I identified the bottleneck through monitoring and logs, then applied traffic throttling and optimized database access patterns to gradually recover the system.

Q10. How did you detect incidents?

A:
We defined service-level metrics and alerting rules for key components.

This reduced incident detection time from around 25 minutes to about 2 minutes.

Q11. What kind of monitoring did you set up?

A:

Q20. How do you ensure idempotency in message consumption?

A（锁定版）

We ensured idempotency mainly at the consumer side.

Each message carried a unique business key, and before processing we checked whether the message had already been handled.

If it was already processed, we skipped the business logic to avoid duplicate effects.

A（锁定版）

🔒 Q26. How do you handle database deadlocks in production?

A（LOCKED VERSION）

When a deadlock occurs in production, we first identify the involved transactions and SQL through database logs and monitoring information.

In most cases, the root cause is either inconsistent update order across services or long-running transactions holding locks for too long.

To fix this, we typically enforce a consistent resource access order, reduce transaction scope, optimize queries with proper indexes, and add retry logic at the application level to safely handle occasional deadlocks.

Q27. Can you describe another production incident you handled?

A（LOCKED VERSION）

Another incident involved increased API latency in production.

We noticed response times gradually increasing, while error rates remained low.

After investigation, the root cause was a slow database query caused by missing indexes and inefficient query patterns under higher load.

We optimized the query, added proper indexes, and reduced unnecessary data access.

After the fix, latency returned to normal and the system stabilized.
