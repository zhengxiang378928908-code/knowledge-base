## 一、怎么使用这份清单

优先拿三类点：

- 能抬高层级的：平台化、治理能力、稳定性、架构取舍
- 能证明你真做过的：少量关键机制、边界条件、回归口径
- 能支撑面试追问的：bad case、设计原因、风险和下一步优化

不要直接往简历里塞的：

- 接口路径、控制器名、过多类名
- 细碎实现步骤
- 没有业务价值支撑的数字堆砌

## 二、RAG 项目最值钱的点

### 2.1 最适合放简历的点

1. 这不是单轮调用大模型的 demo，而是一个自然语言到 Business DSL 的后端系统。
2. 不是纯 prompt 工程，而是“规则抽取 + 混合检索 + DSL 生成 + 多轮校验 + 回译确认”的完整编排链路。
3. 系统支持多轮澄清和 bad case 纠偏，说明你做的是有状态的 AI 应用，而不是一次性请求。
4. 有固定回归基线和测试体系，说明这条链路可以持续迭代，不是靠人工点几个 case。

### 2.2 最适合面试展开的点

- **业务 DSL 分层**
  价值：LLM 只负责生成业务规则，不直接碰物理表和 SQL，降低了生成复杂度，也让业务人员可以确认结果。

- **多源 Slot 抽取**
  价值：不是把识别全交给 LLM，而是让规则、词典和 LLM 分工，既控制延迟，也控制错误率。

- **多轮澄清状态管理**
  价值：项目里真实保留了 `pending_clarification`、`negative_examples`、`deprioritized_corpus_ids`、`retry_count` 这些状态，说明你做了反馈闭环。

- **检索后的负反馈降权**
  价值：用户否定当前 DSL 后，不只是重新问一次模型，而是把当前范文降权，让下一轮结果能主动避开已知错误路径。

- **回归体系**
  价值：当前口径是 `42 + 26 = 68` 条基线，`backend/tests/` 里还有 `49` 个 pytest 文件。这个点很适合证明你做的是工程系统。

### 2.3 最容易加分的“设计判断”

- 为什么不用纯向量检索：
  电网场景既有强关键词，又有强语义相似，`pgvector + BM25` 混合检索比单路更稳。

- 为什么不把识别全交给 LLM：
  数字、电压、时间窗口这类结构化值，规则和词典更稳；LLM 更适合做语义综合判断。

- 为什么要做多轮澄清：
  电网规则里实体歧义、指标缺失、能力不支持都很常见，不做澄清很难稳定落成可执行 DSL。

- 为什么这个项目有工程价值：
  有状态管理、依赖探活、降级策略和回归基线，所以它更像 AI 后端系统，不是 prompt demo。

### 2.4 简历可直接复用的话术

- 基于既有 Java 微服务体系扩展 AI 规则解析能力，主导自然语言到 Business DSL 的后端主链路设计与落地。
- 设计并落地多源 Slot 抽取、混合检索、多轮校验与回译确认机制，提升复杂电网规则的结构化解析稳定性。
- 构建固定回归基线与测试体系，将规则解析做成可持续迭代的后端服务，而不是单次 prompt 调用。

## 三、变电 3.0 / scheduledTask 最值钱的点

### 3.1 最适合放简历的点

1. 你做的不是“几个定时任务”，而是一套轻量级任务治理平台。
2. 这套平台解决的不是“能定时执行”，而是任务元数据治理、动态启停、执行监控、人工控制和启动恢复。
3. 你不只是写业务模块，还做了失败补偿、消息可靠性、调用边界控制这些平台级能力。
4. 你知道不同锁方案的可靠性边界，不会把“能用”和“严谨生产级”混为一谈。

### 3.2 最适合面试展开的点

- **自研任务治理平台**
  价值：任务平台有独立 starter 源码，支持自动发现、动态启停、执行留痕、手动控制和启动恢复，适合体现平台化治理能力。

- **调用上限控制**
  价值：`maxCallLimit` 不是写在注解里好看，而是真的会在任务执行过程中原子计数，超限后中断任务并落明确状态。

- **长耗时任务的工程化处理**
  价值：像 `TelemetryDataTask` 这种任务，不只是定时拉数，而是按时间窗口滚动拉取、设备 ID 分批、写库再分批、子任务 `Future` 汇总，并且显式把中断往上抛。这类细节很适合证明你做过“慢 IO + 批处理 + 可中断”的任务链路。

- **启动恢复**
  价值：应用重启后会补收遗留运行状态，把日志、配置和 Redis 临时状态重新收敛一致，这很像真实生产系统才会考虑的问题。

- **调度锁比业务锁更严谨**
  价值：调度平台里的锁有 TTL、续租和原子释放，而业务仓库里的基础 Redis 锁只是“基本互斥能力”。这个对比很适合体现判断力。

- **OpenFeign 失败补偿**
  价值：失败调用不是打一条日志，而是完整保留服务标识、方法名、原始参数、错误信息，再由后台补偿任务重放。

- **补偿重试前先做服务健康检查**
  价值：补偿任务不是无脑重试，而是会先按目标服务分组检查可用实例；如果服务整体不可用，就跳过本轮重试，避免把下游已经有问题的服务继续打崩。

- **RocketMQ 两层幂等**
  价值：Consumer 只做短时快速去重，真正的业务幂等在 service 层兜底，这是很标准的生产思路。

- **消息去重分两套策略**
  价值：有的场景按 `uuid` 记录成功/失败次数，有的场景按消息内容指纹做短时处理窗口去重，说明你不是“一把梭”去重，而是按业务语义选策略。

- **告警到工单的失败重试闭环**
  价值：工单创建失败不是只打印错误，而是把失败消息持久化、按批次重试、达到上限后落最终失败，并定期清理成功消息。这种“失败消息表 + 定时补偿 + 生命周期清理”的闭环很适合写进简历。

- **重试成功后补更新业务记录**
  价值：像油温异常工单这类场景，重试成功后不只是把消息标记成功，还会反查工单并回写告警记录、必要时补更新工单告警内容和日志，说明你考虑的是业务闭环，不只是技术重放。

- **故障跳闸链路是“插入 + 子事件 + 异步通知”组合处理**
  价值：消费故障跳闸消息时，不只是落主记录，还会同步落事件、后果记录，并且只对本次真正新插入的数据触发异步短信，避免重试时重复通知。这很适合体现你对幂等和副作用控制的意识。

- **限流 fail-open**
  价值：限流组件异常时默认放行，说明你考虑的是“保护业务连续性”，不是只考虑组件本身正确。

- **规则引擎不是硬编码**
  价值：Drools/KieSession 的使用能说明系统里存在复杂规则、配置化判断和业务解耦需求。

- **规则引擎前先做多源事实拼装**
  价值：像重过载分级预警，不是直接把原始表丢给规则，而是先按设备维度拼好缺陷、隐患、油色谱、冷却器等多源事实，再送进规则引擎执行。这类“规则前的数据装配”很能体现你理解复杂业务系统。

### 3.3 最容易加分的“设计判断”

- 为什么不用 XXL-Job：
  因为核心问题不是复杂编排，而是统一治理和内网环境下的轻量落地。

- 为什么 Feign 失败不只靠 Feign 自己重试：
  短重试适合瞬时抖动，但不适合需要留痕、可追踪、可补偿的调用失败。

- 为什么补偿任务还要先做服务健康检查：
  因为目标服务如果当前整体不可用，继续重放只会放大故障；先跳过一轮，可以把补偿机制从“重试器”升级成“更克制的稳定性组件”。

- 为什么 RocketMQ 去重还要 service 层幂等：
  Consumer 去重只能挡住短时间重复投递，不能代替真正的业务语义幂等。

- 为什么消息去重要分 `uuid` 和消息指纹两种：
  因为不同消息的重复语义不一样，有些是同业务事件重复投递，有些是相同内容短时间并发到达，去重粒度不能混用。

- 为什么工单失败重试不能只靠接口调用方重试：
  因为工单创建往往带业务副作用，重试不仅要考虑调用成功，还要考虑是否已存在执行中工单、是否需要回写告警记录、是否需要补操作日志。

- 为什么故障跳闸链路要把“落库”和“发短信”拆开：
  因为落库是主事务，通知属于副作用。只有先保证业务记录幂等写入，再对真正新增的数据做异步通知，系统才不会在重试时反复发短信。

- 为什么业务锁和调度锁可以不同：
  因为不同场景对锁可靠性的要求不一样，调度平台更强调严谨性和状态可追踪。

- 为什么长耗时任务要显式处理中断：
  因为这类任务经常伴随批量拉取、并发子任务和长时间写库，如果只靠 finally 或吞掉异常，人工停止和应用停机时状态会非常难收敛。

### 3.4 简历可直接复用的话术

- 从0到1主导沉淀轻量级分布式定时任务平台，统一收口任务注册、动态启停、执行监控、手动控制和启动恢复，治理百级定时任务。
- 主导设计 OpenFeign 失败拦截与落库补偿机制，提升跨服务调用失败后的可恢复性和可追踪性。
- 设计补偿重试前置服务健康检查和按服务分组重放策略，避免目标服务不可用时触发无效重试放大故障。
- 参与 RocketMQ 异步告警链路和 Redis Lua 限流能力建设，围绕消息可靠性、任务稳定性和调用边界做工程治理。
- 参与告警到工单的失败消息重试闭环设计，围绕失败消息持久化、批量重试、业务回写和生命周期清理提升工单链路可恢复性。

## 五、当前最推荐的统一主线

如果你只讲一句自己的核心画像，我建议统一成：

> 我是做 Java 后端出身的，这两年在电网业务域里并行做两条线，一条是微服务治理和稳定性能力，比如任务平台、消息可靠性、失败补偿；另一条是 AI 规则解析应用，把自然语言规则做成可回归、可澄清、可持续迭代的后端系统。

这句话的好处：

- 不会把你拆成两个割裂方向
- 既保住 Java 后端主线，也把 AI 应用经验自然带进去
- 面试官一听就知道后面该往“治理能力”还是“AI 工程化”追问
