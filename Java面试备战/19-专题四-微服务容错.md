# 专题四：微服务容错

> 这条线的逻辑：Feign 调用失败怎么办 → 补偿重试怎么做 → Feign 和 MQ 重试为什么不统一 → 分布式事务怎么保证一致性 → 场景设计（订单）
> 面试官问跨服务调用、数据一致性相关的，都从这条线接。

---

## 入口问题："Feign 调用失败你们怎么处理的？"

### 完整话术（直接背）

> 我们微服务之间用 OpenFeign 调用，网络抖动或者下游服务短暂不可用时调用会失败。我做了一套落库补偿机制来处理这种情况。
>
> 核心思路是用 AOP 拦截 Feign 调用。调用失败后，AOP 会自动把完整的调用上下文持久化到数据库——包括目标服务名、方法名、原始参数和异常信息。注意这里保留的是原始参数，不是重新生成的，就是为了保证后面重试和首次调用的幂等基准一致。
>
> 然后有一个定时任务每 5 分钟跑一次做补偿重试，但不是无脑扫表重发。第一步，按目标服务分组，把待重试的记录按服务名归类。第二步，逐个服务做健康检查，查目标服务有没有可用实例。如果整个服务不可用，这一批直接跳过，不发请求。为什么？因为目标服务如果当前整体挂了，继续重放只会放大故障。第三步，才用落库时保留的原始参数发起真正的重试。第四步，每条记录有最大重试次数，超了就标记为最终失败。
>
> 这样补偿机制就不是一个简单的重试器，而是一个更克制的稳定性组件——该重试的时候重试，不该重试的时候主动跳过。

---

## 追问："被调用方怎么保证不重复处理？"

### 完整话术

> 每次 Feign 调用都会带一个业务唯一 ID，比如工单号或者设备 ID 加时间戳的组合。被调方收到请求后，按这个业务 ID 做幂等校验，最直接的做法就是数据库唯一索引。第一次进来正常插入，第二次进来 insert 冲突，直接返回成功。
>
> 补偿任务重试时用的是跟第一次完全一样的参数和业务 ID，所以被调方的幂等逻辑能正确识别这是一次重复请求。这也是为什么我强调落库时要保留完整的原始参数——就是为了保证幂等基准一致。

---

## 追问："Feign 补偿和 MQ 重试为什么不统一？"

### 完整话术

> 这两种重试解决的是不同层面的问题。
>
> Feign 补偿解决的是同步调用失败后的恢复。我需要保留完整的调用上下文——哪个服务、哪个方法、什么参数——后面才能按原样重放。如果把每次 Feign 失败都发到 MQ 再消费重试，多引入一层中间件依赖，链路更复杂更难排查。
>
> MQ 自带重试解决的是消息消费失败后的恢复。消息本身就在 Broker 里，天然支持重投，不需要额外落库。而且 MQ 的重试有退避策略，间隔越来越长，这个能力是自带的。
>
> 简单说就是 Feign 失败时没有现成的重试基础设施，所以我要自己建；MQ 天然有重试能力，直接用就行。另外 Feign 是我们系统微服务内部之间的调用，MQ 是外部系统和我们之间的通信，通信模式本身就不一样。

## 追问："分布式事务怎么保证一致性？"

### 完整话术

> 分布式事务的核心问题是跨服务调用时一个成功另一个失败怎么保证数据一致。常见方案我按实际用的多少来讲。
>
> 第一种是本地消息表，也是最常用的。业务操作和写消息表放在同一个本地事务里，然后定时任务扫消息表把消息发出去，下游消费成功就标记完成，失败就重试。我的 Feign 失败补偿其实就是这个思路——调用失败的上下文落到本地数据库，后台任务定期重放。区别是本地消息表需要业务主动写消息记录，我的方案是 AOP 自动拦截落库，对业务无侵入。
>
> 第二种是 RocketMQ 事务消息。先发半消息，Broker 收到但消费者看不到。然后执行本地事务，成功就提交让消息可见，失败就回滚丢弃。如果超时没回复，Broker 会定期回查 Producer。适合异步通知场景。
>
> 第三种是 TCC，Try-Confirm-Cancel。每个参与方要写三个接口：Try 预留资源、Confirm 确认、Cancel 释放。一致性最强但代码复杂度高。
>
> 第四种是 Seata AT 模式，阿里开源的，对业务代码侵入小，自动生成反向 SQL 做回滚。但需要额外部署 Seata Server。
>
> 我们项目不需要强一致，所以没用 Seata 也没用 TCC。跨服务调用用 Feign 补偿重试保证最终一致，异步消息用 RocketMQ 两层幂等兜底。够用了。

## 追问（场景设计）："下单后要通知库存扣减、支付、风控，怎么设计？"

### 完整话术

> 我会分成同步链路和异步链路两类。
>
> 库存扣减和支付创建预支付单走同步链路，因为它们直接影响下单结果，必须在主流程里完成。如果库存扣成功了但支付失败，要回滚库存。用本地事务加 Feign 调用，失败走补偿回滚，不用上重量级分布式事务。
>
> 风控校验走异步链路，发消息到 MQ，风控服务异步消费。风控不阻塞用户下单，发现问题再反过来冻结订单。
>
> 这样主流程响应快，风控又不拖慢用户体验。

| 项 | 数字 |

| 补偿任务执行间隔 | 每 5 分钟 |
| 补偿流程 | 按服务分组 → 健康检查 → 重放 → 上限兜底 |
| 分布式事务四种方案 | 本地消息表 → 事务消息 → TCC → Seata |
