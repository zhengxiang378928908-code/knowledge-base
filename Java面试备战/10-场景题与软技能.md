## 一、线上问题排查场景

### 场景 1：服务突然变慢

**排查步骤：**
1. 看监控：CPU、内存、网络、磁盘 IO
2. 看 GC 日志：是不是频繁 Full GC
3. 看慢查询日志：是不是数据库变慢
4. 看中间件：Redis 是否连接数满、MQ 是否积压
5. 看线程堆栈（jstack）：是否有死锁、线程阻塞

### 场景 2：内存持续增长（你遇到过）

**排查步骤：**
1. `jmap -heap <pid>` 查看堆使用情况
2. `jstat -gc <pid> 1000` 观察 GC 频率
3. 如果 Old 区持续增长 → 内存泄漏
4. `jmap -dump:format=b,file=dump.hprof <pid>` 抓 dump
5. MAT 分析大对象和引用链

**你的经验**：跑批一次性加载全量数据导致 OOM，改为分批处理。

### 场景 3：死锁排查

```bash
# 抓线程 dump
jstack <pid> | grep -A 30 "deadlock"

# MySQL 死锁
SHOW ENGINE INNODB STATUS;
# 看 LATEST DETECTED DEADLOCK 部分
```

### 场景 4：接口超时

**排查方向：**
1. 网络问题（DNS、防火墙、带宽）
2. 下游服务慢（调用链追踪）
3. 数据库慢查询
4. 锁等待
5. 线程池满

## 三、行为面高频问题

### Q1：你在工作中遇到的最大挑战是什么？

**建议答 DSL 引擎项目：**

> 最大挑战是电网领域的自然语言到 DSL 的准确转换。电网行业术语复杂，同一个设备可能有十几个别名，规则条件可能有多层嵌套。
>
> 我的解决思路是"多源抽取 + 分层校验"：
> - 先用正则和词典做确定性的信息抽取
> - 再用 LLM 做语义层面的判断和汇总
> - 最后通过 Slot 校验、DSL 校验、回译确认三层保障
>
> 并且建立了 299 条槽位覆盖 + 68 条回归测试的验证集，确保迭代过程中不退步。
> 最终系统在真实语料上达到了可用的准确率。

### Q2：你是怎么推动技术方案落地的？

**建议答定时任务 Starter：**

> 在变电 3.0 项目中，各微服务的定时任务分散管理，没有统一的监控和控制手段。我发现这个问题后，先做了一个 MVP（最小可用版本），只包含注解扫描和基本的日志记录，让团队体验价值。
>
> 之后逐步迭代加入动态启停、手动触发、API 调用计数等功能。设计为 Spring Boot Starter 形式，各服务只需引入依赖即可接入，零侵入。
>
> 最终在项目中统一管理了百级定时任务，运维效率明显提升。

### Q3：你和团队成员有过技术分歧怎么处理？

> 我会先理解对方的出发点，然后用数据和实际案例来对比。比如在选择限流算法时，我建议用令牌桶（允许一定突发），同事建议用固定窗口（简单）。我做了压测对比，证明令牌桶在预警高峰场景下的吞吐量更优，最终达成共识。

### Q4：你对未来的职业规划是什么？

> 短期（1-2年）：深耕 Java 后端 + AI 应用开发的交叉领域，在实际项目中积累更多 RAG、LLM 应用的落地经验。
>
> 中期（3-5年）：成长为能独立负责技术方案设计和团队技术决策的技术骨干，推动团队在 AI 工程化方面的能力建设。

### Q5：你为什么离职？

避免说负面的话。建议方向：
- 寻求更大的技术挑战和成长空间
- 对 AI 应用方向很感兴趣，希望在这个方向深耕
- 希望回到福州发展（如果适用）
